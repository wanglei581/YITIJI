# 工程规模控制与反堆砌开发规范

> 最后更新：2026-06-20
> 适用范围：`apps/`、`services/`、`packages/`、`docs/` 的日常开发、修复、清理和上线前收口。

本规范用于控制项目文件数量、单文件体积、重复代码和无效功能堆砌。后续开发必须优先保留能完成真实业务闭环的核心代码，不为“看起来完整”新增无数据、无入口、无验证、无上线价值的代码。

## 一、开发前准入

每个任务开始前必须先确认：

1. 本次只解决一个明确问题，不能把多个独立问题混在同一轮。
2. 写清本次允许修改的文件范围；没有列入范围的文件默认不动。
3. 明确“不做什么”，特别是：不新增重复入口、不引入新的假数据闭环、不顺手重构无关模块；页面 UI 可以按当前任务重做，但必须写清范围和验证方式。
4. 先查正式文档和现有代码，确认是否已有同类功能、已有入口、已有服务或已有验证脚本。
5. 中高风险或跨模块任务，先做方案审查；方案通过后再写代码。
6. 每个新功能开工前必须确认前端、后端、终端、共享类型、共享 UI 和文档分别落到对应目录，并显式声明不涉及的层。目录职责以 `docs/project-structure.md` 为准。

建议每个任务开工前使用文件预算和功能归位声明：

```text
功能归位声明：
- 功能/业务闭环名称：
- 涉及层和具体目录：
  - 前端：apps/kiosk | apps/admin | apps/partner（填写具体页面、路由或服务适配器）
  - 后端：services/api | services/worker（填写具体 module、controller、service、prisma、异步队列或 verify）
  - 终端：apps/terminal-agent（填写具体硬件链路；不涉及则写“不涉及”）
  - 共享类型：packages/shared（填写协议、DTO、常量或纯函数；不涉及则写“不涉及”）
  - 共享 UI：packages/ui（仅限无业务逻辑的通用组件；不涉及则写“不涉及”）
  - 文档：docs/<子目录>（填写产品、API、设备、合规、进度或审查文档）
- 明确不涉及的层：
- 复用确认：是否已有同类入口、服务、类型、组件或 verify：
- 跨层契约：共享类型/协议放 packages/shared，不在各端硬拷贝；前端和 terminal-agent 不直接访问数据库；硬件直连逻辑只放 apps/terminal-agent。

本任务允许修改：
- path/to/file-a
- path/to/file-b

本任务禁止：
- 新增重复入口
- 修改未声明的页面、路由或业务流程
- 引入无验证的新依赖
- 顺手清理本任务外的代码
- 修改未声明层的文件
```

## 二、单文件大小阈值

除生成文件、迁移快照、静态数据快照外，普通源码文件按以下阈值控制：

| 行数 | 处理规则 |
|------|----------|
| 300 行以内 | 理想状态，继续保持单一职责 |
| 500 行以上 | 新增功能前必须评估拆分 |
| 800 行以上 | 不得继续堆新功能；只能做缺陷修复、拆分或收口 |
| 1000 行以上 | 进入重构/拆分清单，后续任务不得继续扩大 |

页面文件不得同时承载数据请求、复杂状态、弹窗、表单、列表、详情、工具函数和大量静态配置。超过阈值时优先拆为：

- `Page.tsx`：页面编排和路由级状态。
- `components/`：可复用或业务内聚的展示组件。
- `hooks/`：页面数据加载、表单状态、轮询和副作用。
- `services/`：接口调用和协议适配。
- `utils/`：无副作用纯函数。
- `types.ts`：模块内共享类型。

后端 service 超过阈值时，优先按业务能力拆分为独立服务、仓储访问、状态机、校验器或 verify 脚本，不把所有逻辑继续堆在同一个 service 文件里。

## 三、反堆砌规则

以下行为默认禁止：

- 为同一功能新增重复入口、同义卡片、重复页面或重复按钮。
- 新建只有静态文案、无真实数据、无状态、无验收路径的占位功能。
- 将 mock 数据包装成生产能力。
- 为未来可能需要的功能提前建大而全的模块。
- 在旧秒哒项目或临时预览代码中继续开发正式功能。
- 引入新依赖但没有清楚说明用途、替代方案和打包影响。
- 将一次性脚本、调试截图、构建产物、缓存目录、临时导出文件长期放入源码仓库。

## 四、删除和清理规则

清理旧代码必须基于证据，不能凭感觉删除。删除前至少确认：

- 没有路由引用。
- 没有 import 引用。
- 没有测试或 verify 脚本依赖。
- 没有文档声明仍是当前功能。
- 不是生产部署、Windows 终端、打印扫描、密钥门禁或硬件链路会使用的代码。

删除、隐藏、迁移页面或功能后必须：

- 在 `docs/progress/current-progress.md` 记录清理原因和范围。
- 如影响后续任务，在 `docs/progress/next-tasks.md` 更新剩余事项。
- 跑最小相关验证，证明没有破坏现有入口和核心链路。

## 五、仓库体积控制

以下内容不得进入正式 Git 跟踪范围，除非有明确交付理由：

- `node_modules/`、构建产物、coverage、缓存目录、临时目录。
- 大批量截图、录屏、PDF 导出、一次性验收素材。
- 本地 `.env`、密钥备份、数据库备份、真实用户文件。
- 自动生成且可稳定再生成的文件。

需要保留截图或外部材料时，优先放到交付目录或外部归档；仓库内只保留少量能证明验收结论的关键证据，并在文档中说明用途。

## 六、审查和验收

写代码前审查方案，写完后审查 diff。交付前至少确认：

- 改动只覆盖本任务允许的文件。
- 没有新增重复入口、重复数据模型或重复服务。
- 没有把 mock、占位页、临时代码包装成生产能力。
- 单文件体积没有继续越过阈值；如越过，必须记录原因和拆分计划。
- 相关 typecheck、lint、build、verify 或浏览器验收已执行。
- `docs/progress/current-progress.md` 和 `docs/progress/next-tasks.md` 已按实际结果同步。

## 七、目录治理与物理迁移规则

当前仓库目录职责以 `docs/project-structure.md` 为准。`apps/`、`services/`、`packages/` 是标准 monorepo 结构，不因“看起来不分类”而直接物理迁移。

目录治理优先级：

1. 先补目录索引、入口说明、影响评估和归档规则。
2. 再清理 ignored 缓存、生成物和无正式引用的临时交付物。
3. 最后才考虑物理迁移源码目录。

物理迁移源码目录必须满足：

- 从干净 `main` 或独立 worktree 开始，不能混入业务功能改动。
- 先完成迁移影响评估。
- 使用 `git mv`，不手动复制目录。
- 每个阶段单独 commit，不使用 `git add .`。
- 同步 `pnpm-workspace.yaml`、根 `package.json`、CI、部署文档、设备文档和进度文档。
- 确认三端 Tailwind `@source "../../../packages/..."` 在等深度迁移后保持不变。
- 后端迁移必须覆盖 SQLite 主验证和 PostgreSQL readiness。
- Terminal Agent 迁移必须先在旧路径卸载 Windows 服务，再在新路径安装服务。
- 完成后必须运行相关 typecheck、build、verify、本地启动验证，并做 Claude + 前端模型双模型审查。
