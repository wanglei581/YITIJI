## 2. 项目简介

职易达 AI 求职服务终端是一套面向高校就业中心、公共就业服务大厅、招聘会现场、产业园区、街道社区和零工市场等线下场景的自助就业服务解决方案。项目将 AI 简历服务、求职材料打印扫描、第三方岗位与招聘会信息入口、政策服务、AI 助手和终端运维后台整合到一体机中，使求职者能够在现场完成“上传或扫描简历 - AI 诊断和优化 - 生成或打印求职材料 - 查询岗位/招聘会/政策 - 扫码前往来源平台”的闭环服务。

项目的核心定位不是招聘平台，而是**AI 求职材料服务工具 + 线下就业服务触点 + 第三方/官方信息入口**。系统不在站内完成简历投递，不向企业转发求职者简历，不提供企业端候选人筛选、面试邀约或 Offer 管理能力。该定位有利于在公司暂无人力资源服务许可证的当前阶段，先以合规、可落地、可复制的工具型服务切入公共就业和校园就业场景。

项目目前已形成较完整的软件与硬件协同架构：一体机前台服务求职者，管理员后台负责运营管理，合作机构后台支持岗位、招聘会、政策等内容维护，Windows Terminal Agent 连接本地打印扫描硬件。项目已完成 AI 简历诊断、简历生成、简历优化、目标岗位匹配参考、职业规划、模拟面试、政策服务、招聘会/校园招聘会展示、企业展示、浏览与外部跳转记录、打印链路、合规文案与生产演示数据隔离等核心代码闭环；正式商业化上线仍需完成生产服务器、生产数据库、Windows 真机、法务文本和小范围试运营验收。

## 3. 执行摘要

### 3.1 项目定位

职易达 AI 求职服务终端定位为“面向公共就业与校园就业场景的 AI 线下自助服务终端”。产品设计以 21.5 寸触控一体机使用场景为前台入口，以奔图 CM2800/CM2820 系列彩色激光多功能一体机为打印扫描硬件基础，以 AI 大模型和智能体流程为求职者提供材料准备、训练和信息辅助服务。

### 3.2 目标用户

| 用户类型 | 典型需求 |
|---|---|
| 高校毕业生 | 简历诊断、简历优化、模拟面试、职业规划、双选会材料打印、岗位和招聘会查询 |
| 社会求职者 | 求职材料制作、文件打印扫描、政策查询、岗位和招聘会信息入口 |
| 就业服务工作人员 | 降低重复咨询压力，提升现场服务承载能力，保留可审计的服务数据 |
| 高校/人社/园区/招聘会主办方 | 建设可部署、可运营、可监管的线下数字就业服务触点 |

### 3.3 核心价值

1. **对求职者**：把简历、面试、职业规划、打印扫描和信息入口集中到现场终端，降低求职准备门槛。
2. **对机构**：以自助终端分流窗口咨询、材料处理和招聘会现场服务压力。
3. **对运营方**：通过 Admin 后台、Partner 后台和 Terminal Agent，实现小团队甚至单人对多点位的运营管理。
4. **对赛事主题**：项目以 AI 工具和智能体流程提升单人运营效率，符合 OPC 挑战赛对“AI+智能体”创业能力的关注。

### 3.4 当前进度

| 模块 | 当前状态 |
|---|---|
| Kiosk 一体机前台 | 核心页面和功能链路已完成，支持 21.5 寸触控场景 |
| AI 简历服务 | 诊断、生成、优化、岗位匹配参考、职业规划已形成代码闭环 |
| 模拟面试 | 已完成设置、对话练习、报告和打印链路 |
| 岗位/招聘会/政策入口 | 已接入来源信息展示、浏览记录和外部入口打开记录；收藏能力按具体对象范围开放 |
| Admin 后台 | 终端、内容、订单、审核、告警、日志等运营能力已形成基础 |
| Partner 后台 | 合作机构可维护岗位、招聘会、政策、企业等来源数据 |
| Terminal Agent | 打印链路、安全基线和 Windows 本地设备交互设计已完成，生产真机仍需复验 |
| 上线准备 | 生产服务器、生产 PostgreSQL、Windows 真机、法务合规、小范围试运营仍待完成 |

以上为项目代码与文档层面的完成度说明，不等同于已经生产上线。生产可用性仍以第 10 章列出的上线验收、真机复验和小范围试运营结果为准。

### 3.5 商业化路径

项目优先采用 B2G/B2B2C 路径：先在高校就业中心、公共就业服务大厅或招聘会现场部署 1 台终端做小范围试点，验证使用率、打印任务、AI 服务次数、扫码外跳、设备在线率和工作人员满意度；再按“单点试运营 - 多点复制 - 区域合作”的节奏扩展。收入结构包括机构端设备/软件/运维服务费、活动现场服务包、C 端打印及 AI 工具增值服务、合规的宣传屏/权益合作等。所有收入预测均需以试点数据校正。

## 4. 大赛适配性分析

### 4.1 与 OPC 挑战赛主题的匹配

2026“创·在青岛”创业大赛 OPC 挑战赛强调“AI赋能OPC创业新时代”，初赛材料重点考察项目创新度、可行性、产业适配性，以及利用 AI、大数据、算力等数字技术，以“AI+智能体”开展单人运营的能力。

职易达的适配点如下：

| 评审关注点 | 项目回应 |
|---|---|
| AI 赋能 | AI 简历诊断、生成、优化、职业规划、模拟面试、政策问答和岗位匹配参考 |
| 智能体单人运营 | Admin 后台监控多终端，Partner 自助维护来源数据，Terminal Agent 自动 claim 打印任务和上报状态 |
| 创新度 | 将 AI 求职服务与线下打印扫描硬件结合，形成“现场服务 - 即时产出 - 后续记录”的软硬一体方案 |
| 可行性 | 已有较完整代码基础和设备设计；采用先试点后复制的低风险路径 |
| 产业适配性 | 对应青岛青年就业创业、公共就业服务、人工智能应用和现代服务业数字化方向 |
| 可推广性 | 适用于高校、就业服务大厅、招聘会、园区、社区等多类线下场景 |

### 4.2 OPC 单人创业逻辑

本项目的 OPC 逻辑不是“一个人手工服务所有求职者”，而是通过产品和智能体把重复服务流程标准化、自动化：

1. AI 承担求职材料诊断、生成、优化、面试训练、职业规划等标准化咨询工作。
2. Partner 后台让高校、人社、园区、招聘会等合作机构自行维护来源信息，降低平台方内容采集负担。
3. Admin 后台集中显示终端、订单、告警、日志和内容状态，便于远程运维。
4. Windows Terminal Agent 负责本地打印任务领取、状态回传、临时文件清理、心跳和故障上报。
5. 运营者可通过数据看板和告警机制管理多个点位，把人力集中在客户拓展、试点复盘、合规审查和关键异常处理上。

以上机制使项目具备“小团队起步、单人主导运营、多点位复制”的现实基础。

## 5. 市场背景与机会

### 5.1 宏观就业服务需求

公开资料显示，教育部预计 2026 届全国普通高校毕业生规模为 1270 万人，同比增加 48 万人。高校毕业生就业服务仍面临总量压力、结构性匹配和求职能力提升等多重需求。求职者不仅需要岗位信息，也需要简历、面试、职业规划、材料打印和政策理解等具体工具。

### 5.2 青岛本地政策环境

青岛市公开发布的青年人才就业创业措施提出，围绕青年人才“就业有岗位、创业有扶持、科研有助力、住房有保障”等方向构建发展生态，并提到每年举办招聘活动 1000 场次以上、发布就业岗位 40 万个以上，建设就业赋能中心、支持驻青高校打造大学生就业创业赋能中心，推动“10+1”创新型产业体系就业支撑计划。

这说明青岛本地存在明确的政策方向：就业服务需要更强的线下触达、更高的数字化程度、更好的青年人才服务体验。职易达可以作为高校、就业大厅、园区和招聘会中的数字就业服务触点，与政策方向形成适配。

### 5.3 线下求职场景痛点

| 场景 | 典型痛点 | 终端解决方式 |
|---|---|---|
| 高校毕业季 | 简历集中修改、就业指导老师忙、双选会材料打印集中 | AI 简历诊断/优化 + 简历打印 + 双选会入口 |
| 招聘会现场 | 求职者临时找岗位、找展位、打印资料、扫码来源平台 | 招聘会信息、参展企业、场馆导览、资料打印、来源二维码 |
| 就业服务大厅 | 政策咨询和材料处理重复性高 | 政策说明、官方入口、材料清单、打印扫描 |
| 园区/社区 | 基层就业服务缺少轻量化工具 | 自助服务终端 + 后台统一管理 |

### 5.4 市场切入判断

本项目不从“全国平台流量”切入，而从“线下点位服务”切入。原因是：

1. 线下求职场景存在明确的即时打印、扫描和材料准备需求。
2. 高校、就业大厅、招聘会和园区天然具备求职者人流。
3. 机构端对合规、可监管、可审计的工具型服务接受度高于招聘闭环平台。
4. 终端设备形成现场入口，有利于积累真实使用数据和运营案例。

## 6. 产品与功能方案

### 6.1 总体产品架构

职易达由四个核心部分构成：

| 子系统 | 使用者 | 核心功能 |
|---|---|---|
| Kiosk 一体机前台 | 求职者 | AI 简历、打印扫描、岗位/招聘会/政策入口、AI 助手、我的记录 |
| Admin 运营后台 | 运营人员 | 终端管理、订单管理、内容审核、告警处理、文件与日志审计 |
| Partner 合作机构后台 | 高校、人社、园区、招聘会主办方 | 岗位、招聘会、政策、企业资料等来源数据维护 |
| Windows Terminal Agent | 一体机本地设备 | 打印任务领取、文件下载、打印执行、状态回传、心跳与故障上报 |

### 6.2 求职者端功能

| 功能模块 | 功能说明 | 业务价值 |
|---|---|---|
| AI 简历诊断 | 上传或扫描简历后进行结构化解析、问题识别和改进建议 | 解决求职者不知道简历问题在哪里的痛点 |
| AI 简历生成 | 通过分步表单生成可编辑简历并导出 PDF | 降低首次制作简历门槛 |
| AI 简历优化 | 基于真实经历重写表达、优化排版和重点呈现 | 提升材料表达质量，但不编造经历 |
| 目标岗位匹配参考 | 将个人简历与目标岗位做三档参考匹配 | 帮助用户判断修改方向，不承诺录用 |
| 职业规划 | 根据用户简历、目标方向和已有记录生成建议单 | 提供个人求职行动建议 |
| 模拟面试 | 选择岗位、行业、难度和时长后进行 AI 对话训练 | 帮助用户练习表达和应答 |
| 打印扫描 | 支持文件上传打印、简历打印、报告打印，扫描能力按真机验收推进 | 满足线下即时产出需求 |
| 岗位信息入口 | 展示第三方来源岗位，提供来源平台二维码或外链 | 只做信息入口，不做站内招聘闭环 |
| 招聘会入口 | 展示招聘会、校园招聘会、参展企业、资料和导览 | 提升现场参会效率 |
| 政策服务 | 展示政策说明、材料清单、官方入口和打印服务 | 做信息指引，不代办、不承诺到账 |
| 我的记录 | 查看本人简历、文档、打印订单、AI 服务记录、按对象开放的收藏、浏览与外部跳转记录 | 形成用户数据闭环 |

### 6.3 机构端功能

| 端口 | 能力 | 价值 |
|---|---|---|
| Admin 后台 | 终端管理、内容审核、订单告警、日志审计、文件管理、AI 配置 | 保障多终端运营和合规管理 |
| Partner 后台 | 合作机构维护岗位、招聘会、政策、企业风采和数据源 | 让数据由来源机构维护，提升可信度并降低运营成本 |

### 6.4 设备与硬件能力

当前项目按 21.5 寸触控一体机使用场景设计，配套奔图 CM2800/CM2820 系列彩色激光多功能一体机。根据项目设备文档，已确认的打印扫描设备能力包括 A4 黑白/彩色打印、自动双面、复印、平板扫描、ADF 多页扫描、有线网络、扫描到 SMB 等。设备不支持 A3，不假设云端远程扫描 API。

商业计划书对硬件能力的表达必须保持审慎：

- 可写：A4 打印、彩色打印、双面打印、ADF 扫描、Windows 本地 Agent 驱动。
- 不写：A3、跨云远程扫描、未确认的打印 API 彩色参数、完全无人值守长期运行。

## 7. 技术方案与 AI 智能体能力

### 7.1 技术栈

| 层级 | 技术方向 |
|---|---|
| 前端 | React + Vite + TypeScript + Tailwind CSS + shadcn/ui + lucide-react |
| 后端 | NestJS 或 FastAPI，当前项目以 NestJS 服务为主要实现方向 |
| 数据库 | PostgreSQL，开发兼容 SQLite；Redis/BullMQ 用于队列和缓存 |
| 文件存储 | MinIO / 阿里云 OSS / 腾讯 COS |
| 设备 Agent | Windows 本地 Node.js/.NET/Python 方案，当前 Terminal Agent 已形成设计和实现基础 |
| AI 能力 | 大模型配置服务、OCR、AI 简历、模拟面试、职业规划、政策问答等 |

### 7.2 AI 能力矩阵

| AI 能力 | 当前业务用途 | 合规边界 |
|---|---|---|
| OCR 简历识别 | 识别图片/扫描件简历，进入简历诊断闭环 | 仅处理用户本人上传或扫描文件 |
| 简历诊断 | 识别结构、表达、完整性问题 | 仅作参考，不代表求职结果 |
| 简历生成 | 根据用户填写事实生成简历 | 不编造学历、项目、证书和经历 |
| 简历优化 | 优化表达和结构 | 基于真实经历，不伪造事实 |
| 岗位匹配参考 | 生成较高/中等/偏低等参考判断 | 不写百分比录用概率，不回传企业 |
| 模拟面试 | 面试问答训练与报告 | 只给本人练习，不参与企业筛选 |
| 职业规划 | 提供行动建议和提升路径 | 不承诺薪资、录用或就业结果 |
| 政策问答 | 帮助理解政策和材料 | 不代办，不承诺补贴到账 |

### 7.3 智能体运营设想

面向 OPC 单人创业，项目后续可将智能体能力进一步用于内部运营：

| 智能体方向 | 作用 | 当前状态 |
|---|---|---|
| 运营巡检助手 | 汇总终端在线率、打印失败、缺纸告警和异常订单 | 基于现有 Admin/Agent 数据可规划 |
| 内容审核助手 | 辅助检查来源数据字段完整性、合规文案和禁用表达 | 部分合规守卫已有基础 |
| 试点复盘助手 | 自动生成点位运营周报、使用量和问题清单 | 需试点数据后实现 |
| 客服知识助手 | 回答打印、上传、AI 简历、政策入口等常见问题 | 可基于 AI 助手能力扩展 |
| 商务材料助手 | 根据不同客户生成高校版、人社版、招聘会版方案 | 可作为内部运营工具 |

这些能力不必在第一阶段全部开发完成，但应作为“AI+智能体单人运营”的明确路线写入计划书。

项目采用 B2G / B2B2C / B2B 合作并行的商业模式，以机构端试点为起点，以 C 端工具服务付费和活动服务为补充。

## 9. 财务预测与测算假设

### 9.1 测算原则

由于项目尚未完成生产试运营，本文不写历史收入、历史用户量或历史装机量。以下为试点经营模型的预测假设，用于说明商业可行性和关键变量，不作为已发生数据。

### 9.2 项目年度总预估收入公式

项目年度总预估收入可按以下公式测算：

```text
项目年度总预估收入 =
全部点位机构端年度服务费
+ 全部活动现场服务费
+ 全部点位C端AI工具服务收入
+ 全部点位C端打印扫描服务收入
+ 全部合规权益/宣传合作收入
```

C 端 AI 工具服务收入可按以下公式测算：

```text
C端AI工具服务收入 =
终端点位数 × 有效运营天数 × 单台日均使用人数 × 付费转化率 × 平均客单价
```

上述公式按“人次”估算，不按独立用户去重，适用于招聘会、就业大厅、高校就业季等现场流动人群场景。打印扫描服务收入应单独按纸张、墨粉、设备折旧、场地方分成和维护成本测算，不能只看流水。本章阶段收入小计先汇总机构端服务费、活动现场服务包和 C 端 AI 工具收入三项，暂不计入打印扫描和权益/宣传合作收入。

### 9.3 三档规模情景假设

以下为预测假设，需通过真实试点修正。保守档按小范围试点测算，基准档和积极档按后续扩点或区域推广情景测算。考虑高校、人社大厅等场景具有公益属性，早期收入重心按机构端服务费、活动服务包和机构采购免费额度设计；保守与基准阶段 C 端自费 AI 工具收入仅作为补充项，积极推广阶段因点位、人流与转化率提升，C 端贡献上升但仍以机构端收入为主。

| 项目 | 保守试点 | 基准扩点 | 积极推广 |
|---|---:|---:|---:|
| 终端点位数 | 3 台 | 10 台 | 30 台 |
| 有效运营天数 | 180 天 | 240 天 | 300 天 |
| 单台日均使用人数 | 20 人 | 35 人 | 50 人 |
| AI 工具付费转化率 | 5% | 10% | 15% |
| AI 工具平均客单价 | 6 元 | 8 元 | 10 元 |
| 预测 AI 工具收入 | 3,240 元 | 67,200 元 | 675,000 元 |
| 机构端年度服务费假设 | 60,000 元 | 250,000 元 | 900,000 元 |
| 活动现场服务包收入假设 | 12,000 元 | 60,000 元 | 180,000 元 |
| 阶段收入小计 | 75,240 元 | 377,200 元 | 1,755,000 元 |

计算口径示例：

```text
保守试点 AI 工具收入 =
3 台 × 180 天 × 20 人/天 × 5% × 6 元 = 3,240 元

基准扩点 AI 工具收入 =
10 台 × 240 天 × 35 人/天 × 10% × 8 元 = 67,200 元

积极推广 AI 工具收入 =
30 台 × 300 天 × 50 人/天 × 15% × 10 元 = 675,000 元

活动现场服务包收入参考口径 =
保守试点 4 场 × 3,000 元/场 = 12,000 元
基准扩点 15 场 × 4,000 元/场 = 60,000 元
积极推广 30 场 × 6,000 元/场 = 180,000 元
```

机构端年度服务费和活动现场服务包收入同样为经营预测假设，不代表已签约收入。上述阶段收入小计仅汇总机构端服务费、活动现场服务包和 C 端 AI 工具收入，尚未计入打印扫描收入、权益/宣传合作收入或机构采购免费额度；正式提交前，应以真实客户报价、设备采购单和试点服务内容替换。

### 9.4 机构端服务费假设

机构端年度服务费可按点位、功能、服务响应等级和是否包含硬件分档。正式报价必须以设备采购价、部署成本、服务范围和客户预算为准。参赛计划书可使用以下假设区间进行展示：

| 服务方式 | 假设收入区间 | 说明 |
|---|---:|---|
| 软件与后台年度服务费 | 1 万-3 万元/台/年 | 不含硬件，适合客户自有设备或集成商供货 |
| 软硬件打包租赁 | 待供应链询价后确定 | 包含一体机、打印机、部署、运维，成本敏感 |
| 招聘会现场服务包 | 2,000-8,000 元/场 | 视活动规模、服务时长、打印量和人员支持确定 |

以上区间仅为预测假设，不代表已成交价格。

### 9.5 成本结构与单台成本占位测算

| 成本类型 | 内容 |
|---|---|
| 硬件成本 | 触控一体机、打印机、扫码设备、网络设备、备机 |
| 软件研发 | 前端、后端、后台、Agent、AI 服务、测试与安全加固 |
| 云服务成本 | 服务器、数据库、Redis、对象存储、短信、OCR、大模型调用 |
| 运维成本 | 部署、巡检、耗材、远程支持、现场维修 |
| 合规成本 | 用户协议、隐私政策、数据安全、法务审查、日志留存 |
| 商务成本 | 试点沟通、方案制作、培训、渠道合作 |

正式提交前应补充供应商报价。若暂未取得报价，可先使用以下占位测算模型，后续用真实询价替换：

| 单台年度成本项 | 预测假设区间 | 说明 |
|---|---:|---|
| 硬件折旧 | 4,000-8,500 元/台/年 | 以触控一体机、打印机和基础外设合计投入按 3 年摊销估算；正式版须替换为报价单 |
| 云服务与基础软件 | 2,000-6,000 元/台/年 | 服务器、数据库、对象存储、短信、OCR、AI 调用基础成本；高用量另计 |
| 耗材与日常维护 | 3,000-8,000 元/台/年 | 纸张、墨粉、易耗件、远程支持和常规巡检 |
| 部署与备件预留 | 2,000-5,000 元/台/年 | 初装、网络、备件、临时维修和应急替换 |
| 单台年度基础成本合计 | 11,000-27,500 元/台/年 | 预测假设，不代表真实采购价或正式报价 |

上述区间仅用于参赛阶段判断商业可行性。真实商业报价应以“硬件供应商报价 + 云服务账单 + 耗材价格 + 运维服务范围 + 场地方分成”重新测算。

### 9.6 盈亏平衡判断

项目能否快速实现盈亏平衡，主要取决于三件事：

1. 硬件由客户采购、租赁还是项目方自投。
2. 机构端服务费能否覆盖基础运维和云服务成本。
3. C 端 AI 工具服务和打印扫描是否能形成稳定复购。

最稳妥的早期路径是：优先争取机构端试点服务费或活动服务费，不以大规模自投硬件作为第一步；待试点数据证明使用率和付费意愿后，再考虑租赁或区域复制。

按 9.5 的单台年度基础成本假设，早期单台终端建议由机构端服务费覆盖约 1.5 万-2.5 万元/年的基础成本，C 端 AI 工具、打印扫描和活动服务包作为增量收入。若机构端服务费低于基础成本，则必须通过打印扫描收入、活动现场服务包或场地方共同采购补足，否则不应扩大自投硬件规模。

## 11. 竞争分析

### 11.1 竞品类型

| 类型 | 优势 | 不足 | 职易达差异 |
|---|---|---|---|
| 普通打印店/校园打印点 | 低成本、用户熟悉 | 只解决打印，不提供求职辅导和信息入口 | AI 求职服务 + 打印扫描 + 数据闭环 |
| 在线简历工具 | 模板丰富、传播快 | 缺少现场硬件交付和机构管理能力 | 线下即时打印和机构部署 |
| 招聘平台 | 岗位数量大、线上转化强 | 需要招聘服务资质和平台闭环能力 | 不做招聘平台，只做来源入口和材料工具 |
| 政务/校园信息屏 | 传播政策和通知 | 缺少交互、AI 服务和打印扫描 | 可操作、可产出、可记录 |
| 自助政务终端 | 稳定、规范 | 通用性强但就业服务垂直度不足 | 聚焦求职材料和就业服务场景 |

### 11.2 核心壁垒

1. **场景壁垒**：深度聚焦求职材料、招聘会、校园就业、政策服务等线下高频场景。
2. **软硬一体壁垒**：不只是网页，也连接打印扫描硬件和本地 Agent。
3. **合规壁垒**：从产品文案、数据记录到后台功能均围绕非招聘闭环设计。
4. **运营壁垒**：三端架构支持多机构供给、多终端管理和可审计运营。
5. **数据闭环壁垒**：用户的简历、文档、打印订单、AI 服务记录、收藏、浏览和外部跳转记录各有归属，便于后续服务。

## 12. 合规与风险控制

### 12.1 合规边界

项目当前阶段必须坚持：

| 可以做 | 不做 |
|---|---|
| AI 简历诊断、生成、优化、打印 | 站内完成招聘投递 |
| 模拟面试和职业规划 | 承诺录用、面试或薪资 |
| 第三方岗位与招聘会信息入口 | 平台收简历给企业 |
| 去来源平台投递/扫码投递 | 企业端候选人筛选 |
| 去来源平台预约/扫码预约 | 面试邀约和 Offer 管理 |
| 政策说明、材料清单、官方入口 | 代办补贴或承诺补贴到账 |
| 浏览和外部入口打开记录 | 记录第三方平台后续投递或预约结果 |

### 12.2 数据安全

1. 用户文件通过临时签名 URL 访问，不使用永久公开链接。
2. 简历、扫描件、证件类文件应设置清理周期和访问日志。
3. 管理员访问用户文件须留痕。
4. 打印任务状态须持久化，回调和状态更新须具备幂等能力。
5. 生产环境禁止混入演示数据。
6. AI 结果失败时不生成假结果，不把演示数据包装成真实数据。

### 12.3 关键风险与应对

| 风险 | 等级 | 应对措施 |
|---|---|---|
| 被误解为招聘平台 | 高 | 全文强调工具服务和来源入口，按钮和商业模式不写站内招聘闭环 |
| 生产部署不稳定 | 高 | 先做 1 台终端试运营，完成服务器、数据库、真机验收后再扩点 |
| 硬件故障影响体验 | 中 | Agent 心跳、状态上报、日志、备机和现场巡检流程 |
| AI 输出不准确 | 中 | 免责声明、事实校验、用户可编辑、失败不产假结果 |
| 数据安全风险 | 高 | 文件 TTL、签名 URL、审计日志、最小权限、生产密钥轮换 |
| 商业化转化不足 | 中 | 先验证真实使用率和付费意愿，再扩大硬件投入 |
| 合作机构数据质量不稳定 | 中 | Partner 维护 + Admin 审核 + 来源字段强制展示 |

## 13. 组织与团队计划

项目适合以 OPC 或小团队方式启动：

- 创始人负责产品、客户沟通、试点推进和合规把关。

## 15. 结论

职易达 AI 求职服务终端具备较高的参赛可行性和落地价值。项目不是从重资产、大平台、大规模招聘闭环切入，而是从高频、刚需、合规、可试点的线下就业服务工具切入；既服务求职者的材料准备和求职训练，也服务高校、人社、园区、招聘会等机构的现场服务数字化需求。

项目最大的差异化不在于单个 AI 功能，而在于“AI 求职服务 + 线下打印扫描硬件 + 合作机构数据供给 + 运营后台 + 本地 Agent”的组合能力。该组合使项目既能讲清楚 AI 创新，也能讲清楚商业落地、机构采购和单人运营逻辑。对 OPC 挑战赛而言，项目应重点强调“AI+智能体支撑单人运营”“公共就业服务触点”“合规非招聘平台”“先试点后复制”四条主线。

正式参赛前，建议尽快补齐三类材料：第一，创始人与主体资格信息；第二，实机照片、操作视频和界面截图；第三，首个试点点位的合作意向或调研记录。只要这些证据补齐，项目在书面评审中的可信度和专业度会明显提升。
