## 一、定位与目标客户

**一句话定位**：政企版不是第二个产品，是同一台「AI求职打印服务终端」面向**政府/机构买单方**的交付形态——买单方是人社局/高校/园区，使用方仍是求职者，求职者侧全免费。

| 客户类型 | 部署场景 | 买单方式 | 核心诉求 |
|---|---|---|---|
| 人社局 / 公共就业服务中心 | 人社大厅、就业驿站 | 政府采购招标：按台售卖 + 年度运维 | 分流窗口压力、政策触达、服务人次数据、适老化 |
| 零工市场 / 劳务市场运营方 | 零工驿站大厅 | 政府购买服务 / 运营方采购 | 求职登记引导、岗位信息触达、简历帮扶 |
| 高校就业指导中心 | 就业中心大厅、宿舍区 | 高校经费 / 按年订阅 | AI 简历与面试练习、校招信息、就业指导数据 |
| 招聘会主办方 / 工会 / 园区 | 招聘会现场、园区服务站 | 按场次租赁 / 采购 | 现场服务能力、材料打印、主办方品牌露出 |

对应市场先例（见 [market-research-2026-07.md](./market-research-2026-07.md)）：无锡智慧人社 81 项自助一体机、邯山零工驿站求职一体机、安康零工市场智能终端——均为政府采购/招标模式，硬件参考价 1.8–2.5 万/台 + 年度运维。

**差异化卖点（投标话术）**：现有人社一体机只做"查询+打印证明"，本产品带完整 AI 求职服务（简历诊断/优化/生成、模拟面试、岗位匹配、职业规划、政策 AI 指引），且岗位/招聘会严格做第三方来源信息入口，与公共就业服务定位天然契合。

## 二、设计总原则（先想清楚，再动手）

1. **单代码库，配置驱动，绝不 fork**。政企版 = 标准版 + 一组终端级/机构级配置项 + 可选外设插件。遵守仓库"不新增第二套标准"治理铁律；所有差异收敛到一个 `deploymentProfile` 配置面（§五）。
2. **能力项可单独勾选，不捆绑**。招标需求千差万别（有的只要免费模式，有的要读卡登录），每个能力项独立开关、独立报价、独立验收。
3. **合规更严而不是更松**。政务场景叠加《个人信息保护法》《居民身份证法》、等保要求；身份证信息**只核验不留存明文**；数据报表**只出脱敏聚合，永不出个人明细**（沿用智慧校园"不采集学生个人明细"先例）。
4. **订单驱动交付**。G1（纯软件）随 Phase 3 启动即可交付；G2–G4（外设/渠道/私有化）签约后再开发，避免无客户空投入。
5. **不伪造能力**。投标清单里未真机验收的能力（扫描、读卡）标注"选配/按项目实施"，不承诺未验证事项——沿用项目"不伪造能力"红线。

## 四、能力项详细设计

### E1 整机免费模式

- **实现**：`PriceConfig` 终端级配置全部 item 置 `free`（或终端级总开关 `billingEnabled=false`）。
- **Kiosk 行为**：`billingEnabled=false` 时，确认打印页/AI 付费点**不渲染任何价格与支付 UI**，直接"确认打印"；订单照常落库（`payChannel=free`、`amountCents=0`）——免费单落库是合规硬要求（compliance §8.5），也是 E2 报表的数据来源。
- **防误配**：免费模式为终端级显式配置 + Admin 改动写审计；与付费终端可在同一后端共存（同一运营方混合点位场景）。
- **可选配额**：支持"免费但限额"（如每人每日免费黑白 20 页，防滥用刷纸）：`PriceConfig` 补 `dailyFreeQuota` 项，超额后提示"今日免费额度已用完"（政企点位默认不启用，招标要求时开）。

### E2 运营数据报表 + 服务数据大屏（政企客户的续约依据，G1 最重要交付物）

政企客户买完设备，最关心"这台机器服务了多少人"。这是续约和验收的核心。

- **数据源（全部已有，只缺聚合端点）**：`Order`/`PrintTask`（打印量）、`AiResumeResult`/`MockInterviewSession`（AI 服务量）、`BrowseLog`/`ExternalJumpLog`（信息触达量）、`EndUser` 注册数、终端心跳（在线率）。
- **新增**：`GET /admin/reports/terminal-operations`（按终端/按时间段聚合）+ Admin「运营报表」页 + CSV 导出。指标：服务人次、打印页数（黑白/彩色）、AI 简历诊断次数、模拟面试次数、政策查询次数、岗位/招聘会浏览与外部跳转次数、设备在线率。
- **服务数据大屏（选配）**：Kiosk 待机屏新增"服务数据"模板（今日服务 X 人次、累计优化简历 X 份……），素材化接入既有 `AdPlaylist` 播放方案——政务大厅爱看,零开发新页面框架。
- **合规硬线**：一切指标为**计数聚合**，不含任何个人明细、手机号、简历内容；报表页不提供个人下钻。Partner 端复用同一聚合口径（总体方案 §八-7 Partner 统计接真）。

### E3 终端冠名与品牌配置

- 复用既有 `Terminal` 设备档案 + `TerminalScreensaverConfig`：新增 `brandingJson`（机构冠名名称、Logo URL、欢迎语），Kiosk 首页顶部/待机屏/打印小票页脚展示"XX区人社局 · AI求职服务终端"。
- **不触碰 UI 冻结**：冠名是既有页面的配置化文案/图片槽位，不改布局、不改入口结构；实现前按惯例给用户过一次视觉确认。

### E4 大龄友好模式

- 终端配置 `accessibilityMode`：全局字号放大档（CSS 变量缩放，触控目标已 ≥48px 不变）、高对比配色档、AI 助手「小青」语音引导常驻入口强化。
- 零工市场/人社大厅场景刚需（大龄灵活就业者），投标加分项；高校版默认关闭。

### E5 政策服务政企深化

- **政策 AI 匹配指引单**（总体方案 A14）：求职者口述/勾选自身情况 → AI 匹配可能符合的政策 → 生成 A4 指引单（政策要点 + 材料清单 + **官方办理入口二维码**）打印带走。合规：info-only，不承诺到账、不代办、不留存身份证/银行卡信息。
- **求职登记辅助**：政府"求职登记"是人社业务，我们**只做辅助**——引导填写登记信息 → 打印《求职登记表》纸质表 + 官方登记入口二维码，由本人交窗口或扫码自行登记。系统不代提交、不留存登记表内容（打印完成即入既有 TTL 清理）。文案红线："本机不代办登记，请前往窗口或扫码到官方平台办理"。
- **政策内容供给**：人社局作为 Partner 机构入驻（既有 Organization + PolicyPost 链路），本局政策由本局账号维护、Admin 审核——数据供给关系与现有架构零冲突。

### E6 身份证读卡登录（G2 核心，硬件项）

**目标**：大厅场景大量用户无手机或不便扫码，刷身份证 3 秒登录。

- **硬件**：二代证读卡器（华视/精伦等主流型号，USB 接 Windows 一体机；选型标准：有 Windows x64 SDK、支持公安部安全模块）。
- **数据流（复用 QR 登录的 Agent 本地桥接安全模型）**：
  1. Kiosk 登录页新增"刷身份证"入口（终端配置开启才显示）→ 调 Terminal Agent `127.0.0.1` 本地端点 `POST /idcard/read-session`；
  2. Agent 调读卡器 SDK 读卡 → **在 Agent 内存中完成**：取姓名 + 证件号 → 计算 `idNumberHash`（服务端盐 HMAC）→ 姓名脱敏（王*雷）→ **明文证件号与照片立即丢弃，不落盘、不上传**；
  3. Agent 携终端凭证调后端 `POST /member/auth/idcard-verify`（仅传 hash + 脱敏姓名 + 读卡器核验结果）→ 后端按 `idNumberHash` 匹配/创建 EndUser → 返回一次性 loginToken → Agent 转交 Kiosk 完成登录；
  4. 全程写审计（登录事件，不含证件信息）。
- **模型改动**：`EndUser` 增加可空 `idNumberHash`（唯一索引）+ `idVerifiedAt`；与手机号账号可绑定合一（刷证登录后提示"绑定手机号可在其他终端使用"）。
- **安全红线**：证件号明文**任何一层都不落库不落日志**；照片不留存；hash 盐只在服务端 env；Agent 端点只监听 127.0.0.1（既有单实例锁与本地代理模式复用）；《居民身份证法》合规审查随首个订单做法务评审。
- **Agent 外设插件框架**：借本项建立 Agent 外设桥接的统一模式（本地 HTTP 端点 + 权限 + 审计），后续扫描仪（E7）、U盘、高拍仪都走同一框架——一次投入多次复用。

### E7 真机扫描链路

- 奔图 CM2800 硬件支持扫描（ADF 50 页），当前 Agent 无扫描代码（前端为诚实演示）。政企场景刚需：证明材料扫描归档、纸质简历扫描进 AI 诊断。
- **方案（按 CLAUDE.md 既定三选一，推荐①）**：① Agent 调 Windows WIA/TWAIN 驱动扫描 → 生成 PDF → 走既有 `kiosk-upload` 入库；② 打印机扫描到 SMB 目录 + Agent 监听；③ U盘中转。
- 扫描文件按 `FileObject` 敏感等级 + TTL 清理；纸质简历扫描直通 `/resume/parse` OCR 链路（后端已支持扫描版 PDF ≤3 页）。

### E8 电子社保卡扫码登录（G3，渠道项）

- 用户用掌上 12333/支付宝亮电子社保卡二维码 → 一体机扫码枪读码 → 后端调人社电子社保卡开放渠道核验 → 登录。
- 依赖：电子社保卡服务渠道接入审批（周期不可控，**签约后立即启动申请**，对标短信审核教训）；扫码枪外设走 E6 建立的 Agent 插件框架。
- 亮点：电子社保卡"亲情服务"可为家人代办场景预留。

### E9 人脸识别核验（谨慎项，默认不做）

- 仅当招标文件硬性要求时评审启动；人脸属敏感个人信息，需单独隐私影响评估 + 明示同意 + 不留底片；优先引导用 E6/E8 替代。**本方案默认不承诺此项。**

### E10 私有化/本地化部署包（G4）

政务客户常要求"数据不出本地/不出政务云"。分两档：

- **档 A（优先推）：政务云单租户部署**。现有栈（NestJS + PostgreSQL + Redis + 对象存储 + nginx + PM2）整体部署到客户指定政务云，复用已验证的预生产部署 runbook（Gate 2 裁剪运行时归档 + 构建 + 迁移 + 回滚模式已跑通多轮）；对象存储适配客户侧（COS 协议兼容或本地 MinIO——`FILE_STORAGE_DRIVER` 抽象已存在，需补 MinIO/S3 driver）。
- **档 B（按需）：完全离线内网部署**。额外解决：AI 大模型（对接客户已采购的政务大模型/本地化模型——`AiModule` Provider 架构已支持多 vendor 配置，新增一个 OpenAI 兼容 endpoint provider 即可）、OCR（百度 OCR 私有化或替代）、短信（客户短信网关）。**档 B 每一项外部服务都要在合同里写清由谁提供**，避免"离线但要用云 AI"的矛盾需求。
- 交付物：部署包（版本化归档 + sha256）、部署手册、验收清单、回滚手册——全部由既有 Gate 流程文档模板改造。

### E11 运维 SLA 包

- 软件侧：Alert 模型 + 处理工单流转（总体方案 §八-10）、缺纸/卡纸/碳粉阈值告警（Agent WMI 已上报状态，补耗材项）、Admin 告警 → 指派 → 处理记录 → 关闭闭环、可选短信/企微通知运维人员。
- 服务侧（合同条款）：响应时效承诺（对标共享打印行业"5 小时补纸"痛点，承诺优于它）、月度运营报告（E2 报表导出）。

## 五、技术架构：配置面设计（政企版的"开关总成"）

**新增 `TerminalDeploymentProfile`（终端级，additive migration）**，与既有 `Terminal` 档案、`TerminalToolboxConfig`、`TerminalScreensaverConfig`、`TerminalSmartCampusConfig` 并列，沿用同一"Admin 配置 → 公开 Kiosk config 白名单收窄下发"模式：

```
TerminalDeploymentProfile {
  terminalId        String  @unique
  edition           String  // standard | gov
  billingEnabled    Boolean // E1（false=整机免费）
  dailyFreeQuota    Json?   // E1 可选限额
  brandingJson      Json?   // E3 冠名（名称/logoFileId/欢迎语）
  accessibilityMode String  // off | senior（E4）
  loginMethodsJson  Json    // ["sms","qr","idcard","ssc"]（E6/E8 开关）
  reportVisibility  Json?   // E2 大屏模板开关
}
```

- **公开下发白名单**（Kiosk 无鉴权可读）：只下发渲染必需项（billingEnabled、accessibilityMode、loginMethods、branding 展示字段），**不下发**机构 ID、合同信息、配额内部参数——沿用终端配置已建立的白名单收窄先例。
- **Admin 页**：终端管理详情内加"部署形态"Tab（不新增一级菜单）；全部修改写 `terminal.deployment_profile.update` 审计。
- **Agent**：读卡器/扫码枪作为外设插件注册，心跳上报外设在线状态（Admin 外设管理页借此从空壳变真——顺手闭合审计矩阵 P2 缺口）。
- **机构关系**：政企客户作为 `Organization`（既有）入驻，其终端归属该机构（Terminal 已有机构关联）；政策供给走 Partner 后台，运营报表按机构隔离（Partner 只见本机构终端数据）。

## 六、数据安全与合规（政企版专属清单）

1. **个人信息最小化**：身份证明文零留存（仅 HMAC hash）；人脸默认不做；报表零个人明细；沿用会话纯内存、TTL 清理、脱敏展示。
2. **《居民身份证法》/《个人信息保护法》**：E6 上线前法务评审 + 隐私政策增补"身份证核验"专节 + 用户读卡前屏幕明示告知。
3. **等保（网络安全等级保护）**：政务项目普遍要求等保二级（部分三级）。现有安全基线（HMAC 签名 URL、AES-256-GCM 凭证加密、审计日志、防重放）是好底子，但**等保是体系认证不是代码特性**——列为 G4 前置工作项：差距评估 → 整改（日志留存时长、账号策略、备份制度）→ 测评机构认证。投标前只承诺"按等保 X 级标准建设"。
4. **数据归属**：合同明确运营数据归客户、平台仅受托处理；私有化档数据完全在客户侧。
5. **红线重申**：政企版一切能力不引入招聘闭环；求职登记只辅助打印不代办；政策只指引不承诺。

## 七、商务与交付模式

**报价结构（建议模板，具体价格商务定）**：

| 项 | 模式 |
|---|---|
| 终端设备（27寸一体机+奔图打印机+机柜） | 按台一次性（市场参考 1.8–2.5 万/台区间） |
| 软件平台使用费 | 按台/按年订阅 |
| 能力项选配 | E6 读卡（含硬件）/ E7 扫描 / E8 社保卡对接：单项报价 |
| 年度运维（E11 SLA） | 按台/按年，含耗材响应时效承诺 |
| 私有化部署（E10） | 项目制一次性 + 年维保 |
| AI 用量 | 包含额度 + 超量阶梯（后端已有 AI 用量统计支撑计量） |

**标准验收清单（随合同）**：设备上电自启 → 刷证/扫码/短信登录 → AI 诊断出报告 → 打印出纸 → 免费模式无支付元素 → 报表出数 → 告警触发与处理 → 断网恢复。全部映射到既有 verify 脚本 + 真机验收清单（`docs/device/production-deployment-and-windows-host-checklist.md` 扩展政企节）。
