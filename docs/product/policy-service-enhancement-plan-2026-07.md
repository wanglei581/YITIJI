# 政策服务板块商用化功能设计方案（2026-07，v2 已吸收 Codex 审查意见）

> 定位说明：本文档是 `commercial-grade-feature-plan-2026-07.md` §4.5「政策服务板块」与 `gov-enterprise-edition-design.md` E5「政策服务政企深化」的**详细设计展开**，不是另起的第二套标准。实施顺序、合规红线、验证门禁均沿用既有正式文档口径。
>
> 版本记录：v1 由 Claude 基于竞品调研 + 代码现状产出；v2 吸收 Codex analyzer 全量审查意见（事实修正 8 处、合规修正 5 处、技术方案重写 3 处、分期与上位 Phase 0 约束对齐）。
>
> 触发背景：用户判断当前 `/renshi` 政策服务页功能过于简单，不足以覆盖求职人群在求职过程中对政策信息的真实需求，要求基于同类产品调研补全为完整的商业化能力。

## 一、竞品与行业调研结论

### 1.1 同类产品盘点

| 产品/模式 | 核心功能 | 使用人群 | 解决的问题 |
|---|---|---|---|
| **人社自助服务一体机**（各地人社局，如深圳 1100+ 台） | 社保信息查询与打印、参保证明打印、灵活就业登记、失业金申领、待遇资格认证 | 参保人、失业人员、退休人员 | 免排队自助办事、24 小时可达 |
| **「家门口」就业服务站**（人社部推广，15 分钟就业服务圈） | 政策咨询、岗位发布、求职登记、职业指导、创业服务、社保查询、权益维护等 42 项服务；示范站点六大功能（需求排摸/岗位筹集/供需匹配/能力提升/援助帮扶/创业指导） | 社区求职者、就业困难人员、灵活就业者 | 就业服务下沉到社区，一站式受理 |
| **苏州「就业政策智能推送」平台** | 政策匹配（登录后与省厅数据比对，23 个匹配事项精准推送）、政策检索（85 个热点事项，含事项简述/政策依据/服务对象/办理条件）、模拟计算（技能提升补贴、社保个人缴费、年假等 4 项）、我的收藏 | 高校毕业生、就业创业人员 | 「人找政策」低效，用户不知道自己符合哪些政策 |
| **东软「政策计算器」**（南京/武汉人社已落地） | 政策知识图谱拆解（条件-程序-途径-标准结构化）、用户画像标签、智能匹配推演（政策找人）、个性化分析报告（权益测算+办理流程指引）、自然语言问答 | 高校毕业生、灵活就业人员、失业人员、中小微企业 | 信息不对称、政策覆盖盲区、推送不精准 |
| **国家大学生就业服务平台**（教育部+人社部） | 《全国高校毕业生就业创业支持政策清单》专栏、AI 助手「智慧小业」、政策宣传专栏 | 高校毕业生 | 权威政策集中入口 |
| **哈尔滨惠企政策精准推送平台** | 政策全量集成、快查快享、精准推送、免申即享 | 企业 | 企业侧政策红利触达 |

### 1.2 求职人群的真实政策需求（按人群拆分）

| 人群 | 高频政策/事项需求 |
|---|---|
| **高校毕业生**（毕业学年 + 离校 2 年内） | 一次性求职创业补贴（低保/零就业家庭/特困/残疾/助学贷款群体）、就业见习、档案托管与转递、报到落户、基层就业（三支一扶/西部计划/特岗，学费补偿+考研加分+定向招录）、灵活就业社保补贴 |
| **失业人员** | 失业登记、失业保险金申领条件与流程、技能提升补贴、就业困难人员认定、领金期间的社保与再就业帮扶 |
| **灵活就业人员** | 灵活就业参保登记、社保补贴申领指引、平台就业创业支持、创业担保贷款 |
| **返乡务工/农民工** | 技能培训与补贴、异地社保转移接续、劳动维权、返乡创业扶持 |
| **创业人员** | 创业担保贷款、一次性创业补贴、场地房租补贴、税费减免 |
| **就业困难群体** | 困难认定、「一对一」就业援助、公益性岗位、社保补贴 |

### 1.3 对本项目的借鉴取舍

**适合引入**（信息指引类，合规安全）：
1. 政策速配 / 政策找人 —— 用户自填条件 → 匹配**可能相关**的政策清单（苏州/东软的核心价值，即文档 A14/E5 的「政策 AI 匹配指引单」）
2. 补贴模拟测算（仅供参考，小范围试点） —— 苏州测算模式，但必须带地域/政策版本/有效期参数
3. 办事指南结构化 —— 事项简述/服务对象/办理条件/材料清单/办理地点/办理时长/官方入口（12333 事项库的字段口径）
4. 政策清单专栏 —— 按人群组织的权威政策清单（对标全国毕业生政策清单专栏）
5. 打印赋能 —— 本终端相对纯线上平台的**差异化优势**：政策指引单、材料清单、办事流程可现场打印带走
6. AI 政策问答 —— 仅基于已发布政策库的命中内容做解释，附来源与官方入口

**明确不做**（合规红线 / 越界能力）：
- ❌ 任何形式的补贴**申领办理**、进度查询、到账查询（需人社系统对接资质，属政务办理闭环）
- ❌ 对接省厅/人社实名数据做自动匹配（苏州模式依赖政务数据授权，本项目无此资质；改为**用户自填条件的本地匹配**）
- ❌ 「免申即享」「代办」「包过」类承诺表述；速配结果**禁止输出「符合/不符合」的资格结论**（平台无资格审核权）
- ❌ 社保缴费、待遇资格认证等政务办理事项（只做「怎么办、去哪办」的指引 + 官方入口二维码）

## 二、现状与差距（已按代码逐项核实）

### 2.1 已具备的底座（不推倒重做）

- Kiosk `/renshi` 政策服务页：4 Tab（就业政策/社保指南/就业登记/政策公告）+ 身份筛选 + 官方入口 QR + 收藏 + 打印跳转
- 后端 `PolicyPost` 模型（kind=`policy_guide`/`notice`，String 字段非数据库枚举）+ 完整审核状态机 + Partner CRUD + Admin 审核发布 + 审计
- Partner「政策公告管理」页、Admin「政策信息源」审核页
- 浏览/外部跳转/收藏记录闭环（targetType=`policy`，**仅对库内已审核发布记录有效**）
- AI 助手：真实 LLM 回答 + 政策意图分类注入 `/renshi` 跳转动作（回答未以已发布政策库为依据）
- 招聘会资料打印桥（`fair_material_print_bridge`）：服务端生成 FileObject → HMAC printFileUrl → `/print/confirm` → `Order.originType/originId` 服务端派生 —— **政策打印闭环可直接复用此模式**

### 2.2 主要差距（G1–G11）

| # | 差距 | 影响 |
|---|---|---|
| G1 | 「社保指南」「就业登记」两个 Tab 前端硬编码，来源与同步时间写死、**无法证明内容仍然有效**，扫码按钮全是 ComingSoon 占位 | 违反「不伪造能力」口径，内容无法运营更新 |
| G2 | 无政策速配/指引单闭环（A14/E5 未实现）；AI 助手虽是真实 LLM 回答，但**未以已发布政策库为依据**，存在编造政策的风险面 | 缺少同类产品核心价值「人找政策 → 带走一张指引单」 |
| G3 | 无补贴测算 | 用户最关心「大概能有什么支持」无法回答 |
| G4 | 前后端 audience 枚举不一致（前端 `flexible` vs 后端 `migrant`），后端 `migrant` 政策被降级为 `general` | 数据分类错误，已是 bug |
| G5 | 打印全部跳裸 `/print/upload`（该页会把未知 `source` 归一为 `document`，政策上下文丢失） | 打印订单无法回溯政策来源，闭环断裂 |
| G6 | `PolicyPost` 缺办理时长/适用地域/办理地点/生效失效期/文号/原文 URL 等字段 | 指南信息深度不够，无法支撑时效与来源治理 |
| G7 | 无政策浏览/跳转聚合统计端点；**政策来源打印、速配事件的数据目前根本不存在**（需先埋点再谈聚合） | 政企版续费依据（E2/L4）缺数据支撑 |
| G8 | Admin 审核页看不到正文全文；Partner 无机构分级权限（权限矩阵文档要求校方仅校内公告） | 审核有效性与权限矩阵不符 |
| G9 | `RenshiPage.tsx` 944 行，超工程红线（>800 行禁堆新功能） | 必须先拆分再增强 |
| G10 | 首页「补贴指引」深链到 `tab=social`（实为社保指南），语义错位 | 入口语义混乱 |
| G11 | 5 条内置政策（`BUILTIN_GUIDES`）用虚拟 ID，而服务端收藏/浏览/外跳只认库内已审核发布记录 → **内置政策的记录闭环实际是断的** | 「全链路记录」宣称与事实不符 |

### 2.3 与上位文档的阶段约束（必须遵守）

- `commercial-grade-feature-plan-2026-07.md` 明确当前 Phase 0「不扩范围、不塞新功能」；
- `gov-enterprise-edition-design.md` 要求 E5 在 Phase 0 与支付域完成后启动。

因此本方案将任务切为 **A0（收口期内允许：仅真实性/bug 修复）** 与 **A1 及以后（收口完成后启动）** 两档，不把新功能标为当前 P0。

## 三、目标信息架构（Kiosk 政策服务页）

页面仍为 `/renshi`（保留深链兼容），信息架构从「4 个平铺 Tab」升级为「1 个速配入口 + 5 个内容区」：

```
政策服务
├── 🎯 政策速配（顶部主入口，Phase B）
│     3 步自助问答（身份 → 状态 → 加分项）→ 可能相关的政策清单 → 打印政策指引单
├── Tab1 就业政策（现有 PolicyPanel 增强）
│     按人群筛选的政策库：毕业生/灵活就业/返乡务工/创业/困难群体/通用
├── Tab2 补贴测算（Phase B2，小范围试点）
│     首期仅 1 个测算器（技能提升补贴参考），参数版本化+审核发布
├── Tab3 办事指南（合并原「社保指南」+「就业登记」，后端化，Phase A1）
│     社保查询打印 / 失业登记 / 就业登记 / 档案转递 / 医保备案 / 社保卡 …
├── Tab4 政策公告（现有 NoticePanel，保持）
└── 底部：常用材料打印包（保留，接上打印桥）
```

交互与合规基线不变：仅信息指引 · 不代办；每条内容展示来源与时效信息；外部入口一律二维码/官方链接；触控目标 ≥48px，主按钮 ≥56px；竖屏 27 寸布局。

## 四、功能设计（按模块）

### F1 页面重构与真实性修复（Phase A0，收口期内可做）

- **无行为拆分** `RenshiPage.tsx`（944 行）为 `pages/renshi/` 目录：壳 + 各 Panel + components + data。单文件 ≤300 行目标，纯重构不改行为。
- 修正首页深链：「补贴指引」改为「社保指南」（仍指向 `?tab=social`，入口名与 Tab 一一对应；补贴指引内容由「就业政策」入口承载，避免两个入口指向同一 Tab 的同义重复）。✅ 已实施（2026-07-11）。
- audience 枚举对齐：后端 `POLICY_AUDIENCES` 增加 `flexible`（DTO 白名单 + shared 类型 + 前端映射三处同步）；前端人群 chip 补 `migrant`（返乡务工）；`fromPublished` 映射修正。
- **移除 ComingSoon 占位与硬编码假时效**：社保/就业登记面板中无真实官方 URL 的「扫码」按钮直接不渲染；硬编码「同步时间」文案删除或改为「内容整理参考，办理以官方为准」的诚实表述（后端化之前的过渡态）。
- G11 过渡处理：内置政策卡的收藏按钮在后端化完成前隐藏（或仅本地收藏并明示），不再出现「点了收藏但服务端静默失败」。

### F2 政策速配 + 政策指引单（Phase B，= A14/E5，政企版核心卖点）

**流程**（全程匿名可用）：

1. **3 步问答**（每步单屏大按钮，≤3 步符合触控口径）
   - 第 1 步 我是：高校毕业生（毕业学年/离校 2 年内）/ 失业人员 / 灵活就业 / 返乡务工 / 创业者 / 其他
   - 第 2 步 我的状态（按身份动态出题，均可选「不确定」）
   - 第 3 步 加分项（多选可跳过）
2. **匹配引擎（服务端确定性规则，非 LLM）**：输入为用户勾选项 + 终端所在行政区划；规则来自政策条目的结构化条件（见 F7 规则模型）。输出三档：
   - **可能相关** —— 展示依据的条件项
   - **需补充信息或向官方确认** —— 列出「不确定」的条件项
   - **按当前选项暂未匹配到相关政策** —— 附 12333 官方渠道兜底
   - ⛔ 禁止输出「符合 / 不符合」的资格结论（平台无资格审核权）。
3. **结果页**：可能相关的政策卡清单（依据条件、所需材料、办理路径、官方入口 QR）+ 醒目免责声明「结果仅为信息参考，是否符合以官方审核为准」。
4. **政策指引单打印**：一键生成 A4 PDF，**只包含用户勾选要带走的政策**（政策名、材料清单、办理地点、官方二维码、生成时间、免责声明）；**默认不打印用户的条件摘要**——低保/残疾/失业等状态属敏感画像，纸张遗留公共终端会造成泄露；如用户主动要求打印条件核对页，逐项确认后才打印。
5. **AI 增强（Phase C 可选开关）**：结果页「问 AI 助手」仅携带**匹配到的政策条目 ID**进入 `/assistant`（不携带用户勾选的敏感条件），AI 基于命中政策的已发布版本内容做解释。

**隐私设计（Critical，实现前必须落）**：
- 勾选条件仅存会话内存，速配完成/超时/离开页面即清除；**不落库、不保存到「我的」**（如未来要保存，需单独设计告知-同意-TTL-删除链路后另行评审）；
- 不采集身份证号/社保号/手机号；速配事件埋点只记录「发生了一次速配 + 命中政策 ID 列表」，**不记录勾选的条件明细**；
- 敏感条件不进入发给第三方 LLM 的任何 prompt。

### F3 补贴测算器（Phase B2，小范围试点）

Codex 审查确认：简化的「选证书等级→输出金额」模型不可商用（补贴受地域、政策版本、有效期、参保条件、证书日期、申领次数、受益主体制约；如就业见习补贴受益主体是**单位**而非求职者个人）。修正后的方案：

- **首期只上线 1 个测算器**：技能提升补贴参考测算（选择终端所在地市 + 证书等级 + 是否参加失业保险满 12 个月 → 输出「参考金额档 + 依据文号 + 有效期 + 官方申领入口」）。
- 其余测算器（灵活就业社保补贴等）在首个测算器验证运营模式后再扩。
- 见习补贴**不做个人金额测算**，改为「了解见习政策」信息卡（说明补贴发放对象是见习单位，个人获得的是基本生活费保障）。
- 参数模型 `PolicyCalcConfig`：地域（行政区划代码）、参数 JSON（String 存储）、依据文号、官方原文 URL、版本号、生效/失效日期、审核状态——**参数走审核发布流程，禁止任意 JSON 直接生效**；金额用整数分计算。
- 页面必须展示：依据文号 + 原文链接 + 参数更新时间 + 「以官方最新标准为准」。

### F4 办事指南后端化（Phase A1，消 G1）

- 新增 `kind='service_guide'`（`kind` 是 String 字段，扩展点在 DTO `POLICY_KINDS` 白名单、`assertKindFields()` 必填校验、三端类型映射与渲染分支）。
- 内容链路：Partner 录入 → Admin 审核 → Kiosk 展示（复用现有状态机）。**平台自营内容**：建立「平台运营」Organization 记录承接 `sourceOrgId` 必填外键（不绕过机构模型）；同时需为 Admin 新增创建/编辑端点（当前 Admin 只有审核/发布权）——**录入方与审核方分离，Admin 自建内容也不得自审自发**（双人原则）。
- 迁移策略：现有 7 条硬编码内容 + 5 条内置政策整理入库，**seed 后停在 `pending` 状态，由管理员人工核对来源与时效后才审核发布**；Kiosk 切读接口后删除硬编码。**API 失败时不用可能过期的内置正文兜底**（fail-closed，展示「内容加载失败 + 12333 兜底入口」），避免向用户呈现无法保证时效的政策内容。
- 官方入口 URL 增加服务端校验：HTTPS + 域名白名单（gov.cn 等官方域优先），防止钓鱼链接进入二维码。

### F5 政策打印闭环（Phase B，消 G5，复用招聘会资料桥模式）

~~v1 的 `/print/upload?source=policy&policyId=xxx` 查询参数方案不可用~~（`PrintUploadPage` 会把未知 source 归一为 `document`，参数不沿打印会话传递）。改为复用已验证的 `fair_material_print_bridge` 模式：

1. Kiosk 调 `POST /api/v1/policies/:id/guide-sheet`（或速配结果批量版），**服务端校验政策为已发布版本**后生成 PDF；
2. 服务端落 `FileObject`（短 TTL 临时签名）并返回内部 HMAC `printFileUrl`；
3. Kiosk 直接进入 `/print/confirm`（跳过上传页）；
4. PrintJobs 依据 `fileId` 的服务端关系派生 `Order.originType='policy_guide_sheet'` / `originId`——**不信任任何前端查询参数**；
5. 「我的打印订单」显示「来自：XX 政策指引」，运营统计从 `Order.originType` 聚合。

材料清单打印同模式。指引单模板需通过中文长文本溢出、二维码可扫性验证，并在 Windows + 奔图真机完成出纸验收。

### F6 AI 政策问答升级（Phase C）

- 现状：AI 真实 LLM 回答但不以政策库为据。升级为：政策意图命中时，检索已发布 `PolicyPost`（标题/摘要/条件的轻量检索，暂不引入向量库），仅将**命中条目的已发布版本内容**注入上下文，回答必须附「来源机构 + 同步时间」与跳转 action。
- 守门：无命中则回答「建议到政策服务页查看或咨询官方 12333」，禁止裸编政策；llm-guard 增加政策承诺类表述拦截（「保证领到」「一定符合」等）；AI 回复末尾固定免责句。

### F7 数据模型扩展（Phase A1 起分批；遵循仓库 JSON-as-String 双 schema 惯例）

`PolicyPost` 增量字段（全部可选，向后兼容，不动现有状态机；`Json?` 一律改为 `String` 存 JSON 字符串 + 读取 `JSON.parse`，与 `tagsJson` 惯例一致）：

```prisma
// kind 白名单扩展: 'policy_guide' | 'notice' | 'service_guide'（DTO 层，非 DB 枚举）
// audience 白名单扩展: + 'flexible'
regionCode      String?   // 行政区划代码，如 "370200"
regionName      String?   // 展示名，如 "青岛市"
processingTime  String?   // 办理时长，如 "10个工作日"
officeLocation  String?   // 线下办理地点
materialsJson   String?   // 结构化材料清单 JSON 字符串 [{name, note?}]
docNumber       String?   // 依据文号，如 "人社部发〔2026〕x号"
officialDocUrl  String?   // 官方原文 URL
effectiveDate   DateTime? // 政策生效日
expireDate      DateTime? // 政策失效日（到期 fail-closed 自动下架）
lastVerifiedAt  DateTime? // 最近人工复核时间（内容治理）
```

**政策版本化（Phase A1，Critical）**：当前编辑即回 `pending+draft`，会把已发布旧版直接下掉——商用政策库不可接受。引入版本模型：`PolicyPostRevision`（新版本独立走审核，审核通过后原子切换为当前发布版；旧版留档可审计）。速配/测算/AI 均只引用「当前发布版本」。

**速配规则模型（Phase B 前置设计）**：`[{key,label,required}]` 平铺标签不足以表达真实政策条件。规则独立建模（随政策版本走）：

```
PolicyMatchRule {
  policyId + revision           // 绑定政策版本
  regionCodes: string[]         // 适用地域
  effective/expire              // 有效期（继承政策）
  expr: 条件表达式树             // 支持 AND / OR / NOT / unknown 三值逻辑
  reasons: 原因码表              // 三档结果的解释文案 key
  version                       // 规则自身版本
}
```

条件求值遇 `unknown`（用户选「不确定」）时结果落「需向官方确认」档。规则首期随 seed/代码维护，Admin 可视化编辑器为 Phase C 择期项。

### F8 埋点与运营统计（事件采集随 Phase B 同步上线，展示 UI 可 Phase C）

- **先埋点**（Phase B）：速配完成事件（仅记录命中政策 ID 列表，不记条件明细）、指引单生成/打印事件（走 `Order.originType`）、测算使用事件。浏览/外跳沿用现有 BrowseLog/ExternalJumpLog。
- **后聚合**（Phase C）：`GET /admin/policy-stats`、`GET /partner/policy-stats`（本机构隔离）：按政策条目/时间段聚合浏览量、官方入口跳转量、指引单打印量、速配次数。Admin 工作台与 Partner 数据统计页各加「政策服务运营」卡片。
- 新端点配限流与审计；监控项含：速配错误率、过期政策展示告警。

### F9 内容治理与管理侧增强（Phase A1 为主）

- **Admin 审核详情**（A1）：审核表格加「查看详情」抽屉，展示 content 全文 + 全字段 + 版本对比，消 G8 审核盲区。
- **政策时效 fail-closed**（A1）：`expireDate` 到期自动 `publishStatus=expired`（复用现有队列/定时任务）+ Admin 到期前提醒列表 + 过期仍被引用（速配/测算规则）时告警。
- **Partner 机构分级权限**（A1，按 partner-permission-matrix）：校方仅校内公告、人社机构可发全域政策；实现为 Organization 机构类型 + service 层校验。
- **来源证据完整性**（A1）：发布机关、文号、官方原文 URL、适用区划、生效/失效日、最近复核时间为政企交付的内容质量基线；Admin 审核清单强制核对。
- 速配规则可视化维护（C 择期）。

## 五、合规与隐私边界（本方案的硬约束，全程有效）

1. **只做信息指引**：所有功能输出均为「指引/参考/入口」，办理一律引导至官方渠道（12333、当地人社、官方平台）二维码或链接。
2. **不碰政务办理与实名数据**：不做补贴申领、进度查询、待遇认证；速配基于用户自填条件，不采集身份证号/社保号。
3. **敏感画像保护（PIPL）**：低保、残疾、零就业家庭、失业等勾选项构成敏感个人信息画像——仅会话内存、用后即清、不落库、不打印、不进第三方 LLM prompt；埋点不记条件明细。
4. **文案白名单**：用「可能相关 / 需向官方确认 / 参考金额 / 建议咨询官方」；禁止「符合 / 不符合 / 已通过 / 保证 / 免申即享 / 代办 / 到账」。速配结果页、测算器、指引单三处必须带免责声明与来源标注。
5. **来源与时效治理**：每条政策/指南/测算参数展示发布机关 + 文号 + 原文链接 + 复核时间；过期 fail-closed 自动下架；API 失败不用过期内容兜底。
6. **不伪造能力**：无真实数据的入口不渲染占位按钮；未接入的功能不出现。
7. **打印文件安全**：指引单 PDF 走临时签名 URL + TTL，沿用现有文件安全规范；指引单不含用户敏感信息。
8. **官方 URL 白名单**：外部入口二维码的 URL 需服务端 HTTPS + 域名白名单校验。

## 七、验收标准（摘要）

1. `/renshi` 无硬编码假来源/假同步时间；无占位按钮；内置内容全部经人工核对入库或明示为参考。
2. 速配 3 步内出结果；三档表述合规（无资格结论）；「不确定」输入正确落「需向官方确认」档。
3. 指引单可打印、进打印订单、`Order.originType` 可回溯来源；真机出纸通过；PDF 中文/QR 验证通过。
4. 隐私：速配条件不落库、离开即清、不进 LLM、埋点无条件明细——逐项走查。
5. 所有新增文案通过合规守卫；每条内容有发布机关 + 文号/原文链接 + 复核时间。
6. 过期政策自动下架且速配/测算不再引用；API 失败 fail-closed。
7. 三端 typecheck/lint/build 全绿；policies verify + 迁移双 CI 通过；27 寸竖屏触控走查通过（主按钮 ≥56px）。

## 八、遗漏项自查（调研与审查中补充的点，均已并入上文）

1. 前后端 audience 枚举 bug（G4）与首页深链错位（G10）——Phase A0 先修。
2. 内置政策记录断链（G11）——Codex 审查发现的现状缺陷。
3. 政策版本化——编辑即下架旧版是商用政策库的硬伤，前置到 A1。
4. 政策时效 fail-closed——过期政策继续展示是政企客户最敏感的合规事故点。
5. 打印回溯断裂（G5）——复用招聘会资料桥模式，服务端派生来源关系。
6. 敏感画像隐私（PIPL）——速配条件的会话级保护、不打印、不进 LLM。
7. 测算参数治理——地方标准各地不同且常调整，参数必须版本化 + 审核发布 + 标注文号。
8. 无障碍与低数字素养适配——速配问答用大按钮单选、避免输入框；预留「问 AI 助手」语音入口（复用现有 2C+ 语音能力，择期）。
9. 12333 官方渠道兜底——任何无法回答/无数据场景，统一展示全国 12333 热线与官网入口。
10. 录入/审核分离——Admin 自建内容不得自审自发（双人原则）。
