## 一、需求总结与提炼（对用户诉求的结构化重述）

用户原始诉求提炼为 7 条：

1. **对标市场**：调研同类产品（人社一体机 / AI 简历 / AI 面试 / 自助打印 / 高校生涯平台），把它们的优势功能合规地吸收进本项目。
2. **定位收口**：确定首页各大板块下每个功能页面的**类型定位**（信息入口型 / 工具型 / 交易型 / 资产型 / 运营型）。
3. **全链路设计**：每个功能从 Kiosk 用户界面 → 后端 API → Admin 管理端 → Partner 机构端的功能、数据传输、权限一体化设计。
4. **登录体系**：把 C 端登录（短信 / 扫码）与政企场景身份核验一次性想清楚。
5. **支付体系**：补上目前全仓缺失的支付域（微信/支付宝扫码支付、订单、退款、对账、核销）。
6. **AI 全面注入**：每个页面都要有 AI 大模型能力，且必须"真正解决问题"，不做装饰性 AI。
7. **商用级标准**：可运营、可计费、可审计、可运维、可验收；遗漏之处由本方案补全。

**一句话产品定位（调研后修正版）**：市场上「人社一体机有渠道没 AI」「AI 简历/面试工具有能力没线下出口」「自助打印有支付闭环没内容」——三者从未在一台设备合体。本产品 = **AI 求职能力 × 打印物理出口 × 政企部署渠道** 的唯一合体，商业模式四层收入（见 §三），全程不触平台内投递红线。

## 五、登录与身份体系（现状 + 补齐路线）

**现状（已验证）**：
- C 端：手机号+短信验证码（虚拟键盘、纯内存会话、刷新即游客）；QR 扫码登录（Terminal Agent 127.0.0.1 本地代理持 claimToken，手机只 confirm，安全模型已过审）。
- B 端：Admin/Partner 用户名+密码 → JWT，`@Roles` 守门。
- 阻塞：`TencentSmsSender.sendCode()` 仍 NOT_IMPLEMENTED——**生产 C 端登录走不通，P0 中的 P0**。

**补齐路线**：

| 阶段 | 项 | 说明 |
|---|---|---|
| 已完成（代码） | 腾讯短信真发 | **2026-07-02 核实：`TencentSmsSender.sendCode` 已是完整真实实现**（TC3-HMAC-SHA256 签名 + 真调 SendSms + E.164 + 超时 + 脱敏日志 + 错误分类），并有 `verify:sms-provider` / `verify:sms-send` / `verify-member-sms-provider-errors` 三个脚本守门。剩余仅为**外部流程**：签名/模板审核、生产 CAM 密钥进 env、真号 E2E，然后置 `SMS_PROVIDER=tencent`。非开发任务，尽早提审 |
| P1 | 微信/支付宝扫码登录接真 | LoginPage 已有 UI 占位；走开放平台 OAuth，服务端换取 openid→绑定/新建 EndUser；**支付与登录复用同一 openid 体系**，先做微信 |
| P2（政企版） | 身份证读卡登录 | Terminal Agent 接读卡器（本地外设），走 Agent 本地代理模式（复用 QR 登录的 127.0.0.1 桥接安全模型）；仅取姓名脱敏+证件核验结果，不留存证件号明文 |

## 六、三端协同与数据流总架构（每个功能统一走这条链）

```
求职者(Kiosk 竖屏/手机H5)
   │  ①操作/上传/支付扫码
   ▼
Kiosk 前台(React) ──── /api/v1 ────►  NestJS API (43 modules)
   │                                   │  Prisma → PostgreSQL
   │                                   │  Redis(会话/队列/nonce)
   │                                   │  COS(临时签名URL, TTL≤30min)
   │                                   │  AI Provider(LLM/OCR/ASR/TTS, 密钥仅服务端)
   │                                   │  支付网关(微信/支付宝, 回调验签+幂等) ←— 新增 C-5
   ▼                                   ▼
Windows Terminal Agent ◄── claim/heartbeat ──┤   审计 AuditLog(全写操作)
  打印机驱动/扫描/读卡器/U盘                  │
                                             ▼
        Admin 后台(运营审核/计费配置/告警/对账) ↔ Partner 后台(数据供给/统计回看)
```

数据流通用契约（所有新功能必须遵守，均已有代码先例）：
1. 外部数据：Partner 提交/导入 → `pending+draft` → Admin 审核发布 → Kiosk 展示（带 source 字段五件套）。
2. 用户产物：Kiosk 操作 → 后端落 `FileObject`（purpose/assetCategory/expiresAt）→ 沉淀到 `/me/*` 对应资产分组 → 可回看/打印/删除。
3. 行为记录：只记浏览/收藏/外部跳转/打印/AI 调用（BrowseLog/ExternalJumpLog），永不记投递/筛选结果。
4. 钱：全部经 `Order`（唯一事实源），免费单也落库；支付凭证只在服务端；回调验签+防重放+幂等（复用 Webhook 已有的 HMAC+时间窗+nonce 模式）。
5. 审计：Admin/Partner 全部写操作、管理员访问用户文件、退款、价目修改必须写 `AuditLog`。

## 七、支付域 C-5 详细设计（本方案最大的新增域）

> 现状：`Order` 底座已迁入（type/payStatus/amountCents/taskStatus，打印任务创建即建 unpaid 订单），Admin 只读订单页已有。以下为增量设计。审计报告曾建议"8 月上线可先线下收款规避支付域"——**若走商用化，C-5 是第一优先级**；两者不矛盾：先免费/线下模式上线，C-5 作为上线后第一个大版本。

### 7.1 数据模型（additive migration）

- `Order` 扩展：`priceSnapshotJson`（下单时价目快照）、`payChannel(wechat|alipay|free|voucher)`、`paidAt`、`refundStatus`。
- 新增 `PaymentAttempt`：orderId、channel、prepayId、qrCodeUrl、status(created|scanned|paid|closed|failed)、channelTxnId、callbackRaw 摘要、幂等键。一单可多次尝试，成功唯一。
- 新增 `RefundRecord`：orderId、amountCents、reason(user_cancel|print_failed|admin)、status、channelRefundId、操作人、审计关联。幂等。
- 新增 `PriceConfig`（终端级价目，**后台手动动态调价，2026-07-02 用户确认为必做**）：terminalId?（空=全局默认价，非空=单终端覆盖）、item(bw_page|color_page|duplex_discount|ai_report|ai_interview_report|material_pack|template_resume)、amountCents、enabled、effectiveAt（可选定时生效）。Admin 改价即时生效、无需发版；每次改价写 AuditLog（改价人/时间/旧值/新值）；已下单订单以 `Order.priceSnapshotJson` 快照为准，改价不影响历史订单与对账。**政企点位可整机 free**。
- 新增 `VoucherGrant/VoucherRedemption`（与既有 BenefitGrant 打通）：券核销幂等+落库审计；"补贴券核销 ≠ 政府补贴到账"文案红线。

### 7.2 支付流程（微信 Native 为主，支付宝当面付同构）

1. `/print/confirm`（或 AI 报告付费点）→ `POST /api/v1/orders/:id/pay`：服务端按 PriceConfig 计价、生成价目快照、调用微信 Native 下单 → 返回 `code_url`。
2. Kiosk 屏幕展示动态二维码（价格、页数、倒计时、"扫码即同意服务协议"）；轮询 `GET /orders/:id/pay-status`（或 SSE）。
3. 微信回调 → 验签（平台证书）→ 防重放 → 幂等更新 PaymentAttempt→Order.payStatus=paid → **触发打印任务允许被 Agent claim**（未支付订单 Agent 不可 claim；免费单直接放行）。
4. 打印失败终态（Agent 上报 failed）→ 自动创建 RefundRecord 走原路退款 → 通知用户（`/me/notifications` 已有）→ Admin 告警。
5. 超时未支付 → 关闭 PaymentAttempt，订单可重新发起。

安全：证书/商户密钥只在服务端 env；回调必须验签+幂等（合规文档 §8.5 硬要求）；**不得把支付异常伪装成打印任务状态**；金额一律整数分。

### 7.3 三端功能

| 端 | 功能 |
|---|---|
| Kiosk | 确认页价目明示（对标市场：黑白 ¥0.3/页级）、扫码支付、支付状态、失败退款提示、`/me/print-orders` 显示支付状态与退款进度、电子凭证（可打印小票样式） |
| Admin | 价目管理（终端级、审计）、订单列表升级（金额/渠道/退款）、**手动退款（两步确认+审计）**、日对账页（渠道账单 vs Order 差异清单）、免费额度配置、券活动配置 |
| Partner | 不碰 C 端支付；仅看本机构相关的展示套餐账单（L4，后期） |
| Agent | claim 前校验 `payStatus in (paid, free)`；打印失败上报触发退款钩子 |

### 7.4 对账与财务（商用级必备，用户诉求中的隐含遗漏，本方案补全）

- 每日定时任务拉取渠道账单 → 与 Order/PaymentAttempt 对账 → 差异生成 Admin 待办。
- 支持导出对账 CSV；发票问题一期用"运营方线下开票"口径，页面留联系方式，不做电子发票系统。

## 十、分阶段开发计划（在既有 next-tasks P0 之上叠加，不冲突）

> 节奏原则沿用仓库铁律：一窗口一任务一分支；每任务先写目标/非目标/允许文件/验证方式；>30 行 diff 双模型审查；每域补 verify 脚本进双 CI。

### Phase 0：上线收口（当前进行中，按 next-tasks 执行，不扩范围）
短信审核提审→真号 E2E（代码已就绪）、正式域名 HTTPS、生产 PG/Redis/COS、Windows 真机出纸、法务定稿、小范围试运营。**本方案不向 Phase 0 塞任何新功能。**
产出验收：试运营 1 台终端跑通「登录→AI 诊断→优化→打印（免费/线下收款）→我的回看」全链路。

### Phase 1：商业化底座（试运营后第一个大版本，约 4-6 周）
1. **C-5 支付域**（§七全部）：模型→微信 Native→回调幂等→Kiosk 收银台→Admin 价目/退款/对账→Agent claim 门禁→`verify:payment` 脚本。
2. **付费点接线**：打印计费、A12 深度诊断报告、面试报告打印、模板简历打印。
3. **微信扫码登录接真**（与支付同 openid 体系）。
4. **隐私可见化**："打印完成后自动删除"上屏承诺 + 帮助中心说明（零后端改动，纯文案+既有清理任务）。
验收：真机扫码支付出纸；打印失败自动退款到账；对账页 0 差异；免费单落库；全链路审计可查。

### Phase 2：AI 深化与后台补齐（约 6-8 周，可并行两条线）
- 线 A（AI）：A9-A16（岗位语义搜索、逛展攻略、政策 AI 指引单、Roleplay 面试、上传智能分类、助手动作跳转）；AI 简历诊断 Phase 2 报告导出（next-tasks 已列）。
- 线 B（后台）：§八 P0 3-4 项（招聘会岗位明细/大屏字段录入）+ P1 5-11 项（Admin 用户、Partner 统计接真、死按钮、告警工单、耗材上报）。
验收：生产新建招聘会前台无空区；Partner 能看到自己数据的效果统计；告警有处理闭环。

### Phase 3：政企版与生态（按订单驱动，约 8 周+）

> 政企版已由用户确认立项（2026-07-02），完整设计见 [gov-enterprise-edition-design.md](./gov-enterprise-edition-design.md)：11 个可勾选能力项（E1-E11）、`TerminalDeploymentProfile` 配置面、G1 软件包（免费模式+运营报表+冠名+大龄模式+政策指引单，3-4 周可先行作为投标演示包）→ G2 外设包（读卡/扫描/告警工单）→ G3 渠道包（电子社保卡）→ G4 私有化部署包。

- 政企登录件：身份证读卡（Agent 桥接）、电子社保卡扫码；整机免费模式开关。
- 真机扫描链路（TWAIN/WIA 或 SMB 监听）+ 证件照（拍摄+AI 换底+排版打印）。
- 机构展示套餐（L4）计费与 Partner 账单。
- A17-A19；轻生涯测评；FairBooth 展位（如有招聘会现场订单）。
- 多终端规模化：远程配置下发、灰度发布、监控大盘。
