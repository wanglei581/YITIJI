## 〇、一句话定位

**校园招聘专区 = 用应届毕业生这个高频刚需人群，把"内容偏空的招聘会页"升级为"第三方校招信息入口 + 求职材料打印 + AI 求职服务"的旺季流量入口。**

仍严格属于"第三方/官方信息入口 + 工具服务"，**不是平台自营校招**，不触碰任何招聘闭环。

---

## 一、为什么要做 · 解决什么问题

1. **招聘会页"太空"的根因是缺一个高频刚需场景，校招正好补上。**
   应届生是线下一体机最典型、最密集的用户：秋招/春招集中期，他们反复打印简历（多份）、三方协议、报到证、成绩单、证件照——这正是本终端的核心收入功能。校招专区把"看信息"和"就地打印材料"连成一条线。

2. **给终端一个"为什么现在来"的季节性钩子。**
   校招有强季节性（秋招 9–11 月、春招 3–4 月）。带校招时间线/倒计时的专区，会在旺季持续把人引到终端；淡季也能沉淀政策/材料服务。普通招聘会列表没有这种节奏感。

3. **把信息入口与核心营收（打印 + AI 简历）真正打通。**
   现状「招聘会」基本只能"看 → 跳走"。校招专区可做成"看到这场校招会 → 打印参会企业岗位清单 + 准备多份简历 + 打印三方协议模板 + AI 诊断简历"，每一步都合规、且都创造使用量。

4. **给合作机构（高校就业中心 / 人社）一个数据落地理由。**
   有了面向应届生的专区曝光位，高校就业办更有动力把双选会数据推到我们终端，反过来充实内容、形成正循环。

5. **契合"AI 求职"定位。**
   应届生最需要 AI 简历诊断与求职路线规划（feature-scope 已列 P3），校招专区是这些 AI 能力最自然的入口。

## 二、合规判定与红线

校园招聘是合规敏感词，必须严格按"第三方信息入口 + 工具服务"实现。逐条对照 [compliance-boundary.md](../compliance/compliance-boundary.md)：

| 可以做 ✅ | 不能做 ❌（秒哒效果图里有，但**不可照搬**） |
|---|---|
| 展示第三方/官方校招会、应届岗位（带来源机构 / 外部 ID / 同步时间 / 来源链接） | 平台内一键投递、立即投递、内推投递 |
| 「查看岗位」「去来源平台投递」「扫码投递」「查看招聘会」「去来源平台预约」「扫码预约」 | 把简历直接投给企业 / 平台收简历转交企业 |
| 校招材料打印（简历、三方协议空白模板、成绩单、证件照）→ 走现有打印流 | 企业筛选候选人 / 面试邀约 / Offer 管理 |
| 校招政策问答（三方协议、报到证、档案、应届身份、落户补贴）→ AI 助手 | "AI 推荐内推""投递工作人员""一键报名进入"（秒哒图 `32-校企合作` 里有，禁止） |
| 记录：浏览、收藏、外部跳转、打印、AI 调用 | 记录投递结果、面试状态、候选人评价 |

**命名约束**：对外叫 **"校园招聘专区 / 校招服务"**，强调"信息入口 + 求职材料服务"。页面顶部必须挂标准合规说明条：
> 本系统仅展示第三方来源校招信息，不参与投递/报名流程，请前往来源平台办理。

结论：在严格边界下完全合规，不引入任何新的合规风险点（所有能力都是"看信息 / 打印 / 问 AI / 跳来源平台"）。

## 三、推荐方案（方案 A）

**形态**：招聘会页顶部"校园招聘专区"引导卡 → 进入独立 `/campus` 聚合页。复用现有招聘会 / 岗位 API，前端为主；直接解决"页太空"，季节钩子 + 材料打印闭环；合规风险最低。后续可平滑加到首页一级入口，不冲突。

被否决的备选（记录决策上下文）：
- **B 升级为首页一级入口**：曝光最高，但改信息架构、淡季首页位略空、且 feature-scope 约定业务入口走首页卡片需评审 → 暂不选，留作后续平滑升级。
- **C 只增强招聘会卡片不做独立专区**：改动最小，但解决不了"季节钩子 + 材料打印 + 政策"这条增量价值线 → 不选。

## 四、`/campus` 专区页结构

沿用项目设计系统（白底卡 + 1px 边框 + 轻阴影），**不照搬秒哒大渐变 / 毛玻璃**；21.5″ 触控友好，主按钮 ≥56px。

```
┌──────────────────────────────────────────────┐
│ PageHeader：校园招聘专区  ·  来源：第三方/官方   [返回] │
├──────────────────────────────────────────────┤
│ ① 季节横幅卡：2026 春招进行中 · 关键节点提示       │  ← 给"为什么现在来"
├──────────────────────────────────────────────┤
│ ② 校招时间线：网申→宣讲→双选会→签约→三方→报到     │  ← 横向时间轴 (P1)
├──────────────────────────────────────────────┤
│ ③ 本地校园双选会（复用招聘会数据，过滤 school 类）  │
│    卡片做厚：企业数 · 岗位数 · 面向届别 · 来源标签   │
│    [查看招聘会]                                  │
├──────────────────────────────────────────────┤
│ ④ 应届岗位精选（复用岗位数据，过滤"应届可投"）      │
│    来源平台标签 · [查看岗位][去来源平台投递][扫码]   │
├──────────────────────────────────────────────┤
│ ⑤ 校招材料与政策（合规营收钩子）                  │
│    打印：简历 / 三方协议模板 / 成绩单 / 证件照 → 打印流 │
│    政策问答：三方 / 报到证 / 档案 / 落户 → AI 助手    │
├──────────────────────────────────────────────┤
│ ⑥ AI 求职助手：应届简历诊断 / 求职路线规划         │
├──────────────────────────────────────────────┤
│ 合规说明条（必挂）                               │
└──────────────────────────────────────────────┘
```

## 五、分期计划

| 阶段 | 范围 | 数据/改动 | 估时 |
|---|---|---|---|
| **P0** | `/campus` 专区页（区块 ①③⑤⑥ 入口版）+ 招聘会落地页加专区入口 + **把招聘会列表卡片做厚**（企业数 / 岗位数 / 来源标签 / 届别） | 纯前端，复用现有 API，无 schema 改动，可演示 | ~2–3 天 |
| **P1** | 校招时间线/日历组件 ②；应届岗位精选 ④（接岗位 API + 应届过滤）；校招材料打印快捷入口跳现有打印流；政策问答跳 AI | 为 Fair/Job DTO 增可选 `audienceType` 标签（应届/实习/社招），合作机构后台可标注，提高过滤准确度（小改动） | ~3–4 天 |
| **P2** | AI 求职路线规划（应届版）；校招季本地订阅提醒（到点弹提示，不外发）；合作机构后台看校招专区曝光统计 | 视需求 | 择期 |

**复用清单**：`getJobFairs` / 岗位 API、`Card/Button/PageHeader/EmptyState/StatusBadge` 等公共组件、现有打印流 `/print/confirm`、AI 助手入口。新增仅 1 个页面 + 2 个小组件（季节横幅、校招时间线）。

**过滤实现（P0 不改 schema）**：复用 [JobFairsPage.tsx](../../apps/kiosk/src/pages/job-fairs/JobFairsPage.tsx) 现有关键词识别逻辑（`detectFairCategory` 已识别 校/高校/大学/应届/双选/校企 → `school_enterprise`）；岗位侧 P0 先用"应届可投"标签关键词过滤，P1 再用 `audienceType` 字段精确化。

## 六、明确不做（红线）

- ❌ 平台内投递 / 收简历 / 内推闭环；❌ 企业筛选 / 面试 / Offer；❌ "投递工作人员""AI 推荐内推""一键报名"（秒哒图有，禁止）
- ❌ 长期保存三方协议 / 成绩单等敏感件（走现有临时签名 URL + 自动清理）
- ❌ 大渐变 / 毛玻璃堆叠（按设计系统克制风格）
- ❌ 任何"一键投递 / 立即投递 / 平台投递"类文案

---

## 七、验收标准

- 按钮文案全部合规（仅用 §二 允许文案）
- 所有校招会 / 岗位展示来源机构 + 外部 ID + 同步时间 + 来源链接
- 专区页挂合规说明条
- 只记录浏览 / 收藏 / 跳转 / 打印 / AI 调用，不记录投递结果
- typecheck / lint / build 通过；合规禁词扫描通过
- 21.5″ 触控与移动端布局正常（主按钮 ≥56px，可点击区域 ≥48px）
