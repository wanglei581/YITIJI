# AI数字人引导员需求规划

> 状态：规划中  
> 目标阶段：Phase 9 UI Polish / AI助手增强  
> 当前结论：先做轻量 3D 数字人引导员，不在 Phase 7 数据链路阶段开发。

## 1. 背景与目标

AI求职打印服务终端面向人社大厅、高校就业中心、人才服务中心、招聘会现场等线下自助服务场景。用户通常处于“不会操作、需要咨询、想快速办理”的状态，因此前台需要一个更自然的引导层，帮助用户完成简历、打印、扫描、招聘会导览、政策查询和模拟面试等任务。

AI数字人不是为了装饰页面，而是承担“就业服务引导员”的角色：

- 降低首次使用门槛，让用户知道可以做什么。
- 在关键流程中解释下一步操作，减少卡住和误触。
- 承接 AI助手的对话能力，把自然语言问题转成页面跳转或操作引导。
- 在招聘会、政策查询、简历服务、打印扫描等场景中提升现场服务体验。
- 为人社局、高校、合作机构演示提供更直观的服务亮点。

## 2. 产品定位

推荐名称：

- AI就业服务引导员
- AI公共就业服务助手
- AI招聘会导览员

不建议使用：

- AI招聘官
- AI面试录用官
- AI候选人筛选官
- AI企业招聘助手

原因：当前项目定位是“AI求职打印服务终端”，不是自营招聘平台。数字人应服务求职者和公共就业服务流程，不应表达为替企业招聘、筛选或决策。

## 3. 推荐技术路线

### 3.1 第一阶段推荐方案

采用“本地 3D 模型 + Web 前端渲染 + AI助手意图路由”的轻量方案。

技术方向：

- 渲染：Three.js / @react-three/fiber
- 模型：VRM 或 GLB，本地静态模型优先
- 动作：idle、眨眼、点头、轻微转头、招手
- 语音：浏览器 TTS 或后续接入云端 TTS
- 嘴型：第一版用音频振幅驱动简单嘴型，第二版再升级 viseme
- 对话：复用 AI助手能力
- 页面联动：通过 intent router 打开简历、打印、招聘会、政策等页面

### 3.2 不建议第一阶段使用的方案

不建议一开始接入高仿真人 SaaS 数字人，例如真人视频数字人、复杂实时视频驱动方案。原因：

- 成本不可控。
- 网络依赖强。
- 接入和隐私合规复杂。
- 与页面交互、打印扫描、招聘会导览等本地流程结合不够灵活。
- 容易把产品重心从“公共就业服务”带偏到“炫技演示”。

这些方案可以用于宣传片或高端演示，但不建议作为产品内主数字人能力。

## 4. 功能范围

### 4.1 MVP 必做

第一版只做“可见、可听、可引导、可跳转”的数字人。

功能清单：

- 3D 数字人展示。
- idle 动作、眨眼、点头。
- 数字人文字气泡。
- TTS 播报。
- 播放语音时简单嘴型同步。
- 快捷问题按钮。
- 页面跳转 intent。
- 静音按钮。
- 关闭/收起数字人按钮。
- WebGL 不可用时降级为普通 AI助手文本框。

### 4.2 MVP 暂不做

- 不做人脸识别。
- 不做身份识别。
- 不录制视频。
- 不保存用户音频。
- 不把面试结果发送给企业。
- 不做企业端候选人推荐。
- 不做平台内招聘闭环。
- 不做真人级视频数字人。

## 5. 场景设计

### 5.1 首页

作用：告诉用户这台机器能做什么。

典型话术：

- “你好，我可以帮你做简历、查招聘会、打印材料、查询就业政策。”
- “你可以选择简历服务、打印扫描、岗位信息或招聘会服务。”

可触发 intent：

- `open_resume_service`
- `open_print_scan`
- `open_jobs`
- `open_job_fairs`
- `open_policy_service`
- `open_profile`

### 5.2 AI助手页

作用：数字人成为主要对话入口。

能力：

- 回答就业政策、招聘会、简历、打印扫描相关问题。
- 根据用户意图跳转页面。
- 对无法处理的问题给出明确边界说明。

示例：

- 用户问：“我想打印简历。”
- 数字人答：“我带你进入简历打印流程。”
- 系统跳转：`/print/upload` 或 `/resume/source`

### 5.3 简历服务

作用：引导用户上传、扫描、诊断、优化、打印简历。

引导点：

- 上传文件时提示支持格式。
- 扫描纸质简历时提示放置方式。
- 诊断报告页解释评分只是参考。
- 优化建议页强调不生成虚假经历。

合规话术：

- “简历诊断结果仅供求职准备参考，不代表真实招聘结果。”
- “优化建议基于你已有内容，不生成虚假经历。”

### 5.4 打印扫描

作用：减少用户在设备操作上的困惑。

引导点：

- 提示放置纸张。
- 解释 ADF 和平板扫描区别。
- 提示打印参数选择。
- 打印失败时解释可能原因。

示例：

- “请将纸质文件放入自动进纸器，并确认页面朝向。”
- “当前打印机可能缺纸，请联系现场工作人员或稍后重试。”

### 5.5 招聘会服务

作用：数字人作为“招聘会导览员”。

能力：

- 介绍招聘会时间、地点、主办方。
- 引导查看参会企业。
- 引导打开展馆导览图。
- 根据行业或企业名称提示展区。
- 引导打印活动资料。

合规边界：

- 只引导“扫码查看来源平台”。
- 不表达“帮你投递简历”。
- 不表达“企业可以查看你的简历”。

### 5.6 AI模拟面试

作用：后续亮点功能，第一阶段只规划，不立即开发。

能力规划：

- 根据用户简历和岗位方向生成面试问题。
- 数字人语音提问。
- 用户语音或文本回答。
- 输出表达、逻辑、专业匹配、岗位理解等训练报告。

合规话术：

- “本报告仅作为面试训练参考，不代表真实招聘结果。”
- “面试训练数据默认仅供本人查看，不提供给企业。”

## 6. 组件结构建议

建议新增前端组件目录：

```text
apps/kiosk/src/features/avatar/
  AvatarGuide.tsx
  AvatarScene.tsx
  AvatarBubble.tsx
  AvatarControls.tsx
  avatarIntents.ts
  avatarMessages.ts
  useAvatarSpeech.ts
  useAvatarIntent.ts
```

组件职责：

| 组件 | 职责 |
|------|------|
| `AvatarGuide` | 数字人总入口，控制显示、收起、静音、消息 |
| `AvatarScene` | Three.js / 3D 模型渲染 |
| `AvatarBubble` | 对话气泡和快捷问题 |
| `AvatarControls` | 静音、关闭、重播等控制 |
| `avatarIntents` | 定义页面跳转意图 |
| `avatarMessages` | 不同页面默认话术 |
| `useAvatarSpeech` | TTS 和嘴型同步 |
| `useAvatarIntent` | 意图识别和路由跳转 |

## 7. 数据流

```text
用户点击快捷问题 / 输入问题
  ↓
AI助手或本地 intent 识别
  ↓
返回 answer + intent
  ↓
AvatarBubble 展示回答
  ↓
useAvatarSpeech 播放语音
  ↓
如存在 intent，执行路由跳转或打开对应功能
```

第一阶段可以先不用真实大模型意图识别，使用本地规则：

- “打印简历” → `open_print_scan`
- “扫描简历” → `open_resume_service`
- “招聘会” → `open_job_fairs`
- “政策” → `open_policy_service`
- “岗位” → `open_jobs`

## 8. 页面接入范围

第一批接入：

- 首页
- AI助手页
- 简历服务入口页
- 打印上传页
- 扫描开始页
- 招聘会详情页
- 招聘会展馆导览页

不接入：

- Admin 后台
- Partner 后台
- 表格密集管理页
- 操作审计页

原因：后台是工作台，数字人会干扰高频管理操作。

## 9. 隐私与合规边界

数字人能力必须遵守以下约束：

- 默认不启用摄像头。
- 默认不启用麦克风。
- 麦克风使用前必须有明确提示。
- 不做人脸识别。
- 不做情绪识别。
- 不保存音频原始文件。
- 不保存视频。
- 面试训练报告只给求职者本人。
- 不把简历、面试报告、训练结果推送给企业。
- 不做候选人排序。
- 不做录用概率预测。
- 不使用“录用率”“企业推荐分”“候选人匹配排名”等文案。

允许文案：

- “岗位匹配参考”
- “面试训练报告”
- “表达能力参考”
- “求职准备建议”
- “扫码查看来源平台”

禁止文案：

- “AI推荐候选人给企业”
- “企业查看简历”
- “候选人筛选”
- “一键投递”
- “录用概率”
- “录用率”
- “Offer 预测”

## 10. 降级方案

必须支持降级，避免设备性能或浏览器能力不足时页面不可用。

降级条件：

- WebGL 不可用。
- 3D 模型加载失败。
- 设备性能过低。
- 用户关闭数字人。
- TTS 不可用。

降级表现：

- 隐藏 3D 模型。
- 保留普通 AI助手文本框。
- 保留快捷问题按钮。
- 页面功能不受影响。

## 11. 性能要求

Kiosk 一体机优先，避免 3D 效果拖慢主流程。

要求：

- 首屏不因数字人阻塞核心入口渲染。
- 3D 模型延迟加载。
- 模型大小控制在合理范围，第一版建议小于 10MB。
- 默认帧率目标 30fps 即可，不追求高画质。
- 数字人动画不能遮挡主操作按钮。
- 提供“关闭数字人”入口。

## 12. 分阶段计划

### Phase 9.1：方案落地与静态数字人

目标：建立数字人组件骨架。

交付：

- `AvatarGuide` 组件。
- 本地 3D 模型加载。
- idle 动画。
- 文字气泡。
- 静音/关闭控制。
- WebGL 降级。

### Phase 9.2：语音与嘴型

目标：让数字人能说话。

交付：

- TTS 播放。
- 简单嘴型同步。
- 重播当前提示。
- 页面默认欢迎语。

### Phase 9.3：功能引导与页面跳转

目标：让数字人能完成引导动作。

交付：

- 快捷问题按钮。
- 本地 intent router。
- 首页、简历、打印、招聘会页面接入。
- 常见问题话术库。

### Phase 9.4：AI助手融合

目标：数字人接入 AI助手。

交付：

- 用户问题输入。
- AI回答展示。
- 回答转 TTS。
- 根据意图打开页面。
- 无法回答时给出边界说明。

### Phase 9.5：AI模拟面试官

目标：做成求职训练亮点。

交付：

- 根据简历和岗位方向生成问题。
- 数字人语音提问。
- 用户文本/语音回答。
- 生成训练报告。
- 报告可保存、打印。

## 13. MVP 验收标准

第一版通过条件：

- 首页能显示数字人。
- 模型加载失败时页面仍可用。
- 数字人可收起、可静音。
- 数字人能显示文字气泡。
- 数字人能播放 TTS。
- 数字人能通过快捷问题跳转到简历、打印、招聘会、政策页面。
- 不出现招聘闭环禁用文案。
- 不启用摄像头识别。
- 不保存音频/视频。
- `pnpm lint` 通过。
- `pnpm typecheck` 通过。
- `pnpm build` 通过。
