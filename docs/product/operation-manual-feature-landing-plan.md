## 一、结论

手册中最适合本项目吸收的不是“招聘平台能力”，而是围绕一体机求职工具的几类能力：

- 打印前敏感信息检查与遮挡。
- 上传体检、统一转 A4、打印材料包。
- 简历解析后的逐字段人工修正。
- 异常事件时间线、终端推荐与任务改派。
- 模板库、按岗位方向优化简历，但必须先完成合规改名和数据流收口。

第一批 MVP 应聚焦“四件可用能力”：

1. `Kiosk` 打印前 PII 检查。
2. `Kiosk` 上传体检 + 统一 A4 + 打印材料包。
3. `Kiosk` 简历字段人工修正 + 原文对照。
4. `Admin` 异常事件时间线。

暂不把“按岗位方向优化简历”放入第一批 MVP。该能力有价值，但最容易滑向招聘闭环，应作为第二批，在话术、字段、数据流三层合规改造完成后再上线。

用户截图中的能力也应按同一原则处理：不能做“应用广场式”平铺，而应收敛成一个面向求职者本人的 `AI求职材料中心`。该中心的核心价值是让用户在一体机现场完成“上传 / 扫描 -> 分析 -> 修正 -> 优化 -> 生成材料 -> 隐私检查 -> 打印带走”。

## 二、截图功能纳入评估

### 2.1 建议优先纳入

| 截图功能 | 项目内命名 | 用户群体 | 解决的问题 | 优先级 |
|---|---|---|---|---|
| 简历分析 | 简历体检 | 初次使用 AI 简历服务的求职者 | 不知道简历哪里有问题 | P0 |
| 简历信息抽取 / 结构化抽取 | 简历结构化解析 | 上传 PDF / Word / 扫描件的求职者 | 简历内容难复用、难编辑、难套模板 | P0 |
| 简历优化 / 简历内容优化 | 简历优化建议 | 已有基础简历但表达弱的求职者 | 内容冗长、重点不清、缺少量化表达 | P1 |
| AI 简历生成 | 简历初稿生成 | 应届生、转岗、无完整简历用户 | 从零写简历门槛高 | P1 |
| AI 面试助手 / AI 面试官 | 面试训练 | 有明确面试准备需求的求职者 | 缺少低成本练习和反馈 | P2 |
| Java 面试陪练 | 技术岗专项陪练 | 技术岗求职者 | 需要垂类题库和表达训练 | P3 |
| 职业规划 | AI 求职路线规划 | 应届生、转岗、离校未就业人群 | 不知道先看什么岗位、先补什么材料 | P2 |
| 职业证件照 | 求职证件照 | 现场急需材料的求职者 | 求职材料不完整，需要现场补齐并打印 | P2 |

### 2.2 可做但必须改造

| 截图功能 | 改造后命名 | 改造要求 |
|---|---|---|
| HR 知识问答 | 求职政策问答 / AI 求职助手 | 不提供企业 HR 管理、用工管理、候选人筛选类回答 |
| 简历 ATS 智能优化 | ATS 友好度检查 / 岗位方向优化 | 不接真实 ATS，不承诺通过率，不产生投递状态 |
| 人岗匹配度判断 | 岗位适配参考 | 结果只给求职者本人，不开放给合作机构或企业 |
| 生成优化岗位 JD 信息 | 岗位要求拆解 | 只解释第三方已有岗位，不替企业生成招聘 JD |
| 找工作-岗位选择大师 | 岗位选择建议 | 只推荐查看方向和材料准备动作，投递仍跳来源平台 |
| 简历合规虚假审查 | 简历风险提醒 / 表述一致性检查 | 不使用“虚假判定”“诚信评分”“候选人审查”等表达 |
| 业务陪练（知识库） | 面试沟通陪练 / 求职话术陪练 | 不做企业销售、招聘运营或员工培训场景 |

### 2.3 明确不做

| 截图功能 | 不做原因 |
|---|---|
| AI 自动招聘机器人 | 偏企业招聘自动化，容易触碰候选人推荐、筛选和招聘闭环 |
| 问答式生成职位 | 偏企业岗位发布 / 招聘供给侧能力 |
| 生成优化岗位 JD 信息（企业侧版本） | 是企业招聘工具，不是求职者材料工具 |
| 企业侧 ATS / 简历排序 / 候选人筛选 | 直接触碰项目合规红线 |

### 2.4 外部背景判断

- 2026 届全国普通高校毕业生规模预计 1270 万人，同比增加 48 万人，说明高校毕业生就业服务仍是高频刚需。
- 人社部、教育部 2026 年“就业公共服务进校园百日行动”覆盖政策、招聘服务、职业指导、困难帮扶、创业服务等场景，和本项目“线下公共就业服务一体机”的定位吻合。
- 因此，本项目应优先增强求职者现场可完成、可打印、可带走的服务，而不是建设招聘方闭环系统。

## 三、合规红线与改造规则

本项目仍是“AI 求职材料工具 + 打印扫描 + 第三方/官方来源入口”，不是招聘平台。以下能力禁止开发：

- 平台内一键投递、代投。
- 平台内收取求职者简历并转给企业。
- 企业端候选人筛选、面试邀约、Offer 管理。
- 候选人推荐给企业。
- 自营招聘闭环。
- 企业自主发布岗位并直接收简历。

岗位和招聘会动作只能外链化：

- 岗位：`查看岗位`、`去来源平台投递`、`扫码投递`。
- 招聘会：`查看招聘会`、`去来源平台预约`、`扫码预约`。

手册中涉及“岗位定制中心”的原词必须改造：

| 原词 | 改为 | 原因 |
|---|---|---|
| 投递城市 | 意向城市 / 目标城市 | “投递”暗示平台参与投递 |
| 投递方向 | 目标岗位方向 / 求职方向 | 避免招聘闭环语义 |
| 岗位定制材料 | 按岗位方向优化简历 | 定位为本人材料工具 |
| 一键生成 | 生成 / 生成优化版 | 项目禁用“一键”类招聘暗示 |
| 求职信投递 | 自荐信 | 仅供本人打印或下载 |
| 匹配度 / 命中率 | 可保留，但必须标注“仅供参考，不代表录用或投递结果” | 避免被理解成招聘结果 |

后端字段也必须避开招聘闭环语义，禁止出现 `applyStatus`、`deliveryResult`、`candidate*`、`submitToCompany` 等字段。优化产物只能归属求职者本人，不能出现指向企业侧的外发字段。

## 四、专家评审结论

### 产品合规评审

可直接落地：PII 检查、材料包、统一 A4、上传体检、字段修正。  
可改造后落地：按岗位方向优化简历、模板库、自荐信、参考匹配度。  
必须禁止：站内投递、候选人管理、企业查看简历、面试邀约、Offer 管理。  
最大风险：把“按岗位方向优化简历”做成变相招聘闭环。

补充结论：截图中的能力可以加入，但不能以“应用市场 / 招聘专区”的形式复制到一体机。项目内只保留面向求职者本人的材料工具入口，所有产物只能进入“我的材料、下载、打印、外部来源跳转”，不能进入企业收件箱、候选人池或平台投递状态。

### UI/UX 评审

一体机应是“服务中心式 Kiosk”，不是“信息大盘式 Kiosk”。

- 底部导航继续固定为：首页 / AI助手 / 我的。
- 首页只放业务入口和轻状态，不放 KPI、大表格、完整识别结果。
- 单屏只做一件事：上传、体检、字段修正、模板、打印参数、PII 检查、支付、进度分屏完成。
- 主触控按钮不小于 72px，避免依赖悬停和复杂文本输入。
- Kiosk 使用大卡片、短路径、强状态；桌面后台才承接复杂表格和运营指标。

### 技术架构评审

现有架构可以承接这些能力，但不能继续只靠页面 state 拼功能。优先补四层基础：

1. 求职者资产归属层：`FileObject`、`AiResumeResult`、`PrintTask` 需要串到 `EndUser`。
2. 简历工作区层：承接字段修正、版本、模板产物、目标岗位方向。
3. 材料处理层：承接上传体检、PII 扫描/遮挡、材料包。
4. 订单支付层：承接学生免费、校园卡、退款、支付异常时间线。

短期仍建议沿用 `services/api + BullMQ`，不要为了形式强行把任务迁到空壳 `services/worker`。等材料处理、PII 遮挡、批量渲染稳定后，再迁移 processor。

AI 接入原则：大模型只负责结构化抽取、诊断建议、改写候选文本和面试反馈，输出必须是可校验的 JSON / 字段结果。最终简历、材料包、证件照排版、PDF/DOCX 版式由后端模板和渲染链生成，不能让模型直接生成不可控的最终版式文件。

安全前置原则：材料处理任务必须绑定 `EndUser` 或匿名一次性访问凭证；匿名任务不能只靠 `taskId` 查询；过期任务必须在读取路径失效并可被 cleanup；上传原文、手机号、邮箱、地址、身份证等敏感参数不能以开放 blob 形式长期落库。

## 五、推荐信息架构

### Kiosk

固定三 Tab：

- 首页：AI 简历服务、打印扫描、岗位信息、招聘会服务、设备轻状态、最近任务轻提示。
- AI助手：咨询、路径分发、服务快捷入口。
- 我的：我的材料、AI 记录、打印记录、外部浏览/跳转记录。

核心流程一：AI 简历服务

`首页 -> AI简历服务 -> 上传/扫描简历 -> AI解析与诊断 -> 字段人工修正 -> 优化建议 -> 模板选择 -> PII检查 -> 打印`

核心流程二：打印扫描

`首页 -> 打印扫描 -> 上传/扫码/U盘/扫描 -> 上传体检 -> 统一A4 -> 单文件打印或打印材料包 -> 打印参数 -> PII检查 -> 确认/支付 -> 打印进度`

核心流程三：岗位/招聘会

`首页 -> 岗位信息/招聘会 -> 列表 -> 详情 -> 去来源平台投递/扫码投递 或 去来源平台预约/扫码预约`

### Admin

- 打印机/终端管理：状态、暂停接单、维护备注、可用性。
- 告警中心：异常事件时间线、重新下发、改派、处理记录。
- 订单管理：支付方式、退款、学生免费/校园卡，需真实订单域后再做。
- 工作台：只展示已有真实统计端点的指标，不编造。

### Partner

本批不新增 Partner 侧复杂能力。Partner 继续负责岗位/招聘会数据源维护、Excel/API/Webhook 接入、同步日志。

## 六、阶段开发方案

### Phase A：基础补洞

目标：让后续“我的材料、历史复用、材料包、字段修正”不再停留在页面态。

- `services/api`：为求职者文件、AI 结果、打印任务建立 `endUserId` 归属。
- `apps/kiosk`：登录态存在时上传文件和 AI 任务绑定当前用户；未登录仍允许临时匿名流程。
- `services/api`：新增材料处理域骨架，先支持体检和 PII 扫描任务状态。
- 验收：登录用户上传文件后可在“我的”查到；匿名文件仍按 TTL 清理；管理员访问仍走审计。

### Phase B：Kiosk 第一批可用能力

目标：让一体机现场能跑通“上传 -> 体检 -> PII 检查 -> 打印”。

- 上传体检：加密、超大、格式、页数、是否可 A4 化。
- 统一 A4：所有打印材料默认进入 A4 输出参数；禁止 A3。
- PII 检查：识别手机号、邮箱、身份证片段、地址等，用户选择保留/遮挡。
- 打印材料包：多文件选择、排序、页数汇总、合并后进入现有打印任务。
- 简历字段修正：AI 解析结果页增加字段编辑、保存、版本对比。
- 验收：在 21.5 寸一体机完成真实上传、体检、PII 确认、打印；异常有清晰回退。

### Phase C：Admin 运营闭环

目标：把手册的服务保障能力下沉到后台。

- 异常事件时间线：打印任务、Agent 上报、重试、失败、改派、人工处理串成一条线。
- 终端粗粒度推荐：先基于在线、离线、缺纸、活动任务，不承诺耗材最优。
- 任务改派：校验目标终端在线、未暂停接单，再重新下发。
- 维护备注 / 暂停接单：暂停后调度不再派单。
- 验收：后台能定位一笔失败任务的完整事件，能改派到另一台在线终端。

### Phase D：合规改造后的简历增强

目标：上线“按岗位方向优化简历 + 模板库”，不触碰招聘闭环。

- `apps/kiosk/resume/target`：字段统一为目标岗位方向、意向城市、求职场景。
- `services/api`：优化产物归属本人，只能进入我的材料、打印或下载。
- 模板库：按岗位类型、风格、页数筛选；输出 A4 兼容版式。
- 自荐信：只作为本人材料，不提供企业发送入口。
- 验收：全链路无站内投递、无投递状态、无企业接收字段。

### Phase E：订单支付与校园场景

目标：学生免费、校园卡、退款、对账成为真实订单域能力。

- 新增或完善 `Order`、`PaymentAttempt`、`Refund`、`BenefitGrant` / 校园卡交易记录。
- Kiosk 支持学生免费/校园卡支付方式。
- Admin 支持订单查询、退款、对账、支付异常时间线。
- 验收：免费单也落库审计；退款幂等；不把支付异常伪装成打印任务状态。

## 八、UI/UX 设计要点

### 秒哒参考图使用规则

秒哒只作为 UI/UX 参考，不作为工程继承对象。可迁移的是流程组织方式，不迁移招聘闭环能力。

可迁移：

- 上传方式卡 + 三步说明 + 固定主 CTA。
- 当前会话文件列表、上传状态、删除/预览模式，但只限本次会话。
- 设备状态横幅、打印须知卡、价格说明卡，但字段必须来自真实数据。
- 模板卡片 / 材料卡片浏览模式，用于求职信、面试感谢信、作品集封面等打印型材料。

禁止迁移：

- 企业端、候选人详情、企业看板、企业 ATS、平台内投递闭环。
- 公共终端首页常显个人资产、优惠券、积分、打印次数等信息。
- A3/B5、边距、打印质量、AI 打印优化、打印机手动选择等高密打印参数。
- 智慧校园、本地就业专区、城市活动 banner 等与材料检查链路无关的大块内容。

### Phase B Kiosk 顺序

Phase B 不新增底部 Tab，也不新增“应用广场”。入口放在现有 `AI 简历服务` 与 `打印扫描` 二级流程里。

推荐顺序：

1. 来源选择：本机上传、扫码上传、U 盘、扫描。
2. 上传体检：先判断格式、大小、页数、加密、是否可 A4 化；阻断项先拦住。
3. 材料处理确认：只在“需处理”时出现，说明将统一 A4、旋转、裁边或提示重传。
4. 多文件材料包：如有多个文件，在体检之后、PII 之前完成勾选、排序、页数汇总。
5. PII 检查：只处理敏感信息保留/遮挡，不混入打印参数。
6. 打印参数：只保留份数、黑白/彩色、单双面、方向等高频项；弱化自定义页码等触控键盘输入。
7. 打印确认：展示文件摘要、隐私处理摘要、费用和设备状态，再进入打印进度。

长阅读页要求：上传体检结果页、PII 检查页、打印确认页需要持有 Kiosk busy/reading 状态或刷新空闲计时，避免 180 秒待机宣传屏打断用户确认。

### 上传体检页

- 文件列表显示：文件名、大小、页数、格式、体检状态。
- 状态分为：通过、需处理、阻断。
- 阻断示例：加密文件、格式不支持、过大。
- 需处理示例：图片模糊、边缘倾斜、尺寸非 A4。
- 主按钮：`继续打印` / `重新上传` / `处理后继续`。
- 页数未知时显示“页数待识别”，不得写死为 1 页污染费用和双面建议。
- 未接入的扫码上传 / U 盘导入必须是不可用态或明确待接入，不做可点击死页。

### PII 检查页

- 插入在打印参数确认前或确认页前。
- 顶部显示全局步骤：上传文件 -> 材料检查 -> 打印设置 -> 确认打印。
- 顶部显示命中数量和风险等级。
- 列表项包含：类型、命中位置、建议动作。
- 用户动作：`保留`、`遮挡`。
- 多命中场景提供批量动作：`按建议处理`、`全部保留`，再允许逐项微调。
- 默认建议遮挡身份证片段和详细住址，手机号/邮箱由用户确认。
- 必须提示：检查仅用于本次打印前确认，不向第三方发送。

### 打印材料包页

- 多文件勾选与排序。
- 每个文件显示页数、A4 处理状态、体检状态。
- 底部固定显示总页数、预计用纸、预计费用。
- 生成后的材料包仍走签名 URL 和 TTL 清理。

### 字段人工修正页

- 左侧原文/解析来源，右侧结构化字段。
- 支持逐字段编辑、保存、恢复建议值。
- 保存后进入版本对比。
- 不把字段修正结果发送给企业。

### Admin 异常事件时间线

- 左侧任务摘要：文件名、终端、打印参数、状态、错误码。
- 右侧时间线：提交、领取、下载、校验、打印、失败、重试、改派、人工处理。
- 操作：重新下发、改派、标记已解决、添加备注。

## 九、验收标准

### 合规验收

- 无“一键投递 / 立即投递 / 平台投递 / 投递简历 / 企业收简历 / 候选人管理”等文案。
- 岗位和招聘会只保留外部来源跳转。
- 任何简历、扫描件、材料包都不发送给企业。
- PII 检查和遮挡结果不外发第三方。

### Kiosk 验收

- 21.5 寸触控下主流程不依赖鼠标悬停。
- 主按钮清晰，关键操作可单手点击。
- Phase B 主操作按钮目标高度按 72px 设计；次级图标按钮不得承担关键确认动作。
- 上传、体检、PII、打印参数、确认、进度都有 loading/error/empty/retry 状态。
- 体检 / PII / 确认页不会被待机宣传屏打断。
- 未实现能力必须标注“开发中/流程演示”，不得伪装为可用。

### 技术验收

- 文件访问仍使用临时签名 URL。
- 敏感文件和材料包有 TTL 清理。
- 管理员访问用户文件有审计。
- 队列任务可重试、幂等，重复消费不重复出纸。
- API 合同测试覆盖禁用字段和禁用文案。

### 真机验收

- 在一体机完成上传、PII 检查、打印参数、真实出纸。
- 打印机离线、缺纸、卡纸、文件过期、哈希不匹配都有明确提示。
- 改派只允许派到在线且未暂停接单的终端。

## 十、当前不建议立即做

- 首页 KPI 化。
- Kiosk 大表格多维筛选。
- 新增第四个底部 Tab“材料中心”。
- 首页 6 到 9 个 AI 能力大卡的应用广场。
- 把 AI 助手文字对话做成进入材料处理流程的必经步骤。
- 在 PII 页展示完整手机号、邮箱、住址原文或做成复杂表格。
- Phase B 就上线字段逐项编辑、左右对照、版本对比重编辑器。
- 站内投递、站内预约、候选人管理。
- 学生免费/校园卡支付，直到订单域补齐。
- 历史材料跨设备复用，直到 `EndUser` 资产归属补齐。
- 精细耗材最优调度，直到 Agent 心跳真实上报耗材、纸盒、SN 等字段。
