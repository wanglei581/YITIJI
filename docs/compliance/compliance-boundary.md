## 一、背景说明

当前公司**暂无人力资源服务许可证**。

本系统的定位是：AI求职打印服务终端，提供工具服务（简历、打印、扫描）和第三方信息入口（岗位、招聘会），**不构成自营招聘平台**。

开发前必须阅读本文档，确认不触碰合规红线。

---

## 二、绝对禁止的功能

以下功能**一律不得开发**，无论产品或设计如何要求：

| 禁止功能 | 说明 |
|---------|------|
| 平台内一键投递 | 用户不能在本平台内直接将简历投给企业 |
| 平台内收取求职者简历给企业 | 不能将用户简历转发给企业方 |
| 企业端候选人筛选 | 不能提供企业筛选候选人的界面或接口 |
| 企业端面试邀约 | 不能提供企业发起面试邀约的功能 |
| 企业端 Offer 管理 | 不能提供 Offer 发放和跟踪功能 |
| 候选人推荐给企业 | 不能主动将候选人信息推送给企业 |
| 自营网络招聘闭环 | 整体不构成招聘平台 |
| 企业自主发布岗位并直接收简历 | 企业不能在本系统内发布岗位并直接收简历 |

## 四、数据边界要求

### 4.1 岗位数据必须包含的字段

```typescript
interface JobData {
  source_org_id: string;      // 来源机构 ID
  external_id: string;        // 来源平台岗位 ID
  source_name: string;        // 来源机构名称
  source_url: string;         // 来源平台链接
  sync_time: DateTime;        // 数据同步时间
  review_status: string;      // 审核状态
  publish_status: string;     // 发布状态
}
```

### 4.2 招聘会数据必须包含的字段

```typescript
interface JobFairData {
  source_org_id: string;
  external_id: string;
  source_name: string;
  source_url: string;
  sync_time: DateTime;
  review_status: string;
  publish_status: string;
}
```

### 4.3 岗位详情页必须展示

- 来源机构名称
- 数据同步时间
- 外部 ID（可折叠展示）
- 外部投递链接（去来源平台投递按钮）
- 数据来源说明（例："本数据来自 XXX 平台，本系统不参与招聘流程"）

### 4.4 系统只允许记录的用户行为

| 允许记录 | 禁止记录 |
|---------|---------|
| 浏览岗位 | 企业筛选结果 |
| 收藏岗位 | 第三方后续处理信息 |
| 外部跳转（记录跳转行为，不记录投递结果） | 面试邀约记录 |
| 打印记录 | 候选人评价 |
| AI服务调用记录 | 企业端任何数据 |
| 招聘会浏览记录 | 第三方平台后续结果 |
| 政策浏览记录 | 官方平台办理结果 |

2026-06-12 已接真的浏览与跳转记录边界：

- `BrowseLog` 仅记录登录会员本人浏览过的岗位 / 招聘会 / 政策目标快照。
- `ExternalJumpLog` 仅记录登录会员本人打开过的来源平台投递入口、来源平台预约入口或官方入口。
- `targetTitle`、`sourceName`、`sourceUrl`、`externalId` 必须由服务端从已审核且已发布的目标对象补齐，前端不得伪造。
- 记录失败不得阻断用户浏览、查看二维码或打开来源平台入口。
- 不记录第三方平台上的投递结果、预约结果、企业查看、筛选、面试或任何第三方后续处理信息。

### 4.5 企业展示（CompanyProfile）边界（2026-06-12 新增，长期红线）

企业展示 = 企业风采 / 来源企业与岗位导览，**不是招聘平台、不是企业发岗平台**：

| 只能做 | 不能做 |
|--------|--------|
| 展示来源机构提供并经管理员审核发布的企业信息（含 logo/封面/宣传片，媒体随企业资料一并审核） | 企业自主入驻、企业 HR 后台 |
| 企业关联「已审核已发布」岗位的导览 | 平台内投递、企业收简历 |
| 「去来源平台投递 / 扫码投递 / 去来源平台查看」外部引导 | 企业查看/筛选候选人、面试邀约、Offer 管理 |
| 详情页指标（在招岗位/所在城市/员工规模/展位号）按 Admin 开关展示，数据为空不造假 | 写死统计数字、展示假企业/假岗位/假视频 |
| 岗位匹配参考：仅本人三档参考（较高/中等/偏低），不代表录用结果 | 百分比匹配度、录用概率、结果回传企业 |

- 企业数据必须含来源四要素（source_org_id / external_id / source_name / source_url），默认 `pending + draft`，管理员审核通过后才能发布。
- Partner 仅维护本机构来源数据（导入/编辑一律回 pending+draft 强制重审）。
- 浏览企业 / 打开企业来源页进入 §4.4 的 BrowseLog / ExternalJumpLog（targetType=company_profile，action=external_open），同样不得记录任何投递/办理结果。

## 五、文件安全要求

用户文件（简历、扫描件、证件照、身份证复印件）属于敏感数据，必须遵守：

| 要求 | 说明 |
|------|------|
| 临时签名 URL | 所有文件访问必须使用带时效的签名链接，不得使用永久公开链接 |
| 有效期限制 | 敏感文件签名 URL 有效期建议不超过 30 分钟 |
| 自动清理 | 系统应支持配置文件自动过期清理策略 |
| 管理员访问日志 | 管理员每次访问用户文件必须记录日志 |
| 不长期保存敏感文件 | 身份证、简历等文件不应永久保存，设置清理周期 |
| 删除日志保留 | 文件删除后必须保留删除操作日志（不保留文件本身） |

## 六、打印接口安全要求

奔图 CM2800ADN/CM2820ADN 系列对接时，安全规范如下：

| 要求 | 说明 |
|------|------|
| appKey / appSecret 只存服务端 | 禁止在前端代码、环境变量文件中保存 |
| 服务端签名 | 所有奔图 API 请求由后端服务签名后发出 |
| 禁止前端生成签名 | 前端不得包含任何签名逻辑 |
| 打印文件使用临时链接 | 提交给打印机的文件 URL 必须是有时效的签名链接 |
| 回调验签 | 接收奔图回调时必须验证签名，拒绝无效请求 |
| 回调幂等处理 | 同一打印任务回调可能重复触发，必须做幂等处理 |
| 打印任务状态落库 | 每个打印任务的状态变更必须持久化到数据库 |
| 操作日志 | 所有打印相关的管理员操作必须记录日志 |

- [ ] 是否出现"一键投递"、"立即投递"等禁用文案？

- [ ] 打印任务回调是否做了验签和幂等？
- [ ] 管理员文件访问是否记录了日志？

## 八、营销 / 权益 / 套餐 / 补贴 / 支付合规边界

> 新增于 2026-06-07，关联规划：[next-tasks.md](../progress/next-tasks.md) §个人资产中心 + 权益活动 + 服务套餐 产品规划（Phase C-2 ~ C-6）、[feature-scope.md](./feature-scope.md) §六。
> 本节为「个人资产中心 / 权益活动中心 / 服务套餐 / 支付核销 / 政策补贴」补充合规边界，**不推翻 §二~§七，仅做维度补充**。

### 8.1 招聘闭环红线（重申，永不突破）

权益活动、服务套餐、企业赞助等任何商业化能力，**都不得**借机引入 §二 的 8 类招聘闭环功能（一键投递 / 收简历给企业 / 企业筛选候选人 / 面试邀约 / Offer 管理 / 候选人推荐 / 自营招聘闭环 / 企业发岗直收简历）。

### 8.2 补贴类功能边界

| 只能做 | 不能做 |
|--------|--------|
| 政策说明 | 承诺补贴到账 / 必得 |
| 材料清单 | 未授权代用户申请 |
| 官方申请入口（外链） | 长期保存身份证 / 银行卡 / 社保等高敏材料 |
| 申请材料打印 | 把「补贴券核销」包装成「政府补贴金发放」 |
| 申请指引 | — |

- 「我的权益」中的「补贴资格提示」只是**信息指引**，不得展示金额到账、发放状态等承诺性文案。
- 补贴申请材料如含身份证 / 银行卡 / 社保等，沿用 §五文件安全：临时签名 URL + TTL 清理，不长期留存。

### 8.3 企业赞助边界

- 企业可赞助权益（如赞助打印券 / 材料包），**仅体现为品牌露出 + 权益发放**。
- 企业**不得**通过赞助活动获取用户简历、候选人数据或任何可回流企业的求职者材料。
- 赞助不得形成「求职者 → 企业」的简历投递 / 候选人池。

### 8.4 服务套餐边界

- 套餐**只卖工具服务和打印服务**（AI 简历优化、面试训练、证件照、打印额度等），**不卖「录用结果」**。
- 套餐文案**禁止**「保面试 / 保录用 / 补贴必到账 / 名企内推 / 代投简历」等承诺。
- AI 模拟面试 / 面试训练结果**只给求职者本人**，不推送企业（沿用 feature-scope §六）。

### 8.5 支付 / 核销 / 退款边界

- 支付凭证 / 密钥（微信、支付宝、校园卡等）**只存服务端**，前端不得保存，沿用 §六打印接口安全的密钥原则。
- 券 / 免费次数 / 补贴券**核销必须幂等 + 落库审计**；**免费单也落库**。
- 退款**幂等**；**不得把支付异常伪装成打印任务状态**。
- 「补贴券核销」限于平台内服务 / 打印券，**不等于、也不承诺政府补贴金到账**。

### 8.6 营销 / 权益功能开发检查清单

- [ ] 是否新增了任何招聘闭环功能（投递 / 收简历 / 候选人 / 面试 / Offer）？
- [ ] 补贴类是否只做指引（说明 / 清单 / 官方入口 / 打印 / 指引），没有承诺到账或代申请？
- [ ] 企业赞助是否杜绝了简历 / 候选人数据回流企业？
- [ ] 套餐文案是否无「保录用 / 保面试 / 补贴必到账」承诺？
- [ ] 支付密钥是否只在服务端？券核销 / 退款是否幂等且落库审计？
- [ ] 高敏材料（身份证 / 银行卡 / 社保）是否只用临时签名 URL + TTL，未长期保存？

### 8.7 AI 简历优化 / 导出 / 收费边界（2026-06-29）

AI 简历诊断、优化、生成、导出、模板重排和打印属于工具服务，可以作为套餐、单次服务、优惠券或会员权益售卖；收费对象是服务处理、成果导出、文件保存和打印，不得包装成招聘结果。

允许做：

- 向求职者本人展示简历诊断、优化建议、优化前后对比、最终预览、在线编辑、AI 一键调整、导出和打印。
- 按服务成果收费：如最终优化成果导出、PDF / PNG / DOCX / TXT / Markdown 下载、长期保存、二次优化、打印页数、套餐包或券核销。
- 对已付费且未修改的同一成果，在有效保存期内允许重新下载 / 重新打印入口；是否再次收取打印费按打印订单和套餐规则执行，不能重复扣同一导出权益。同一成果必须由服务端成果版本 ID、内容哈希或 `FileObject` 标识判定，导出 / 打印权益核销以该标识作为幂等键；内容修改生成新版本，新版本按服务规则重新计费或核销。
- 使用系统内已审核已发布岗位，或用户手动粘贴岗位名称 / JD / 专业方向作为本人简历优化参考。
- 将优化后或派生成果物保存为 `FileObject`，使用临时签名 URL 预览 / 下载，并按文件保存期限策略进入本人文档或简历记录。

禁止做：

- 承诺「保面试 / 保录用 / 录用概率 / 通过率 / 名企内推 / 代投简历 / 100% 格式保真」等结果或保真承诺。
- 为企业生成候选人池、向企业或合作机构发送求职者简历、优化稿、诊断报告、联系方式或画像。
- 在 AI 简历优化页新增平台内投递、平台收简历、候选人推荐、企业查看简历、企业筛选、面试邀约或 Offer 管理。
- 将用户粘贴的外部 URL 自动抓取后当作已审核岗位入库，或把任意外部 URL 触发为来源平台投递 / 预约记录。
- 让 AI 新增未经用户确认的学校、公司、证书、时间段、薪资、获奖、工作经历等事实。

外部岗位信息边界：

- 首期只支持系统已审核岗位和用户手动粘贴 JD；外部 URL 仅可作为用户提供的参考文本来源提示，不自动抓取。
- 后续如支持 URL 解析，必须先完成来源白名单、robots / 频控、内容安全、日志脱敏、来源展示、用户确认和合规审查；解析结果仍只服务本人优化，不构成平台岗位发布或投递入口。
- 系统不得把用户简历、优化稿或 JD 分析结果回传给外部招聘来源、企业或合作机构。

文件与格式边界：

- 原始简历、语音转写文本、优化稿、导出图片和 PDF 都属于用户求职材料，沿用 §五文件安全要求和 [会员个人数据留存矩阵](./member-personal-data-retention.md)。
- 优化后 PDF / PNG / 打印预览应来自同一模板渲染链路，以保证最终成果物一致；这不等于承诺还原用户上传原件的 Word 样式或排版。
- 导出为图片可以提高最终预览和打印一致性，但图片格式不可编辑；页面必须避免把图片导出描述成可编辑 Word 原文或原版式 100% 还原。
- DOCX / TXT / Markdown 属于模板渲染派生的可编辑内容结构导出，不是用户上传原件，不保证与 PDF / PNG 像素一致，也不承诺还原上传原件的 Word 样式。
- 页面需要向用户展示成果物保存期、到期或长期保存状态；长期保存必须有用户主动确认和条款版本记录。

收费与订单边界：

- 付费动作必须先展示价格、权益抵扣、优惠券抵扣、套餐消耗、打印费用和退款/重试规则。
- 免费次数、优惠券、套餐权益和 0 元单也必须幂等落库并保留审计，不得只靠前端状态判断。
- 支付凭证、退款、对账和券核销只在服务端处理；不得把支付异常、AI 异常或导出异常伪装成打印任务状态。
- 退款与重复扣费处理应按订单 / 支付域建模，不挂靠 `PrintTask` 本身；打印任务只承载打印履约状态。
