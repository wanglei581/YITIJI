# AI 协作规则

> 创建时间：2026-05-23  
> 适用对象：Claude Code、Codex

---

## 一、分工原则

| 角色 | 主要职责 | 工作目录 |
|------|---------|---------|
| Claude Code | 主力开发：编写和修改代码 | apps/、services/、packages/ |
| Codex | 审查、拆解、修正方案；关键问题修复；文档维护 | docs/、reviews/ |

两者不固定分离，可以互相补位，但要避免同时修改同一个文件。Mac 和 Windows 同时开发时，按“一个任务、一个分支、一个清晰登记记录”隔离工作。

---

## 二、共同规则

1. **每次开发前必读**：CLAUDE.md（Claude Code）或 AGENTS.md（Codex），确认当前模块是否涉及合规边界。
2. **不允许擅自新增招聘闭环功能**：任何看起来像"投递"、"候选人"、"企业端筛选"的功能，开发前必须确认合规。
3. **每次完成后更新进度**：修改 `docs/progress/current-progress.md`，记录完成了什么、状态如何。
4. **不同时修改同一文件**：Claude Code 和 Codex 应错开对同一文件的修改，避免 Git 冲突。
5. **不分叉副本**：不要创建 "Claude版"、"Codex版"、"备份版" 等子目录。所有代码和文档都在本仓库。
6. **旧秒哒项目只读**：legacy-miaoda/ 目录下的文件只能参考，不能作为正式开发基础修改。
7. **不直接在 main 上开发**：`main` 只作为共同基线和已收口事实来源；新任务必须从同步后的 `main` 另起任务分支。
8. **聊天记录不是项目事实**：聊天、截图、临时总结只能辅助判断；正式结论以 Git 提交、验证结果和 `docs/progress/` 进度文档为准。

## 三、双设备并行开发规则

适用于 Mac 和 Windows 同时处理同一仓库、同一阶段任务的情况。

1. **每台设备只处理自己登记的任务分支**：不得两台设备同时在同一功能分支写代码；确需接手时，先在 `docs/progress/next-tasks.md` 标注接手原因、当前分支和最新 commit。
2. **任务开始前必须同步基线**：先执行 `git fetch --all`、切到 `main`、拉取最新远程基线，再阅读正式入口文档；不要用网盘、同步盘或压缩包同步整个 Git 工作区。
3. **任务登记先于改代码**：开始前在 `docs/progress/next-tasks.md` 记录任务编号、负责人/设备、分支、状态、允许修改范围、禁止修改范围、是否触碰硬件/数据库/密钥/合规、验证要求和需要同步的文档。
4. **文件范围互斥**：两个并行任务不得同时修改同一核心文件、同一路由、同一数据库迁移、同一硬件链路或同一配置文件；如无法避免，先暂停其中一个任务并合并最新基线后再继续。
5. **Windows 负责真机结论**：Terminal Agent、奔图打印机、扫描、Windows 服务、驱动、真实出纸、断网/重启恢复等结论必须以 Windows 真机或对应现场证据为准；Mac 可以做 API、前端、文档、预生产和非硬件验证，但不能替代真机通过结论。
6. **本地环境各自隔离**：`.env*`、密钥、数据库备份、截图、录屏、打印证据和本地缓存不进 Git；仓库内只记录脱敏摘要、验证结论、证据路径和必要的可复现命令。
7. **完成后双入口同步**：任务完成、暂停、阻塞或移交时，必须同步 `docs/progress/current-progress.md` 和 `docs/progress/next-tasks.md`，写清完成内容、验证结果、遗留问题、下一步负责人和分支状态。
8. **合并后另一台设备再同步**：任务合入 `main` 后，另一台设备必须重新拉取 `main`，以最新 `current-progress.md` 和 `next-tasks.md` 为准继续。

## 四、合规红线（两者共同遵守）

以下功能无论谁开发，都一律不允许：

- 平台内一键投递
- 平台内收取求职者简历给企业
- 企业端候选人筛选、面试邀约、Offer 管理
- 候选人推荐给企业
- 自营网络招聘闭环
- 企业自主发布岗位并直接收简历

详见：[../compliance/compliance-boundary.md](../compliance/compliance-boundary.md)

## 五、进度同步规范

每次完成一个任务后，更新以下文件：

- `docs/progress/current-progress.md`：记录完成内容、验证结果、当前阶段结论、遗留风险和是否可作为项目事实。
- `docs/progress/next-tasks.md`：更新任务状态、下一步负责人/设备、分支、剩余验证和禁止误读的边界。

如果发现需要新增一个重要决策，在 `docs/decisions/` 下新建文件记录，命名格式：`YYYY-MM-DD-决策主题.md`

## 六、代码 review 流程

Codex 审查 Claude Code 提交的代码时，审查记录放在：

```
docs/reviews/codex-[模块]-review.md
```

例如：
- `docs/reviews/codex-kiosk-homepage-review.md`
- `docs/reviews/codex-print-flow-review.md`
- `docs/reviews/codex-compliance-check.md`

审查记录格式建议：

```markdown
# Codex 审查记录 - [模块名]

审查时间：YYYY-MM-DD
审查范围：文件列表

## 发现的问题
...

## 建议修改
...

## 合规检查
- [ ] 按钮文案是否合规
- [ ] 是否存在招聘闭环逻辑
- [ ] 文件安全是否符合要求
```

---

## 七、冲突解决

如果 Claude Code 和 Codex 对同一功能有不同方案，以下原则处理：

1. 以用户（王磊）的判断为最终依据
2. 两方写出各自方案，放在 `docs/decisions/` 供用户决策
3. 决定后记录在决策文件中，后续不再重复讨论
