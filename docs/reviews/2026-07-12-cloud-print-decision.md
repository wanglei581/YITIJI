## 三、候选定义盘点

「云打印」若要点亮，必须先回答"它是什么"。逐一评估五个候选定义：

### 候选 A：第三方网盘拉取打印（百度网盘 / 微信文件 / 夸克等）

用户授权网盘账号，从网盘文件列表选择文件拉取打印。

| 维度 | 评估 |
|---|---|
| 用户价值 | 中偏高。用户求职文件常存在网盘/微信里；但手机扫码上传已提供替代路径——手机端从微信/网盘"用其他应用打开"后经既有上传链路传入，价值增量主要是省去手机中转一步。 |
| 工程量级 | **大**。OAuth 授权流、token 生命周期管理、网盘文件列表 UI（27 寸触控适配）、大文件拉取与格式白名单、每家网盘一套适配。可降险变体：授权动作放在**用户手机侧**完成（手机上 OAuth/选文件后只向本系统上传选中文件），避免公共终端登录网盘。 |
| 外部依赖与费用 | **高**。百度网盘开放平台的资质门槛、审核周期、配额与下载权限**需按具体开放能力做 PoC 核实后才能下结论**（不宜未验证即断言"必须企业资质"）；微信侧无服务端读取聊天文件的 API，`chooseMessageFile` 是小程序内**用户主动选择**的客户端能力，且需注册小程序主体并过审。各家条款变更风险由本项目承担。 |
| 合规风险 | **高**（若在一体机上直接登录网盘）：公共设备会话残留/忘记登出即他人可见全部网盘文件，需强制超时登出、会话隔离、审计一整套设计；手机侧授权变体可显著降险，但网盘文件授权范围与"不长期保存敏感文件"（CLAUDE.md §11）的衔接仍需单独设计。 |
| 与既有能力重叠度 | 低（是真增量），**但已被 F3 的 2026-07-10 范围收窄声明明确排除**；且与 F11 上线前收口原则直接冲突。 |

### 候选 B：其他联网设备打开链接上传打印

手机、平板、电脑等任何联网设备打开链接上传文件，一体机确认后打印。

| 维度 | 评估 |
|---|---|
| 结论 | **架构与产品语义上已被既有「手机扫码上传」覆盖**——其上线文案原文即"手机**或其他联网设备**扫码上传文件"（F2），且项目已明确声明这就是"云上传"的收窄定义并完成代码级实现（F3）。 |
| 残留缺口 | 当前主要交付物是二维码；对没有摄像头或不便扫码的电脑，尚缺短链接/配对码等辅助入口。该缺口属于**既有"手机扫码上传"能力内部的体验增强**，不足以成立第二张首页磁贴。 |
| 重叠度 | 入口层面 100% 重复。若按此定义点亮云打印，就是把同一能力再开一个首页入口，直接违反 §二"不新增重复卡片、不新增同义入口"，并且是 F5 中已被审计否定过一次的老路。 |

### 候选 C：远程提交、到店取件（时间解耦的远程下单）

用户**不在终端旁**时（在家/在校），通过手机 H5 或小程序上传文件、选择门店/终端、（可选）在线支付，获得取件码；到店后在一体机输入取件码确认，本机出纸。

| 维度 | 评估 |
|---|---|
| 用户价值 | **高，是相对文档打印站得住的真增量**。既有扫码上传要求人在一体机旁（二维码由一体机生成）；本候选把"提交"与"取件"在时间和空间上解耦。行业标配模式（F9：印萌/Wepa/PrintWithMe）。 |
| 工程量级 | **大（跨端新链路），且状态模型改造比表面看起来重**。可复用的地基：手机侧页面（F8）、取件码（`Order.pickupCode`）、任务 TTL/未取件过期（F7）、支付底座（F9）。**必须新增**：站外发起入口（不依赖一体机生成二维码的公网下单页或小程序）、终端/门店选择与任务路由、**到店释放前不可被 claim 的状态模型（F13：`awaiting_release`/`reserved` 或到店输码后才创建 PrintTask）**、释放幂等与重复出纸防线、取件码防爆破、门店营业/耗材能力校验、价格锁定与超时退款、到店通知。 |
| 外部依赖与费用 | 中。H5 路线可复用现有域名与短信登录，几乎无新外部依赖；小程序路线需主体注册与审核。在线支付复用已合入的支付域。 |
| 合规风险 | **中高**。公网入口暴露面扩大（防匿名滥用须绑定会员手机号、违规内容代打的内容责任设计）；须严守 F7"不能公网远程直控打印机"——远程只允许**入队等待释放**，出纸必须到店确认，这与既有安全边界方向兼容但需要 F13 所述的状态模型支撑。 |
| 与既有能力重叠度 | 低。上传/打印/取件码/支付复用，新增"站外发起 + 到店释放"这层编排。 |
| 时机 | 与 F11 上线前收口冲突——属于**商用二期扩展**，不应在当前收口阶段启动。 |

### 候选 D：邮件发送打印（简评）

用户把文件发送到指定邮箱，系统收信转打印任务（PrintWithMe/Wepa 均支持）。

- 用户价值：低-中（国内用户邮件习惯弱，微信/网盘场景占绝对主导）。工程量级：**中高**——除收信服务、附件白名单外，还涉及 SPF/DKIM/DMARC、发件人绑定会员、垃圾邮件/病毒/压缩炸弹过滤、加密附件、重复投递幂等、目标终端路由、退信与文件保留策略。合规风险：中（匿名投递滥用）。
- 结论：价值密度明显低于候选 C，若未来做远程提交可作为其附属入口再评估，**不值得单独定义为"云打印"**。

### 候选 E：会员云文档跨终端释放（follow-me printing，Codex 评审补充）

用户先从个人设备把文件存入"我的文档"（会员资产），之后到**任一**允许的终端登录、从云端文档列表选文件并确认出纸；不预先指定门店，也不产生远端可 claim 的任务。

| 维度 | 评估 |
|---|---|
| 用户价值 | 中高。与候选 C 互补：C 是"指定取件点的订单"，E 是"文件跟人走"；对多网点部署形态价值明显，对当前单点/少点部署价值有限。 |
| 工程量级 | 中到大。可复用"我的文档"资产与登录体系；需补：站外会员上传入口、终端可见范围与释放授权、跨终端价格/能力校验、文件 TTL 与敏感文件策略。同样受 F13 约束（释放动作在终端侧发起，天然规避远程 claim 问题，状态模型改造比 C 轻）。 |
| 合规风险 | 中。公网上传入口防滥用；会员文件跨终端可见性需要授权边界设计。 |
| 重叠度 | 低（真增量）。与候选 C 可合并规划为"远程提交能力族"的两种形态。 |
| 时机 | 同 C，属商用二期；且价值依赖多网点运营形态，比 C 更后置。 |

> 其余可想到的形态（IPP/桌面虚拟打印机、浏览器扩展、小程序分享入口等）均为 C/E 的"远程提交渠道"变体，已合并进上述评估，不单列候选。

### 盘点小结

| 候选 | 是否构成对文档打印的真增量 | 当前可行性 |
|---|---|---|
| A 第三方网盘 | 是 | 已被范围收窄声明排除（F3/F4）；外部依赖与公共设备账号安全成本高（依赖细节待 PoC 核实） |
| B 其他设备链接上传 | **否（入口层面重复）** | 已代码级上线；残留的"电脑无摄像头"缺口属既有能力内增强，不足以成立新磁贴 |
| C 远程提交·到店取件 | **是（增量最直接）** | 地基已有，但需 F13 状态模型改造；属商用二期，收口阶段不启动 |
| D 邮件打印 | 弱增量 | 并入 C 再议，不独立成卡 |
| E 跨终端释放（follow-me） | 是（与 C 互补） | 依赖多网点形态，比 C 更后置；与 C 合并规划 |

## 四、处置选项与推荐

### 选项 a：重定义为候选 C（或 C+E 能力族）并排期

把磁贴重定义为"远程提交·到店取件"（**改名，不再叫"云打印"**，消除 F6 词汇冲突），保持 disabled，进 next-tasks 商用二期排期。

- 优点：保住一个有真实价值的规划位；行业标配能力有明确落点。
- 缺点：在收口阶段新增一条排期承诺（与 F11 有张力）；"即将上线"角标继续挂给用户看而实际排期在二期，占位诚实度存疑。**更优的做法是等能力接近可交付时再新增准确命名的入口，而不是长期展示"即将上线"。**
- 适用前提：用户已批准 C/E 的二期立项、有明确负责人和里程碑。

### 选项 b：合并进文档打印，删除该磁贴（推荐）

从首页移除「云打印」磁贴。其语义中已实现的部分（云端队列执行架构、其他设备上传）全部由「文档打印 + 手机扫码上传」承载；未实现且有价值的部分（候选 C/E）作为独立事项记入商用二期候选清单（记录名"远程提交·到店取件 / 跨终端释放"），未来若立项按 §8.1 流程从范围声明开始，**届时再决定是否需要首页入口以及叫什么名字**。

- 优点：与 §3⑧ 先例的**去重原则一致**（同义能力不开第二个入口；注意具体处置与岗位大师不同——岗位大师是"保留磁贴点亮复用"，因为首页此前没有该能力入口；云打印同分组内紧邻的「文档打印」已承载同一能力，保留任何形式的复用磁贴反而制造重复入口，故本案的正确动作是删除）；首页能力集合与 `/print-scan` 服务中心对齐（F2 服务中心本来就没有云打印）；消除 F6 词汇冲突；删除的是一张定义不出增量的模糊承诺卡，不删除任何已实现能力。
- 缺点：需要一次首页代码改动（删一行磁贴 + 布局走查）；未来若做候选 C/E 需重新加入口（成本低，且届时名称/位置可按真实能力定）。
- **执行影响面（经 Codex 评审补全，比初稿更宽）**：
  1. 代码：[HomePage.tsx:328](../../apps/kiosk/src/pages/home/HomePage.tsx) 删一行。F1 已核实无路由/无 import/无测试硬断言依赖。**不顺手清理** `cloud` 图标定义或云端架构术语（图标可能被他处引用，须全仓确认后另行处理）。
  2. 布局：首页子入口网格为固定 3 列（[home-inkpaper.css:624-631](../../apps/kiosk/src/pages/home/home-inkpaper.css)），打印扫描组从 6 磁贴变 5 磁贴，第二行只剩 2 项——需走查竖屏 540×960、窄屏单列、桌面视口下与相邻分组卡的等高关系。
  3. 活跃文档同步：matrix §3.4 云打印行 + §217 行、[ia-consolidation-audit-2026-07-03.md:57](./ia-consolidation-audit-2026-07-03.md)、[ia-consolidation-phase-a-checklist-2026-07-03.md](./ia-consolidation-phase-a-checklist-2026-07-03.md) 占位清单、[home-entry-closure-plan-2026-07-11.md 发现⑤](./home-entry-closure-plan-2026-07-11.md)、current-progress 更新记录、本文档补记执行结果。**归档进度、专利与设备文档不改**（历史/架构语义，保留）。
  4. 验证门禁：`pnpm --filter kiosk typecheck / lint / build` + 回归读取 HomePage 的 verify 脚本（`verify:print-entry-source-split`、`verify:home-toolbox-ui`、`verify:smart-campus-ui`，另跑 API 侧 `verify:terminal-device-config`）+ mock 模式浏览器走查（含 disabled 角标数量、触控命中区、首页滚动高度）。
  5. 词汇治理另立决策（§六 D4）：`cloud_upload` 能力键去向不随本改动顺手处理。

### 选项 c：保持 disabled 占位继续观望

不动代码，维持现状。

- 优点：零成本、零风险；不排除任何未来可能。
- 缺点：matrix §3.4 的澄清备注就是为了终结"无限期观望"而写的——观望必须有终点，否则每轮首页核查都会再撞上这张卡；长期挂"即将上线"是对用户的持续模糊承诺。
- 若选此项，观望终点必须明确为：**「远程提交·到店取件」（候选 C/E）在商用二期立项排期之日**——届时按选项 a 重定义改名；若商用二期规划正式放弃远程下单方向，则立即转为选项 b 删除。观望期间该磁贴不得点亮、不得改跳 `/print/upload`。此选项只在"发布候选完全冻结、不允许任何首页变化"时有短期优势。

### 推荐结论

**推荐选项 b（合并删除 + 候选 C/E 记入二期候选清单）**，理由：

1. 五个候选定义中，构成真增量的 C/E 本身并不叫"云打印"、也不该在收口阶段启动；而卡片如果不能在可见的将来兑现为某个明确能力，"即将上线"就是不诚实占位。
2. 与已确立的 §3⑧ 先例**去重原则一致**（判断标准不是"这个功能好不好"，而是"是否构成同义入口"）：候选 B 证明其字面含义已被覆盖，F5 证明直接点亮的路已被否定过；与岗位大师的差别仅在处置动作——那里保留点亮是因为首页缺该能力入口，这里删除是因为同组已有承载入口。
3. 删除后首页与 `/print-scan` 服务中心的能力集合一致，消除 F6 词汇冲突；未来候选 C/E 立项时可以用准确的名字重新设计入口，不背历史包袱。

次选为选项 c（带明确终点条件的观望），适用于用户已基本确定商用二期会做远程下单、只是现在完全不想动首页代码的情形。
