## 一、结论

当前用户中心不是空壳，已经具备真实的手机号/扫码登录、会员会话、本人数据隔离、简历/文档/AI 记录、收藏、浏览记录、打印订单、权益、消息和反馈等基础能力。但它目前更准确的定位是“真实数据入口聚合 + 局部操作闭环”，还不能按商用级“会员中心”宣称完成。

上线前最需要处理的不是继续加入口，而是以下五类收口：

1. **真实表达**：去掉重复入口、不可用按钮和“建设中”卡片，未完成能力不上线展示。
2. **账户治理**：补敏感操作二次认证、手机号换绑、账户状态、注销和数据导出；当前 `delete/export` 只是工单壳，不是完整实现。
3. **运营闭环**：Admin 用户管理仍为空态，隐私数据请求虽有后端接口但没有正式处理页面。
4. **资产与交易闭环**：简历、文件、活动记录、订单和权益在删除、下载、分页、复用、退款和核销方面口径不一致。
5. **生产证据**：收费/权益模式、PostgreSQL、对象存储、Redis、短信、Windows 一体机和真实打印仍需按同一版本完成预生产与真机验收。

## 二、审计方法与证据边界

### 2.1 已检查

- 正式入口文档：`AGENTS.md`、当前进度、下一步任务、功能范围、用户数据流矩阵、合规边界和数据保留规则。
- Kiosk：`/profile`、`/login`、`/me/settings`、`/me/documents` 的访客运行态；用户中心各页面和 API 适配器源码。
- API：会员认证、会员资产、隐私请求、权益、活动记录、打印订单、消息和反馈模块；Prisma 会员关系。
- Admin：用户管理路由和隐私请求后端接口。
- 自动验证：会员会话、会员资产、收藏/权益、活动记录、打印订单 UI、数据保留、反馈/通知等相关脚本。

### 2.2 证据限制

- 本轮没有使用生产会员账号，不把 mock/访客页面当成登录后生产验收。
- 未执行真实短信、真实支付、真实对象存储、PostgreSQL 生产实例、Windows Terminal Agent 和真机出纸。
- 初始审计阶段 Antigravity 前端模型因个人额度耗尽未返回有效报告；前端/触控结论由 Codex 基于真实页面与代码完成，Claude 完成独立的架构、合规和数据生命周期复核。后续 Wave 0/Wave 1 精确实施计划补齐后，Antigravity 已恢复并给出 `APPROVE`（99/100），Claude 最终复审亦为 `APPROVE`，均无 Critical/Warning；结果详见本任务 CCG `review.md`。
- 当前工作区含用户在途修改，本轮没有修改业务代码，也没有清理这些变更。

## 五、缺口矩阵

### 5.1 上线阻塞（P0）

| 缺口 | 当前实际状态 | 为什么是阻塞 | 建议处置 |
|---|---|---|---|
| 页面真实表达 | 重复“权益活动”、邮箱按钮不可用、凭证/套餐卡显示“建设中”；“身份切换”虽已不在最新 main，仍需防回归 | 商用页面存在无效承诺和无效操作 | 删除重复项；隐藏未完成卡片；去掉邮箱按钮；更新 Profile 真相守卫并复验现有二维码守卫。 |
| 数据删除/账户注销 | `delete` 数据请求目前只删除 Job AI 会话并撤回同意，不删除/匿名化完整账户数据 | 用户看到“注销/删除”时会产生错误预期；无法形成可审计的数据权利闭环 | 先完成分类处置、状态机、二次认证、幂等、审计和对象存储清理，再开放入口。 |
| 数据导出 | `export` 仅创建请求记录，不生成导出包或下载凭证 | 工单可被标记完成但用户拿不到结果 | 用异步任务生成白名单 JSON，存入私有加密存储；通过 10 分钟一次性应用 ticket 由手机下载，不下发对象存储 URL，消费后物理删除。 |
| 隐私请求完成态不可信 | 当前 Admin 可把 `export/delete` 请求置为 `completed`，但 export 无产物、delete 只处理 Job AI 数据 | 会生成“已完成导出/删除”的误导性审计证据 | Wave 0 先禁止这两类请求进入 completed，或明确标为部分执行；真实执行链完成后再开放终态。 |
| Admin 隐私请求处理 | 后端有 `/admin/member-privacy/data-requests`，前端无路由/页面 | 运营无法按 SLA 处理、复核和留证 | 建立数据请求队列、详情、处理记录、失败重试和审计页面。 |
| 敏感操作二次认证 | 普通会员 session 即可覆盖未来注销/换绑 | 公共一体机被接管时风险高 | 注销、换绑、导出下载前重新短信验证，step-up 凭证短时有效、单次使用。 |
| 收费模式一致性（仅收费上线时） | live 支付/退款/权益抵扣/对账未形成同版本完整验收 | 可能卡收银、错扣权益或退款不可追踪 | 首台默认免费模式；若收费，支付、权益、退款、对账必须一起过门禁。 |

> 注：账户删除、数据导出和财务数据留存的法定义务与具体保留期限，必须由法务基于实际经营主体、地区和业务合同确认。本文按商用级可控、可查、可执行的工程底线设计，不代替法律意见。

### 5.3 上线后增强（P2）

| 能力 | 结论 |
|---|---|
| AI 顾问对话历史 | 当前不持久化不是上线阻塞。未来只能在用户明确选择保存、可删除、短 TTL、敏感内容治理完成后实施。 |

### 5.4 应隐藏或移除，而不是继续开发占位

1. 登录页不可用的“邮箱”方式。
2. 重复的“权益活动”入口，只保留一个真实入口。
3. “身份切换”当前最新 main 已不存在；继续由真相守卫禁止它回归。
4. 尚无真实业务闭环的“招聘会扫码凭证”“求职打印套餐”“AI 服务套餐”。
5. 任何会让用户误以为站内可投递、报名、核销或购买的按钮文案。

## 六、账户与数据生命周期关键事实

1. `UserDataRequest` 已有 `export/delete/revoke_consent` 类型，但当前不具备完整执行语义。
2. 真注销不能依赖 Prisma `onDelete: Cascade`：高敏文件需要物理删除，订单/支付/退款/审计可能需要依法保留，通知/反馈需要分类处置。
3. 推荐保留最小匿名账户锚点而不是直接删掉 `EndUser` 行：`phoneHash` 当前非空且唯一，必须用不可逆随机墓碑值替换，`phoneEnc` 替换为不含原号码的随机墓碑密文，昵称清空；原手机号 hash 不得残留在活跃/匿名账户中，从而允许用户以后以同一手机号注册为全新账户且不继承旧资产。具体字段和保留期限由法务确认。
4. `UserDataRequest` 本身不能随账户删除而消失，否则会丢失“曾提出并完成删除请求”的审计证据。
5. 数据导出包必须是短期、高敏、本人可下载的临时文件，不得变成长期备份。

## 七、体验与无障碍审计

### 7.1 做得较好的部分

- 现有浅色卡片、蓝绿品牌色、图标和大触控目标在 27 寸竖屏上可读。
- 未登录资产页有明确登录门禁，不直接暴露会员数据。
- 打印订单筛选已使用 `aria-pressed`，关键按钮尺寸守卫达到 48px。
- 加载、空态、错误提示和轮询退避已有较成熟基础。

### 7.2 需要补齐

- 所有敏感确认弹层必须支持键盘焦点锁定、Esc 退出、焦点归还和屏幕阅读器标题/描述。
- 终端模式不能依赖 hover；主要操作最小 48×48px，危险操作与主要操作必须拉开距离。
- 长列表统一使用后端游标分页或“加载更多”，不能把前 50 条当成全部。
- 弱网时要保留已加载数据，显示低噪声重试，不用全屏错误覆盖用户资产。
- 注销、换绑、退款等流程必须明确“当前步骤、不可逆后果、处理时限、如何撤销/申诉”。
- 手机端不能复用过宽双列卡片；桌面端不能无限拉宽正文；一体机需按 1080×1920 真机视口复验。

### 8.1 通过

- Kiosk：`verify:profile-commercial-first-batch`、`verify:member-session-closure`、`verify:member-print-orders-ui`。
- Kiosk 最新 main 只读复核：`verify:qr-login-ui` 已通过，现行“本机服务不可用？改用手机号登录”降级路径应保留。
- API（以 `PAYMENT_PROVIDER=disabled` 隔离本地生产支付配置）：`verify:member-auth`、`verify:member-assets-c2d`、`verify:member-favorites-benefits`、`verify:activity-logs`。
- API：`verify:member-data-retention`、`verify:feedback-notifications`。

### 8.2 未通过或未形成有效证据

- `verify:member-print-orders`：权威 Prisma schema 已有 `Order.refundedAmountCents`，但当前本地 SQLite 验证库没有重放到包含该列的正式 migration。
- `verify:benefit-redemption`：权威 Prisma schema 已有 `RedemptionRecord`，但当前本地 SQLite 验证库没有重放到包含该表的正式 migration。
- API 首次使用当前环境 `PAYMENT_PROVIDER=wechat` 启动时被生产配置闸门拒绝；改为 `disabled` 后非支付会员验证可运行。未修改用户环境配置。

这些失败说明“当前本地验证基线未完全同步”，不能直接证明对应业务代码错误，也不能把订单/权益验证写成已通过。后续实施前必须先从干净分支按正式迁移流程重建 SQLite 验证库，并同时跑 PostgreSQL readiness。
