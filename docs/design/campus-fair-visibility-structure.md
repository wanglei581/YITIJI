# 校园招聘会可见范围与页面结构摘要

> 抽取时间：2026-06-21
> 来源：主工作区 `.superpowers/brainstorm/66602-1781578016/content/campus-fair-visibility-structure.html`
> 状态：设计决策摘要，不代表已进入开发或上线范围。

## 背景

该 HTML 预览用于比较 Kiosk `/campus` 的当前页面结构与推荐的信息架构方向。T3 不将交互 HTML 预览直接纳入正式仓库，而是抽取其中可执行的设计结论，避免 `.superpowers/` 本地工具目录被 ignore 后静默掩盖有价值内容。

## 当前结构

当前实现可以拿到所有已审核发布的招聘会，再由 `/campus` 按校招关键词或主题筛选，并选择最高分的一场招聘会作为沉浸式单场页面展示。

优点：

- 单场体验完整。
- 沉浸感强。

问题：

- 多所学校同时有招聘会时，其它场次容易被埋掉。
- 缺少学校分类。
- 难以表达“本校优先”。

## 推荐结构

推荐将 `/campus` 调整为“全公开聚合 + 学校分类 + 本校优先”的校园招聘会聚合入口。

核心方向：

- 校园招聘会仍作为公开信息入口，不做私域隔离。
- 前台按学校、城市、状态、时间等维度聚合展示。
- 若终端绑定学校，则默认置顶本校相关场次。
- 点击某一场后再进入单场详情，承载企业速览、参展企业、导览图、AI 求职和打印服务。

优点：

- 不形成招聘闭环。
- 多校数据不会被隐藏。
- 本校场次仍能优先展示。

代价：

- `/campus` 需要从单场页改为聚合页。
- 后台可能需要补充主办学校、面向对象等展示字段。

## 推荐数据流

1. 合作机构或学校发布本校招聘会，招聘会保留 `sourceOrgId` 等来源信息。
2. 管理员审核并补全主办学校、参展企业、岗位、导览、资料和数据大屏信息。
3. 前台只展示已审核、已发布、公开可见的校招会。
4. 校园页展示规则为本校优先，不做本校独占。
5. 预约或投递动作继续跳转到来源平台，本系统只记录浏览或外部跳转行为。

## 合规边界

- 不能提供平台内投递。
- 不能收集求职者简历并提供给企业。
- 不能提供企业候选人筛选、面试邀约或 Offer 管理。
- 按钮文案继续使用“去来源平台投递 / 扫码投递 / 去来源平台预约 / 扫码预约”等合规表述。
