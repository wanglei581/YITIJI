# 扫码器「付款码收款」（B扫C）设备输入探测方案

> 创建：2026-07-06（C5-6 波）
> 状态：**探测方案，未立项实现**。C5-6 已交付屏幕动态二维码收款（C扫B，用户扫屏）为主方案；
> 付款码收款（扫码器读用户手机上的微信/支付宝付款码，服务端调渠道被扫接口）是否实现，
> 由本文档的真机探测结论决定 —— **探测不通过或业务不需要则不做，不为“看起来完整”堆功能**。

## 1. 两种收款形态对照

| 形态 | 链路 | C5-6 状态 |
|------|------|-----------|
| 屏幕动态码（C扫B / Native·当面付 precreate） | 服务端下单出码 → 27寸屏展示 → 用户手机扫码支付 → 渠道回调/查单入账 | ✅ 已实现（主方案） |
| 付款码收款（B扫C / micropay·当面付 pay） | 用户出示手机付款码 → 一体机扫码器读码 → Kiosk 提交服务端 → 服务端调渠道被扫接口 → 入账 | ⏸ 待探测决策 |

付款码收款的前提是**一体机扫码器能稳定读手机屏幕上的一维/二维付款码**，且输入方式可被 Kiosk 浏览器捕获。

## 2. 真机探测步骤（Windows 一体机，按序执行）

1. **设备枚举**：设备管理器确认扫码器呈现为 `HID 键盘`（keyboard wedge）还是 `COM 串口`；记录 VID/PID 与型号。
2. **记事本裸测**：焦点在记事本，扫印刷二维码 —— 确认是否直接输出字符 + 回车结尾；记录是否有可配置前缀/后缀。
3. **屏幕读码测试（关键）**：扫手机屏幕上的微信付款码（一维条码 + 二维码各测）与支付宝付款码；手机分别在最高/中/低亮度、有/无贴膜条件下各测 10 次，记录成功率。**扫码器必须支持屏幕读码（CMOS 影像式）**——激光式扫码器读不了屏幕，直接判不通过。
4. **付款码数字特征确认**：读出内容应为 18 位纯数字；微信以 10–15 开头，支付宝以 25–30 开头。记录实际样本前缀（**样本仅现场核对，不得拍照/记录完整码值** —— 付款码等同支付凭证）。
5. **Kiosk 浏览器捕获测试**：Edge/Chrome Kiosk 全屏页放一个聚焦输入框，验证扫码输出可进入页面；测按键速率（HID wedge 突发键入通常 <50ms/字符，可与人工输入区分）；验证无焦点/焦点在其它控件时的行为。
6. **干扰与安全测试**：非收银页面扫码时输入去向；连续扫码防抖；扫非付款码（任意二维码）时的内容形态。

## 3. 探测通过 → 立项实现时的硬性安全红线

- 付款码 = **敏感支付凭证**（等同一次性银行卡号）：Kiosk 内存即用即弃，**不落日志、不入库、不进前端状态管理持久层**；仅 HTTPS 提交服务端。
- 服务端调渠道被扫接口（wechat micropay / alipay.trade.pay），复用 C5-6 的 Provider 抽象、金额服务端快照、幂等入账路径（`markPaidOnline`）与审计；**Kiosk 依旧零渠道密钥**。
- 付款码格式服务端二次校验（18 位数字 + 渠道前缀路由）；错误码不回显码值。
- 仅收银页收款态接受扫码输入（页面态门控），其余页面忽略 HID 突发输入，防扫码注入。
- 被扫下单是同步扣款：必须处理 `USERPAYING`（等待用户输密码）轮询查单收敛，超时走撤销（revoke/cancel），**不得把待确认态标 paid**。
- 生产门禁沿用：`PRINT_REQUIRE_PAID_BEFORE_CLAIM=true`，claim 门禁在服务端。

## 4. 决策门

满足全部条件才立项（暂命名 C5-6b）：

- [ ] 探测步骤 3 屏幕读码成功率 ≥ 95%（常用亮度）
- [ ] 步骤 5 Kiosk 浏览器可稳定捕获且可与人工输入区分
- [ ] 运营确认存在真实场景需求（用户扫屏不便的比例/反馈）
- [ ] 商户号已开通被扫产品权限（微信 micropay / 支付宝当面付被扫）

任一不满足 → 维持屏幕动态码单方案，本文档留档即可，不写任何付款码代码。
