# Windows 终端批量安装、换机与受控发布设计

> 状态：设计已确认，尚未实施
>
> 创建时间：2026-07-15
>
> 适用规模：5–50 台 Windows 一体机；后续可接入 Intune/MDM，但不以其为首期前提。
>
> 关联文档：[生产部署与 Windows 本地主机换机验收清单](./production-deployment-and-windows-host-checklist.md)｜[生产 Windows Terminal Agent 授权与加固方案](./production-agent-onboarding.md)｜[Windows Terminal Agent 设计](./windows-terminal-agent-design.md)

## 一、结论

首期不另建一套“设备平台”，而是在 Admin 的既有“终端管理”中扩展为**设备机队工作台**。它解决三件事：

1. 新 Windows 主机可由现场安装人员用一次性激活码完成受限预检；
2. 换机时保留稳定的业务终端身份，只有在候选机通过验收、旧机无活动打印任务后才原子切换；
3. 后续以签名 MSI、分批发布、空闲更新和可验证回滚管理 5–50 台主机。

这不是远程桌面、远程 Shell 或“云端直接控制打印机”。云端仍只负责授权、配置、任务协调和审计；Windows Agent 仍只在本地执行硬件操作。

## 二、范围与边界

### 2.1 复用而非重复建设

- 入口：复用 Admin 的“终端管理”，新增“设备机队”工作台视图与深链；不在 Kiosk、首页或“我的”新增入口。
- 业务身份：既有 `Terminal.id`、`terminalCode`、组织归属、启停状态、打印任务和 Kiosk 配置保持不变。`terminalCode` 是稳定服务点编号，不因更换 Windows 主机而改变。
- 物理主机：单独作为设备资产和候选安装记录管理，不能把候选机注册成第二个生产 `Terminal`。
- 已有首次授权：保留 `TerminalBindCode` 与 `exchange-bind-code` 用于**新服务点首次启用**。它的兑换会使旧 token 失效，故不得直接用于“不断机换主机”。

### 2.2 明确不做

- 不做远程桌面、任意远程命令、远程 PowerShell、远程读取文件、远程查看 token 或批量强制更新。
- 不让候选机领取、执行或回写真实打印任务；候选机不启动本地 QR 登录桥，也不访问用户文件。
- 不自动合并以 `terminalCode` 与内部 `terminalId` 混用产生的旧配置；只标记冲突，交由管理员处理。
- 不把“安装包构建成功”“本地浏览器成功”或“模拟打印”写成 Windows 真机或生产验收通过。

## 三、角色与权限

| 角色 | 可以做什么 | 明确禁止 |
| --- | --- | --- |
| 总部设备管理员 | 创建激活码、审批换机/发布、查看脱敏设备状态、执行切换、回滚、审计 | 查看或导出运行 token、远程 Shell、绕过审批直接覆盖生产凭据 |
| 现场运维 | 查看本机诊断和待办、上传无敏感验收摘要、发起故障单 | 创建生产凭据、审批发布、切换终端、读取用户文件 |
| 上门安装人员 | 输入一次性激活码、执行本机预检、提交候选机状态 | 登录 Admin、获得管理员凭据、领取任务、启动 QR 桥 |

高风险操作（生产切换、发布批准、回滚）必须有 Admin JWT、明确二次确认、原因和 `AuditLog` 摘要。明文激活码只在创建时展示一次；数据库、日志、审计和前端状态均不得保存明文。

## 四、信息架构

在现有 Admin “终端管理”内形成以下工作台，不增加同义页面：

```text
终端管理
├─ 概览：在线/离线、告警、版本、候选机、待切换、发布批次
├─ 终端详情：既有档案、健康摘要、配置深链、物理资产历史
├─ 安装与换机：创建激活码 → 候选预检 → 验收 → 原子切换
├─ 发布：签名包、分批范围、暂停/继续、结果与回滚
└─ 告警与审计：只展示诊断摘要、操作人、时间、原因和结果
```

既有屏保、智慧校园、百宝箱等配置仍在原页面维护；工作台只显示配置摘要和跳转链接。公开 Kiosk 的配置响应、页面路由和用户侧业务流程不在本设计范围内。

## 五、身份、凭据与数据模型

### 5.1 三层身份

```text
稳定服务点 Terminal（业务身份）
        │ 1:多（历史）
物理主机 TerminalAsset（设备资产）
        │ 0:1（待验收）
候选安装 TerminalCandidate（受限预检）
```

- `Terminal`：保留既有生产身份和任务归属，不在换机中删除、重建或改写 `terminalCode`。
- `TerminalAsset`：记录设备标签、硬件指纹摘要、OS/Agent/驱动版本、打印机名、安装/退役时间和脱敏诊断摘要。MAC 仅在受保护的详情权限下展示；列表不展示完整 IP、token 或用户数据。
- `TerminalCandidate`：记录目标 `terminalId`、候选资产、状态、一次性授权摘要、预检结果、有效期和并发版本。它不是可服务终端。

### 5.2 凭据演进

现有生产 `agentToken` 需进入**hash-first 迁移**，不能一次清空旧字段：

1. 新建 `TerminalCredential`，保存带 server pepper 的 HMAC-SHA-256 摘要、用途、版本、失效时间和撤销时间；验证采用常量时间比较。
2. 第一阶段服务端双读：优先验证新凭据摘要；仅对已存在的旧 Agent 兼容旧 token 读取，直到迁移完成。
3. 新候选机使用三类互不复用的短生命周期凭据：单次激活码、预检凭据、生产运行凭据。候选机在本地生成运行 token，以 DPAPI 保护，并只向服务端提交其摘要；浏览器与 Admin API 只接触激活码，不接触运行 token。
4. 迁移覆盖全部生产 Agent 并完成回归后，再用独立删除任务移除旧明文字段；该删除不得与换机功能同一 PR 完成。

Prisma 变更必须是 additive，并同步 `services/api/prisma/schema.prisma` 与 `services/api/prisma/postgres/schema.prisma`、对应迁移及 `db:pg:sync:check`。

## 六、安装与换机状态机

### 6.1 候选机状态

```text
created → activation_issued → preflighting → ready_for_acceptance
                                     │                │
                                     └→ failed        └→ cutover_pending
                                                          │
                              expired / cancelled ←──────┼──────→ active / retired
```

候选机处于 `preflighting`、`ready_for_acceptance` 或 `cutover_pending` 时，只能执行本地诊断：DPAPI 保护、Windows 服务安装、网络连通、Agent 版本、WMI/打印机、磁盘、SQLite 可用性、配置格式和本地端口边界。它不得调用现有生产注册、心跳、领取、状态回写或 QR 代理接口。

### 6.2 原子切换

管理员在候选机准备完成后发起切换；服务端单个事务必须同时满足：

1. 候选记录未过期、预检新鲜、目标终端一致、候选硬件指纹未冲突；
2. 目标终端没有 `pending`、`claimed` 或 `printing` 的活动打印任务；
3. 切换版本未被并发操作改变；
4. 候选机运行凭据摘要有效，旧运行凭据可撤销；
5. 管理员权限、二次确认和审计字段齐全。

事务成功后才完成下列动作：激活候选运行凭据、撤销旧凭据、更新一个生产资产关联、标记候选为 `active`、写审计。任何前置条件失败都零副作用。旧 Agent 下一次访问应收到专用“已撤销”错误并停止运行，绝不自动重新注册或抢占新主机。

这保证了业务侧的 `Terminal.id`、`terminalCode`、启停状态、打印任务和既有 Kiosk 配置不变；硬件替换只是同一服务点的资产切换。

## 七、Windows 安装包与受控发布

### 7.1 安装包基线

首选 WiX MSI（而非 MSIX），以适配 Windows Service、Node 运行时、SumatraPDF 和 repair/uninstall：

- 安装目录：签名程序和依赖位于 `Program Files`，不可由 Kiosk 用户修改。
- 数据目录：配置、DPAPI token、SQLite、日志和临时文件位于 `%ProgramData%\\AIJobPrintAgent`，安装程序显式设置 NTFS ACL。
- 安装过程不携带 `TERMINAL_ADMIN_SECRET`，只能输入短时一次性激活码。
- 包、Bootstrapper 和更新 manifest 均须验证 Authenticode 签名、SHA-256 和签名者指纹。

### 7.2 版本发布策略

发布控制面保存版本、签名摘要、兼容范围、目标批次、批准记录、结果和回滚版本；不保存安装包中的秘密。更新由独立受控 Bootstrapper/计划任务执行，不能由正在运行的 Agent 覆盖自身。

| 规模 | 首批 | 后续批次 | 观察与停止条件 |
| --- | ---: | ---: | --- |
| 5 台 | 1 台 | 2 台 → 2 台 | 每批观察至少 24 小时；服务/心跳/预检异常立即暂停 |
| 50 台 | 2 台或 5% | 5–10 台 → 倍增 | 每批观察至少 24 小时；错误率、离线率或回滚触发即暂停 |

只允许空闲终端更新。发布前必须确认无活动打印任务；更新中不得接收新任务；失败回滚到上一个已验证稳定版本。首期不做“强制立即更新”。

## 八、实施顺序与准入门槛

| 阶段 | 交付 | 前置条件 | 不包含 |
| --- | --- | --- | --- |
| F0：只读机队视图 | 健康/版本/配置摘要与原页面深链 | 不触碰生产凭据 | 写操作、候选机、远程更新 |
| F1：安全前置 | 生产管理员凭据轮换、手机号验证、审计验证 | 单独 PR 合并、CI 与受控生产验收完成 | 任何设备切换 |
| F2：安全换机 | 凭据 hash-first、候选机隔离、原子切换、回归/并发门禁 | F1 完成；双 Prisma 迁移评审 | MSI、批量更新 |
| F3：安装与发布 | WiX MSI、ACL、签名、分批发布、回滚 | F2 真机换机验收通过 | 远程 Shell、强制更新 |

当前只能推进 F0 的只读设计/实现准备；F1 未完成前不得开发或部署 F2 的生产切换能力。生产管理员凭据轮换必须由用户本人在合并部署后的 Admin UI 完成，真实密码、数据库连接串和 token 不进入聊天、代码或文档。

## 九、验证与真实验收

### 9.1 代码与回归

- 既有生产 Agent 的心跳、领取、状态回写、QR 本地桥接保持可用；候选机变更不得影响 `Terminal`、心跳、`PrintTask`、订单或 Kiosk 配置。
- 切换因活动任务、过期候选、凭据冲突或并发请求失败时，必须验证零副作用；并发切换仅允许一个成功。
- 旧机撤销后不得领取、回写或执行 QR 代理；新机完成切换后才允许这些生产能力。
- 每项后端 schema/index 变更跑 SQLite 主验证、PostgreSQL readiness、相关 API/Agent/Admin typecheck、lint、verify 和双模型审查。

### 9.2 Windows 现场验收

按既有换机清单执行，并额外确认：

- MSI Authenticode 签名、哈希、repair、uninstall、服务自启与有限失败恢复；
- `%ProgramData%` ACL 使 Kiosk 用户不能读取运行 token；复制 token 到另一台主机不能解密；
- 候选机默认无法领取任务、无法启动 QR 桥；
- 真实 Windows 打印机/WMI、HTTPS Kiosk 的本地桥接/PNA、PDF 与图片物理出纸、Windows PrintService 证据；
- 更新中断、失败回滚和旧版本恢复；
- 每次实机操作前确认 `active_task_count = 0` 且活动任务查询为 `0 rows`，并准备精确清理路径。

所有现场证据保存在受控的仓库外目录；仓库只记录脱敏结论和验收状态。未执行 Windows GUI、物理出纸和清理验证前，不得宣称换机或批量发布完成。

## 十、首个实施任务的范围声明

在 F1 前如需先做 F0，只允许建立只读聚合：

```text
功能：Admin 终端管理内的设备机队只读总览
前端：apps/admin（既有终端管理路由和服务适配器）
后端：services/api/src/device-fleet/（只读投影，不改既有配置服务）
终端：apps/terminal-agent 不涉及
共享类型：仅在跨端契约确有复用时新增
文档：本设计及进度入口
禁止：Prisma 变更、激活码/凭据写操作、Kiosk 路由、打印任务状态、远程发布
```

F0 先验证“现有终端、配置与健康信息能否被准确聚合”；确认真实运营价值后，才在 F1 完成的前提下另起独立分支实现 F2。
