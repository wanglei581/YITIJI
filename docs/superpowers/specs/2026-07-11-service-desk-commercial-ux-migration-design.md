# 蓝白服务台三端商用 UX 迁移设计规范

> 日期：2026-07-11
> 状态：视觉方向已由用户确认；本文为商用 UX 详细候选规范，待用户审阅全文后生效；本文不自动授权代码迁移。
> 适用端：Kiosk 一体机前台、Admin 管理员后台、Partner 合作机构后台。
> 视觉方向：以 `http://127.0.0.1:4188/` 当前亮蓝白服务台原型为视觉参考。
> 核心要求：统一视觉语言，但按角色保留不同的信息密度、任务结构、权限边界与验收方式。

## 1. 决策摘要

用户已明确废弃当前项目正在使用或规划中的「墨青纸感 / 暖米纸 / 墨绿侧栏 / 宋体标题」视觉方向，改为采用 4188 原型的：

- 明亮蓝白服务台气质；
- 深海军蓝文字层级；
- 白色商业表面与浅冰蓝画布；
- 克制的柔和分类色；
- 大而清楚的触控入口；
- 真实状态驱动、一次性且可降级的动效；
- 小青真人顾问作为 Kiosk 的品牌人物资产。

本次不是把 4188 页面截图复制到所有页面，也不是只改颜色。正式目标是建立一套共享设计系统，并把「看得懂、办得成、出错能恢复、结果可追溯、后台能运营、全程合规」固化为商用 UX 门禁。

选定方案为「共享设计系统分层迁移」：

1. 先统一语义 token、壳层和通用组件。
2. 再用三端代表页验证视觉、密度和交互。
3. 按业务域小批迁移，不一次性重写全部页面。
4. 每批同时验证真实功能、完整状态和可恢复性。
5. 最后清理旧主题，不保留两套长期并行的视觉标准。

证据边界：4188 当前已经完成的是 Kiosk 首页、AI助手和「我的」的原型视觉与基础交互 QA；它没有直接证明 Admin / Partner 的最终像素稿，也没有证明正式业务链路。把该视觉 DNA 延伸到双后台是本次用户确认的产品设计决策，必须先通过 UI-1 的 Admin / Partner 代表页验证，再允许全量铺开。

## 3. 商用 UX 的定义

本项目中的「商用 UX」不是视觉精致度的同义词，完整定义为：

> 商用 UX = 清楚的任务目标 × 真实可完成的业务闭环 × 完整状态表达 × 可恢复错误 × 角色与数据边界 × 运营效率 × 可访问性 × 可验证证据。

任何页面即使视觉与 4188 高度一致，只要存在以下任一问题，就不能标记为商用完成：

- 按钮没有真实行为，或用原型弹窗替代正式路由 / API。
- 加载、空态、错误、权限、离线、超时或建设中状态缺失。
- 显示「成功 / 已打印 / 已支付 / 已投递」但没有对应真实终态。
- 完成后无记录、无下一步、无恢复或无后台承接。
- 用户输入因校验失败、登录或网络异常而无提示丢失。
- Admin / Partner 看不到作用范围、审计结果或处理反馈。
- 触控区、焦点、对比度、键盘操作或屏幕阅读器语义不合格。
- 公共终端残留上一位用户的个人内容或敏感状态。
- 用色彩、图标或动效掩盖合规来源和业务边界。
- 只有截图通过，没有真实路由、接口、浏览器、预生产或真机证据。

### 3.1 商用 UX 十项原则

1. **结果优先**：页面首先帮助用户完成一件事，不以展示组件数量为目标。
2. **状态诚实**：系统知道什么、正在做什么、还不知道什么必须明确可见。
3. **闭环完整**：入口 → 操作 → 结果 → 记录 → 后续动作形成可追溯链路。
4. **可恢复**：错误说明发生了什么、影响什么、如何继续，尽量保留用户输入。
5. **控制权明确**：用户可取消、返回、重试或退出；不可逆动作必须再次确认。
6. **角色正确**：Kiosk、Admin、Partner 只看到其角色允许的信息和操作。
7. **合规可见**：来源、隐私、价格、保存期限和平台边界是页面结构，不是隐藏脚注。
8. **包容可用**：触控、键盘、焦点、对比度、缩放和 reduced motion 都能工作。
9. **运营高效**：后台支持筛选、批量判断、证据查看、审计与处理后的即时反馈。
10. **证据可验**：视觉、交互、接口、数据、预生产和硬件结论分层记录，不越级宣称。

## 4. 范围与非目标

### 4.1 本规范覆盖

- 三端共享的颜色、字体、圆角、阴影、间距、图标与动效语言。
- Kiosk Touch、Admin Compact、Partner Comfortable 三种密度配置。
- 页面壳层、导航、页头、卡片、表格、筛选、表单、抽屉、弹窗和状态组件。
- 加载、空态、错误、权限、离线、过期、处理中、未知、成功与失败状态。
- Kiosk 公共终端隐私、触控、设备 / 网络状态与任务连续性。
- Admin / Partner 的高密度数据操作、角色范围、审计和危险操作。
- 分批迁移方式、验收门禁、证据等级和旧主题退出条件。

### 4.2 本规范不做

- 不新增首页入口、底部 Tab、同义卡片或新菜单层。
- 不改变现有路由、API、DTO、数据库、权限或业务状态机。
- 不在视觉迁移中夹带支付、认证、打印、扫描、AI 或硬件逻辑改造。
- 不复制 4188 原型的演示登录、假数据、原型弹窗或路由提示。
- 不把未实现能力做成看似可用的卡片、按钮或成功状态。
- 不建立 Enterprise / HR / 候选人端，不改变招聘平台合规边界。
- 不把 Kiosk 的大卡片直接复制到后台，也不把后台密集表格压进 Kiosk。
- 不在本文阶段删除 `inkpaper.css`、`fusion-youth.css` 或任何运行时代码。
- 不在用户审阅本文前编写实施计划或启动正式页面迁移。

## 5. 设计语言：Blue-White Service Desk

### 5.1 视觉性格

关键词：

- 明亮、可信、清楚、友好；
- 就业公共服务台，而不是营销落地页；
- 现代商业软件，而不是炫技 AI 面板；
- 有品牌温度，但优先保证办理效率；
- 可识别的分类色，但状态色永远保持独立语义。

必须避免：

- 暖米纸主画布、墨绿主色、宋体大标题继续作为现行主题；
- 大面积深色背景、霓虹、紫蓝 AI 渐变或持续光晕；
- 每张卡都使用不同高饱和色；
- 依赖阴影区分所有层级；
- 用圆角和留白牺牲后台数据密度；
- 只有 hover 才能发现的重要操作；
- 把装饰色当作成功、警告或错误状态。

### 5.2 核心语义色

4188 原型中的原始值要映射为共享语义 token，不允许页面长期散落硬编码色值。

| 语义       | 建议 token                  |    基准值 | 用途                         |
| ---------- | --------------------------- | --------: | ---------------------------- |
| 外层环境   | `--sd-color-canvas-outer`   | `#e9f1fb` | 浏览器外缘、Kiosk 舞台之外   |
| 页面画布   | `--sd-color-canvas`         | `#f7faff` | 三端页面主背景               |
| 表面       | `--sd-color-surface`        | `#ffffff` | 卡片、侧栏、顶栏、弹层       |
| 强标题     | `--sd-color-text-strong`    | `#071a43` | 页标题、关键数字、主导航     |
| 次标题     | `--sd-color-text`           | `#17345d` | 卡片标题、正文强调           |
| 正文       | `--sd-color-copy`           | `#566a84` | 说明文字                     |
| 弱正文     | `--sd-color-copy-muted`     | `#64748b` | 元数据；使用前必须验证对比度 |
| 主操作     | `--sd-color-primary`        | `#1769e8` | 主按钮、选中态、链接、焦点   |
| 主操作按压 | `--sd-color-primary-strong` | `#0758d7` | 按压 / active                |
| 分隔线     | `--sd-color-line`           | `#dce6f4` | 卡片和表格分隔               |
| 强分隔线   | `--sd-color-line-strong`    | `#cbd9ed` | 输入框、重点分隔             |

状态语义继续使用独立 token，并在实现时校验 WCAG 对比度：

| 状态 |      前景 |      背景 | 规则                   |
| ---- | --------: | --------: | ---------------------- |
| 成功 | `#15803d` | `#f0fdf4` | 只用于真实完成或健康   |
| 警告 | `#c2410c` | `#fff7ed` | 需要注意但尚未失败     |
| 错误 | `#b91c1c` | `#fef2f2` | 失败、阻断、危险操作   |
| 信息 | `#1d4ed8` | `#eff6ff` | 中性提示、处理中、帮助 |

分类色只用于模块归类、图标背景和弱强调，不表示状态：

| 分类   |      浅底 |      强色 |
| ------ | --------: | --------: |
| 蓝     | `#edf6ff` | `#0f64c5` |
| 薄荷   | `#eefaf6` | `#087f6a` |
| 橙     | `#fff7ef` | `#a94c12` |
| 薰衣草 | `#f5f3ff` | `#594bc0` |
| 青     | `#effbff` | `#067b9d` |
| 沙金   | `#fffaf0` | `#93620e` |

同一屏幕同时出现的强分类色建议不超过 3 种；其余使用浅底、描边或中性色，避免「彩色卡片墙」。

### 5.3 字体与层级

现行字体方向改为现代无衬线：

`SF Pro Display` → `PingFang SC` → `HarmonyOS Sans SC` → `Microsoft YaHei` → `Noto Sans SC` → system-ui。

实施时不依赖某一台机器必然存在的字体，也不因视觉迁移强制新增外部字体下载。

| 层级        |             Kiosk |   Admin / Partner |
| ----------- | ----------------: | ----------------: |
| 品牌 / Hero | 48–64px / 700–800 |  不使用 Hero 字号 |
| 页面标题    |     36–44px / 700 |     24–28px / 700 |
| 区块标题    |     28–32px / 700 | 18–20px / 650–700 |
| 卡片标题    | 20–24px / 650–700 |     15–16px / 600 |
| 正文        | 16–18px / 400–500 |    14px / 400–500 |
| 辅助文字    |           14–16px |           12–13px |

数字、金额、页数、设备计数和表格统计使用等宽数字；长 ID、文件名、来源 URL 必须有可复制和完整查看路径，不靠截断后猜测。

### 5.4 间距、圆角与阴影

- 以 4px 为基础网格，常用间距为 8 / 12 / 16 / 20 / 24 / 32 / 40 / 48。
- Kiosk 主表面圆角使用 20 / 26 / 34px；大圆角只用于 Hero、主要服务面板和大弹层。
- Admin / Partner 圆角收敛为 10 / 14 / 18px，表格单元格不做独立大圆角。
- Kiosk 卡片外阴影基准：`0 18px 60px rgba(30, 71, 124, 0.09)`。
- 后台阴影使用更轻版本：`0 10px 34px rgba(30, 71, 124, 0.06)`，数据区优先用边框和层级，不靠浮起。
- 禁止用阴影表示在线、成功或已完成；状态必须由文字、图标和语义色共同表达。

### 5.5 图标、图片与人物资产

- 同一端内使用同一图标库与统一笔画 / 填充规则。
- 图标必须配文字或可访问名称；关键状态不能只靠图形。
- Kiosk 使用项目已授权的小青真人透明背景资产，保持清晰、真实、非卡通。
- 小青用于 AI 助手、首页顾问入口和明确的顾问语境，不扩散到 Admin / Partner 的普通运营页。
- 不以 AI 生成的另一张人像替换小青；肖像授权、裁切和背景处理须保持一致。
- 业务图片必须有占位、加载失败和替代文本；不使用假企业、假岗位或假设备照片填空。

### 5.6 动效

| 类型            |      时长 | 说明                           |
| --------------- | --------: | ------------------------------ |
| 按压反馈        | 100–140ms | 轻微缩放或位移，立即确认已触发 |
| 状态切换        | 180–220ms | 颜色、透明度、徽章切换         |
| 元素进入        | 200–240ms | 一次性淡入 / 上浮              |
| 页面 / 复杂流程 | 280–360ms | 只用于帮助理解层级             |

动效必须：

- 优先只改变 `opacity` 和 `transform`；
- 不延迟按钮真实响应；
- 不对长列表全部 stagger；
- 不用持续呼吸表现未知或离线；
- 不在支付、打印、扫描尚未到达真实终态时播放成功动画；
- 在 `prefers-reduced-motion` 下退化为瞬时切换或不超过 200ms 的淡入；
- 在一体机真机上验证滚动、输入、虚拟键盘、TRTC 和主要转场不卡顿。

## 6. 统一而不相同：三种密度配置

| 维度              | Kiosk Touch      | Admin Compact             | Partner Comfortable         |
| ----------------- | ---------------- | ------------------------- | --------------------------- |
| 主要用户          | 公共终端求职者   | 高频运营人员              | 机构内容维护人员            |
| 主视口            | 1080×1920        | 1440×1024                 | 1440×1024                   |
| 次视口            | 390×844          | 1280×800                  | 1280×800                    |
| 基础字号          | 16–18px          | 14px                      | 14px                        |
| 最小点击 / 触控区 | 48×48px          | 32×32px，常用控件 36–40px | 36×36px，常用控件 40px      |
| 主按钮            | ≥56px 高         | 36–40px 高                | 40px 高                     |
| 表格行高          | 非主要呈现方式   | 44–48px                   | 48px                        |
| 信息密度          | 低，一屏一主任务 | 高，支持扫描和批量判断    | 中，强调流程与说明          |
| 主导航            | 固定三项底部导航 | 左侧分组导航 + 顶栏       | 共享后台壳层 + 清晰机构身份 |
| 主要交互          | 触控 / 虚拟键盘  | 鼠标 + 键盘               | 鼠标 + 键盘                 |

三端共享颜色、字体、状态、表面和交互语义；不得共享错误的信息密度。统一视觉不是把所有页面做成同一尺寸的卡片。

## 7. Kiosk 商用 UX

### 7.1 信息架构

- 底部导航固定为：首页 / AI助手 / 我的，不新增第四或第五项。
- 首页保留既有六组、27 个固定入口及真实启用 / 禁用状态，不新增重复入口。
- 首页主入口帮助用户快速定位服务；复杂流程仍保留必要的确认、价格、隐私和硬件步骤。
- 「我的」只做入口与概览，明细继续归位到对应业务页，不恢复资产中心堆叠。
- 智慧校园、百宝箱和终端配置能力遵守既有后端策略；未配置时诚实隐藏或显示受控状态。

六组 / 27 项是 2026-07-11 设计快照；正式迁移验收必须重新读取当前首页服务源与路由，数量变化只能来自独立产品决策，不能由视觉原型反向修改业务清单。

### 7.2 每屏结构

每个 Kiosk 页面优先按以下顺序组织：

1. 当前身份、设备 / 网络状态或返回路径。
2. 当前任务标题和一句话说明。
3. 本屏唯一主操作；次操作不与主操作争夺视觉。
4. 输入、预览、价格、隐私或来源等完成任务所需信息。
5. 进度、结果、错误恢复和下一步。
6. 必要的合规 / 隐私 / 帮助说明。

「开始一个任务」应在不超过 3 次清楚决策内可达，但不得为了减少步数跳过付费确认、敏感数据授权、打印参数、外部来源或不可逆操作确认。

### 7.3 公共终端边界

- 页面初始状态不得展示上一位用户的简历、对话、订单、手机号或历史。
- 退出、超时、切换账号和完成敏感流程后，清理内存会话与可见敏感信息。
- 登录前后如需要保留草稿，只能使用既有安全会话机制；不得把简历正文、AI 对话或付款码写入长期浏览器存储。
- 手机号、取件码、文件名和个人字段按最小必要显示；共享屏幕上默认脱敏。
- 会话即将过期时先提醒，允许在安全范围内续期或保存，不突然清空正在输入的内容。
- 小青 / AI 对话未落库时必须明确「本次会话使用」，不得显示「历史已保存」。

### 7.4 触控与输入

- 所有可交互区域最小 48×48px，主要按钮至少 56px 高。
- 相邻高风险操作保留至少 12px 视觉间距，避免误触。
- 瓦片整块可点；图标、标题和说明不得分别形成多个同义点击目标。
- 虚拟键盘出现后，当前输入、错误和主操作不能被遮挡。
- 拖拽、滑动、长按均必须提供单击替代。
- 按下立即有视觉反馈；请求进行中防重复提交，但仍提供取消或状态说明。

### 7.5 身份、文件与 AI 恢复

- 身份至少覆盖：游客、登录中、已登录、登录失败、token 即将过期、已过期、即将自动退出、已安全清场。
- 文件至少覆盖：选择、上传、校验拒绝、处理中、可用、已过期、已删除和重新上传。
- AI 至少覆盖：排队、生成中、流式中断、用户事实确认、重试、取消和安全草稿恢复。
- AI 一键调整不得未经确认覆盖用户编辑稿；有撤回能力时必须清楚可见。
- Kiosk 异常结果优先提供任务号、返回订单 / 记录、再次尝试和现场帮助入口，降低重复提交或重复付款风险。

## 8. Admin 商用 UX

### 8.1 壳层

- 采用白色 / 浅冰蓝轻壳层：白色侧栏、白色顶栏、冰蓝画布、蓝色选中态、深海军蓝文字。
- 不沿用墨绿侧栏和暖纸背景。
- 侧栏保持当前分组、权限和折叠能力；视觉迁移不得改变菜单可见性。
- 顶栏显示面包屑、环境 / 账号身份、全局状态和必要快捷入口，不堆装饰。
- 页面标题、说明、主操作、筛选和内容区形成稳定顺序。

### 8.2 高频运营路径

后台默认优化以下顺序：

> 发现问题 → 筛选缩小范围 → 查看证据 → 执行动作 → 获得处理结果 → 留下审计。

必须支持：

- 搜索、筛选、排序、分页状态清楚并尽量可回到原位置。
- 10 / 50 / 100 行真实数据下表格不破版，表头、关键列和行操作可定位。
- 背景刷新保留当前数据和滚动位置，不用全屏闪白。
- 列表进入详情、抽屉关闭后恢复到触发行和焦点。
- 批量操作明确作用对象数量、范围和不可逆影响。
- 处理完成后更新原行、计数和详情，不只弹一个 Toast。
- 失败时保留筛选、表单和已输入内容，并给出可执行恢复动作。
- 审核、发布、下架、退款、文件清理和终端操作显示目标、影响范围、最终结果与可查审计编号；批量任务显示各项结果。

### 8.3 数据表与操作

- 表格用于比较和运营判断，不把每一行改成大卡片。
- 关键列优先：对象、状态、来源 / 归属、更新时间、异常 / 金额、操作。
- 非关键字段进入详情抽屉或可配置列，避免横向无限扩张。
- 状态徽章必须同时包含文字；颜色只是辅助。
- 行操作优先使用明确动词；危险操作不放在视觉主位。
- 删除、退款、撤销、停用、发布、重分配等操作必须显示对象和后果。
- 支付、订单、打印任务、退款、设备状态分别呈现，不压成一个「完成 / 失败」字段。
- Admin 查看用户敏感文件必须继续受短期签名 URL、权限和审计约束。

### 8.4 后台响应式边界

- 1440×1024 为主验收位，1280×800 为最小正式桌面验收位。
- 1024 宽度下至少可折叠侧栏并完成核心只读与单条操作，不允许白屏或控件完全消失。
- 首轮迁移不把手机作为后台主操作设备；低于 1024 的复杂批量流程可提示使用桌面，但仍须保留安全退出和只读信息。

## 9. Partner 商用 UX

### 9.1 与 Admin 的统一和区分

Partner 与 Admin 复用后台壳层、表格、筛选、表单、抽屉和状态组件，但必须一眼看出当前是机构空间：

- 顶栏持续显示机构名称、机构角色和当前数据范围。
- Partner 身份点缀可使用沙金或薰衣草浅底，但主操作仍使用品牌蓝。
- 不通过另一套主色或整套主题制造身份差异。
- 菜单、按钮和字段只反映 Partner 白名单能力，不显示后再前端置灰 Admin 权限。
- 空态和错误文案明确「本机构」范围，避免误以为是全平台数据。

### 9.2 机构内容工作流

Partner 的主要闭环是：

> 准备内容 / 数据源 → 校验 → 提交 → 等待审核 → 查看结果 → 修正并重新提交 → 观察同步与展示表现。

必须清楚区分：

- 草稿、待审核、审核中、已通过、已驳回；
- 已通过与已发布；
- 已发布与已下架；
- 导入请求已接收与数据已成功入库；
- 同步成功、部分成功、失败和下一次计划；
- 本机构数据与其他机构不可见数据。

编辑后会重置为 `pending + draft` 的内容，必须在保存前说明影响，保存后显示新的审核状态。

### 9.3 导入与同步

Excel、API、Webhook 三种来源分别覆盖：

- 等待校验；
- 校验失败；
- 重复数据；
- 部分成功与失败行下载；
- 凭证未配置 / 已配置但不回显；
- 鉴权失败、限流、超时；
- 计划同步、执行中、成功、部分失败、失败；
- 最近一次同步时间与下一次计划时间。

批量导入不得用一个「成功」Toast 代替结果摘要；必须显示成功数、跳过数、失败数和可恢复动作。

### 9.4 Partner 禁止项

- 不展示求职者简历、AI 报告、手机号、邮箱或候选人画像。
- 不展示其他机构的明细数据。
- 不提供自审自发、候选人筛选、面试邀约或 Offer 管理。
- 不回显 API Key、Webhook Secret 等凭证明文。
- 不以「统一后台」为理由复用 Admin 的越权操作。

## 10. 状态系统

### 10.1 全局状态矩阵

每个页面和关键组件在设计审查时必须声明适用状态；没有该状态也要说明原因。

| 状态              | 必须表达                              | 必须提供                           | 禁止                           |
| ----------------- | ------------------------------------- | ---------------------------------- | ------------------------------ |
| 初始未开始        | 任务目的、所需材料 / 条件             | 清楚的开始动作                     | 空白页或自动触发高风险动作     |
| 初次加载          | 正在加载什么                          | 骨架或局部进度                     | 先显示假 0、假空态             |
| 后台刷新          | 旧数据仍可读、正在更新                | 更新时间、必要时手动刷新           | 全屏闪白、丢失滚动和筛选       |
| 首次空态          | 为什么没有数据                        | 创建、导入、去完成前置条件或返回   | 用假数据填满                   |
| 筛选无结果        | 当前筛选导致无结果                    | 清除 / 调整筛选                    | 与「系统没有任何数据」混为一谈 |
| 部分数据          | 哪些成功、哪些失败                    | 查看失败项、重试或下载报告         | 整批显示成功                   |
| 禁用 / 建设中     | 为什么不可用                          | 前置条件、替代路径或诚实的暂无计划 | 可点击假按钮                   |
| 权限不足          | 当前身份和缺少的权限                  | 返回、切换合法账号或联系管理员     | 把权限问题写成输入错误         |
| 离线 / 设备不可用 | 网络、Agent、打印机或外设哪一层不可用 | 重试、检查项、返回订单 / 任务      | 播放成功动画或写死「设备正常」 |
| 提交中            | 已收到操作、当前处理对象              | 防重复、取消或离开后的查看路径     | 允许重复付款 / 重复提交        |
| 未知 / 待核实     | 系统尚不能确认最终结果                | 查单、刷新、联系客服 / 现场人员    | 归为成功或失败后鼓励重新支付   |
| 成功终态          | 具体完成了什么                        | 回执、记录位置、下一步             | 只有庆祝动画                   |
| 可恢复失败        | 失败原因和未受影响部分                | 修正、重试、返回、保留输入         | 清空表单                       |
| 不可恢复失败      | 当前任务不能继续的原因                | 安全退出、重新开始、人工支持       | 无限重试                       |
| 会话过期          | 数据是否已安全保存                    | 重新登录、继续或安全清场           | 突然跳转且无说明               |
| 已取消 / 已过期   | 取消 / 过期对象与时间                 | 重新创建或返回记录                 | 当作普通失败                   |

### 10.2 状态呈现规则

- 状态 = 图标 + 文字 + 语义色；关键状态增加时间或阶段。
- 「0」只表示真实返回的 0；加载失败或未知显示「—」并说明。
- Toast 只做补充反馈，不能是操作结果的唯一载体。
- 进度条只用于有真实阶段或可计算进度的任务；未知等待使用阶段文案，不伪造百分比。
- 错误代码、requestId、任务号放在可复制的次级区域，主文案先说人话。
- 加载失败、离线或权限不足不使用「暂无数据」掩盖。
- 禁用控件必须有可访问的原因说明，不能只降低透明度。

### 10.3 错误恢复文案模板

错误页、错误卡和表单错误至少回答：

1. 发生了什么。
2. 哪些内容已保存或未受影响。
3. 用户现在可以做什么。
4. 如需人工定位，提供 requestId / 任务号 / 时间。

表单校验失败时：

- 页面顶部给出错误摘要并把焦点移到摘要；
- 字段旁显示同义且具体的错误；
- 保留用户已输入的正确和错误值；
- 不使用「未知错误」「参数非法」「Forbidden」作为主要用户文案。

### 10.4 状态优先级

当多个状态同时存在时按以下顺序处理，避免互相覆盖：

1. 安全 / 权限 / 会话边界。
2. 支付未知、重复提交风险和不可逆操作。
3. 设备 / 网络阻断。
4. 业务处理中或部分失败。
5. 空态、筛选结果和普通提示。
6. 装饰性动效。

## 11. 业务事实不得折叠

商用页面必须按真实对象分别呈现状态，不能用一个通用「完成」替代。

| 场景              | 必须区分                                                   | 不能宣称                          |
| ----------------- | ---------------------------------------------------------- | --------------------------------- |
| 支付与打印        | 支付尝试、订单支付、打印任务、退款 / 对账                  | 支付成功 = 已打印                 |
| 打印硬件          | 已建单、待领取、已领取、打印机执行、系统回传、物理出纸证据 | 后端 `completed` 自动等于纸张已出 |
| 扫描              | 已发命令、设备扫描、文件上传、文件可用                     | 定时器结束 = 扫描完成             |
| AI                | 排队、生成中、流式中断、用户确认、成果保存                 | 有回复 = 已保存 / 已通过事实校验  |
| 文件              | 已选择、上传、校验、处理、可预览、过期 / 删除              | 上传请求发出 = 文件可打印         |
| Partner 导入      | 请求接收、校验、成功 / 跳过 / 失败、审核、发布             | 导入成功 = 已审核发布             |
| 外部岗位 / 招聘会 | 打开来源入口、第三方后续未知                               | 已打开 = 已投递 / 已预约 / 已签到 |
| Admin 批量操作    | 成功项、失败项、未处理项                                   | 部分成功 = 全部成功               |

支付状态为 `pending / unknown / USERPAYING` 等待核实时，界面必须锁住会造成重复扣款的动作，提供查单 / 核实路径，不鼓励重新扫码。

来源链接失效、二维码过期、第三方页面打不开或跳转日志写入失败时，用户仍可返回详情、复制来源信息、刷新二维码或重新打开来源入口；日志失败不得阻断合法外部跳转，也不得伪造第三方办理结果。

## 12. 共享组件行为标准

实现计划可调整具体组件名称，但必须覆盖以下能力：

| 组件模式            | 商用行为                                                        |
| ------------------- | --------------------------------------------------------------- |
| 页面壳层            | 统一画布、导航、身份、环境与焦点顺序                            |
| PageHeader          | 标题、说明、主操作、次操作和作用范围                            |
| Surface / Card      | 清楚层级；卡片不是所有内容的默认容器                            |
| ActionButton        | loading、disabled reason、防重复、危险级别                      |
| StatusBadge         | 文字 + 图标 / 点 + 语义色，不用分类色替代                       |
| StatePanel          | loading、empty、no-result、error、permission、offline、disabled |
| FilterBar           | 搜索、组合筛选、清除、结果数、刷新时间                          |
| DataTable           | 排序、分页、选中数、长内容、固定表头 / 关键列                   |
| DetailDrawer        | 焦点陷阱、Esc、关闭后焦点恢复、对象上下文                       |
| FormSection         | label、hint、必填、字段错误、错误摘要、草稿保护                 |
| ProgressTimeline    | 真实阶段、当前状态、时间、失败节点、恢复动作                    |
| ComplianceNotice    | 来源、隐私、保存期限、外部办理边界，固定可见                    |
| SensitiveDataNotice | 脱敏、访问原因、审计提示和会话清场                              |
| ResultReceipt       | 结果、编号、保存位置、有效期、下一步                            |

通用组件必须有 Kiosk / Admin / Partner 密度变体，不复制三份相同实现，也不让页面通过大量局部 CSS 改回另一套风格。

## 13. 内容、合规与隐私

### 13.1 行动文案

- 使用明确动词：「保存修改」「提交审核」「重新同步」「确认打印」。
- 避免无上下文的「确定」「处理」「去看看」。
- 危险操作写明对象和后果：「撤销该机构的发布权限」。
- 外部投递 / 预约只使用项目合规白名单。
- 建设中能力写明不可用原因，不用营销文案暗示已经上线。

### 13.2 必须可见的信息

- 岗位 / 招聘会：来源机构、同步时间、来源说明、外部办理入口。
- 文件 / 简历：保存期限、到期时间、删除和长期保存边界。
- AI：输入来源、事实确认、失败兜底、是否保存。
- 付费：价格、抵扣、应付、退款 / 重试规则。
- 后台写操作：作用范围、权限、审计结果。
- Partner：本机构范围、审核状态、发布状态和凭证不回显。

### 13.3 禁词覆盖面

合规禁词检查不能只扫描 CTA，还应覆盖：

- 标题、空态、错误、Toast 和通知；
- AI 回复与推荐操作；
- 二维码 / 外链弹层；
- 导出内容和打印回执；
- Admin / Partner 菜单、表格和批量操作。

### 13.4 公共终端清场清单

会话结束或超时后必须清理：

- 登录 token 与内存会话；
- AI 对话和 intent 上下文；
- 文件选择、预览、Object URL 和表单草稿；
- 虚拟键盘内容；
- 可回退到个人页面的浏览器历史 / 页面状态；
- 屏幕上可见的订单号、手机号、取件码和个人字段。

处于付款、打印、扫描、上传或 AI 长任务的忙碌态时，不得被屏保或普通闲置超时粗暴中断；应先遵守既有 busy lock 与任务恢复机制。

## 14. 可访问性与设备适配

### 14.1 基准

- 以 WCAG 2.2 AA 为 Web 基线；Kiosk 触控目标使用比 WCAG 最低值更严格的项目标准。
- 普通文字对比度至少 4.5:1，大字至少 3:1，非文字状态与焦点至少 3:1。
- 文本放大至 200% 不丢失内容或功能；后台复杂表格可使用受控横向滚动，但操作列和上下文仍可定位。
- 焦点必须可见且不被固定导航、软键盘、抽屉或 Toast 完全遮挡。

### 14.2 键盘与辅助技术

- 所有按钮、链接、输入、选择器、菜单、Tab、表格操作和自定义控件可用键盘访问。
- DOM 顺序与视觉顺序一致，不用正 `tabindex` 人工拼顺序。
- 弹窗 / 抽屉打开后焦点进入，Tab / Shift+Tab 闭环，Esc 关闭，关闭后回到触发控件。
- 状态变化使用适当的 `aria-live`，避免频繁播报后台轮询。
- 图标按钮有可访问名称；标签可见文字与 accessible name 一致。
- 不以拖拽、滑动、颜色、声音或动效作为唯一完成方式。

### 14.3 长内容和边界数据

必须验证：

- 两倍长度中文标题和说明；
- 长机构名、长文件名、长来源 URL、长错误信息；
- 0、1、10、50、100 条表格数据；
- 超大金额 / 页数、空值、未知值和极端时间；
- Windows 字体回退和 125% / 150% 系统缩放；
- Kiosk 软键盘、浏览器缩放和横竖屏异常状态。

## 15. 响应与性能体验

- 点击后 100ms 内给按压或处理中反馈。
- 短于约 300ms 的请求避免闪烁式全屏 Spinner；超过该时间显示局部骨架或进度。
- 超过 1 秒的任务说明正在处理的对象；长任务允许离开后从记录重新进入。
- 背景刷新不得清空当前列表、筛选、分页和滚动位置。
- 动画队列封顶，避免对 50 / 100 行表格逐行入场。
- 不在 Kiosk 大面积长期使用 blur、filter 或常驻 `will-change`。
- 图片提供正确尺寸、失败占位和懒加载；小青首屏资产须避免模糊和布局跳动。
- 页面不得因接口失败出现白屏；控制台不得有 React 错误、未处理 Promise 或持续请求风暴。

体验指标只记录低敏摘要，可用于观察：

- 进入 / 放弃 / 完成；
- 错误类型与恢复成功率；
- 支付核实、打印耗时和硬件失败；
- 外链失效率、同步失败率；
- 不记录简历正文、AI 完整对话、付款码、密钥或个人敏感字段。

## 16. 迁移策略选择

### 16.1 未选方案

**一次性全量换装**

- 优点：短期视觉统一。
- 风险：三端同时变化，回归面过大，难定位业务与样式问题。
- 结论：不采用。

**逐页局部 CSS 模仿 4188**

- 优点：单页启动快。
- 风险：token、圆角、状态、密度和交互继续分叉，形成第三套主题。
- 结论：不采用。

### 16.2 选定方案

采用「共享基础 → 代表页 → 业务域批次 → 旧主题退出」：

- 一个共享语义 token 层。
- 三种密度配置。
- Admin / Partner 共用后台壳层和基础组件，角色身份通过内容与小范围点缀区分。
- Kiosk 复用同一视觉 DNA，但使用触控布局。
- 页面只消费语义 token 和共享组件，不复制原型代码。
- 先证明真实状态和业务闭环，再扩大覆盖范围。

## 17. 迁移路线约束（非实施计划）

本节只定义不可跳过的设计验证顺序，不决定具体页面清单、文件、分支、PR、人员或工期；这些内容必须在本文经用户审阅生效后，由单独的实施计划确定。

### UI-0：标准与基础层

目标：先建立单一 authority 与共享基础，不改变业务行为。

- 固化本规范、原型证据边界和旧文档退役关系。
- 定义共享 token、密度、状态、组件与三端壳层合同。
- 盘点旧主题、局部硬编码和旧视觉验证的影响面。
- 明确真实路由、API、权限、busy lock、TRTC 清理、会话隔离和状态机的零行为变化边界。

### UI-1：代表性界面验证

用少量正式页面验证三种密度和真实状态：

- Kiosk 至少覆盖首页型、任务型和个人入口型界面。
- Admin 至少覆盖工作台型和高密度表格 / 详情型界面。
- Partner 至少覆盖机构列表型和表单 / 审核状态型界面。
- 代表性界面未通过用户与验收门禁前，不扩大覆盖范围。

具体代表页由后续实施计划根据当时主线、文件占用、业务风险和用户优先级选择。

### UI-2：按业务域扩展

- 只在共享基础和代表性界面通过后，按真实业务闭环小批扩展。
- 每批保持可独立验证、可回滚，不把所有页面一次性横切。
- 有功能缺口的页面先诚实禁用或另立功能任务，不在视觉批次补假闭环。

### UI-3：其余页面与边界状态

- 完成低频页面、登录 / 404 / 权限 / 服务不可用等系统页。
- 补齐长内容、极端数据、批量部分失败和历史记录恢复。
- 不把功能缺口混在样式提交中；缺少 API / 状态机时另立功能任务，并在 UI 上诚实禁用。

### UI-4：旧主题退出与商用终验

- 证明无路由、import、测试、文档或生产构建依赖后再移除旧主题。
- 全仓确认不存在无人维护的第二套视觉 token。
- 完成三端视觉、功能、状态、权限、可访问性、预生产和硬件验收。
- 同步进度文档与正式视觉入口。

## 18. 每页迁移准入

每页开工前必须写清：

1. 页面对应哪个已有入口和真实业务闭环。
2. 当前路由、API、权限、状态机和数据来源。
3. 允许修改文件与禁止触碰文件。
4. UI 只缺表现，还是缺少真实 API / 业务状态。
5. 适用的全局状态矩阵。
6. Kiosk / Admin / Partner 密度配置。
7. 需要保留的合规、隐私、来源和审计信息。
8. 验证命令、浏览器视口、真实账号和外部硬件边界。

如果缺少真实数据契约：

- UI 可以显示诚实禁用 / 暂不可用状态。
- 不能在视觉迁移 PR 内补一个假闭环。
- 需要另立功能任务，经方案、TDD、安全和业务验收后再点亮。

## 19. 验收门禁

### 19.1 视觉门禁

- 与 4188 的蓝白、深海军蓝、白表面、浅冰蓝画布和克制分类色方向一致。
- 三端共享视觉 DNA，但密度、尺寸和任务结构符合角色。
- 不残留暖米纸、墨绿主色、宋体大标题作为现行页面主语言。
- 默认、hover、focus、active、disabled、loading 和 danger 状态完整。
- 页面在目标视口、系统字体回退和长内容下不破版。

### 19.2 功能与状态门禁

- 原有路由、请求、筛选、分页、抽屉、表单、取消和返回行为不回归。
- 每个启用入口使用真实路由 / API，不复制原型演示逻辑。
- 全局状态矩阵按页面适用范围完成。
- 错误可恢复并保留用户输入；未知态不被伪装成成功。
- 完成后有回执、记录位置和下一步。

### 19.3 三端与权限门禁

- Kiosk 只展示用户能力，不出现后台入口。
- Partner 只看本机构数据，不见个人敏感信息和 Admin 操作。
- Admin 的敏感访问、审核和危险操作继续受权限与审计约束。
- 登录过期、越权、禁用账号和机构隔离有真实负向测试。

### 19.4 可访问性与交互门禁

- 对比度、键盘、焦点、弹层闭环、屏幕阅读器名称、reduced motion 通过。
- Kiosk 所有交互 ≥48px、主按钮 ≥56px。
- 390×844 无横向溢出；1080×1920 触控路径和固定底栏稳定。
- Admin / Partner 1440×1024 与 1280×800 完成表格、抽屉、弹窗和长内容测试。

### 19.5 工程门禁

- 相关 typecheck、lint、build、`verify:*` 通过。
- SQLite 主 CI 与 `postgres-readiness` 不因视觉迁移回归。
- 生产构建不包含 DEV 演示按钮、假数据或原型调试面板。
- 浏览器控制台无应用错误和请求风暴。
- diff 只覆盖本批允许文件；样式迁移不夹带业务逻辑。
- 超过 30 行或跨模块变更按项目规则完成双模型审查；外部模型不可用时必须如实记录，不得写成已通过。

### 19.6 真实环境门禁

- 本地截图或原型 `passed` 只证明视觉参考，不证明正式页面。
- 正式页面需真实 API 浏览器回归和角色负向测试。
- 预生产需验证真实数据、错误、过期、权限和恢复。
- 打印 / 扫描需 Windows、Terminal Agent、奔图真机和异常场景证据。
- 分别记录「系统链路完成」与「物理输出确认」，不能合并表述。
- 支付需真实渠道、金额、回调、查单、退款 / 对账证据，不以 UI 模拟代替。

## 20. 证据包

每批至少保留以下低敏证据，截图原件优先仓库外保存：

- 基线与迁移后目标视口截图。
- 默认、加载、空态、错误、禁用、权限、长内容和极端数据截图。
- 主要交互清单与实点结果。
- 真实 API / 网络结果的脱敏摘要。
- 角色隔离和越权负向结果。
- 可访问性、焦点、触控和 reduced motion 结果。
- typecheck、lint、build、verify 输出。
- 预生产 / 真机证据边界。
- 未完成项、外部依赖和不得宣称完成的范围。

不得把真实手机号、简历、付款码、密钥、完整 token、个人文件、数据库备份或未脱敏 HAR 放入 Git。

## 21. 商用完成等级

| 等级 | 含义                   | 可以宣称               | 不可以宣称                |
| ---- | ---------------------- | ---------------------- | ------------------------- |
| UX-0 | 视觉参考 / 原型        | 方向已确认             | 正式页面已完成            |
| UX-1 | 正式代码本地集成       | 本地视觉与基本交互通过 | 真实业务 / 预生产商用完成 |
| UX-2 | 真实 API 与角色闭环    | 本地集成闭环通过       | 生产、支付或硬件完成      |
| UX-3 | 预生产真实数据验收     | 预生产候选通过         | 正式生产上线              |
| UX-4 | 生产 / 真机 / 运营验收 | 对应范围商用完成       | 未覆盖范围完成            |

普通信息页至少达到 UX-3 才能称为商用候选；支付、打印、扫描和设备能力必须达到对应 UX-4 证据，才能称为正式商用完成。

## 22. Definition of Done

页面只有同时满足以下条件才算本轮迁移完成：

- 使用现行蓝白服务台 token 和正确密度配置。
- 没有旧主题局部回流或页面级平行标准。
- 保留全部真实入口、路由、权限、合规文案和业务状态。
- 适用状态完整且可恢复。
- 启用功能形成真实闭环；未启用功能诚实禁用。
- 完成态给出回执、记录位置和下一步。
- 触控、键盘、焦点、对比度、缩放与 reduced motion 合格。
- 目标视口和边界数据不破版。
- 工程门禁与浏览器回归通过。
- 证据等级和未完成边界记录准确。

「页面看起来像 4188」只满足其中第一项，不能单独作为完成依据。

## 23. 风险与控制

| 风险                               | 控制                                                |
| ---------------------------------- | --------------------------------------------------- |
| 共享 token 一次改变三端            | 新主题先作用域化，代表页通过后再扩大                |
| Kiosk 大卡片污染后台密度           | 三种密度配置 + 后台表格基准                         |
| 原型演示逻辑进入正式代码           | 原型只作视觉参考，正式页复用真实路由 / API / 状态机 |
| 旧文档继续指向 InkPaper            | 给旧文档加历史状态和本规范入口                      |
| 旧 verify 阻止新视觉或拉回旧 class | UI-0 盘点并按新标准更新验证                         |
| 分类色与状态色混淆                 | 分类 token 与语义状态 token 分离                    |
| 样式 PR 夹带功能改造               | 文件预算、纯样式批次、diff 审查                     |
| 页面看起来完成但异常不可用         | 状态矩阵、故障注入和恢复验收                        |
| 公共终端泄露上一用户数据           | 清场清单、busy lock、隐私回归                       |
| 后台美化后效率下降                 | 10 / 50 / 100 行、键盘、批量与恢复位置测试          |
| 一次性全量迁移失控                 | 代表页先行、按业务域小批、每批可回滚                |
| 与并行支付 / 打印任务抢文件        | 一窗口一任务一分支，开工前复核活跃 worktree         |

## 24. 设计治理

- `packages/ui` 的语义 token 与共享组件是运行时视觉 SSOT。
- 页面不得长期硬编码 4188 原始色值；数据可视化的例外色也要进入受控 token。
- 新增组件前先查共享组件，不为单页复制同义组件。
- 设计例外必须记录原因、作用域、回归方式和清理条件。
- 旧主题只有在引用、测试、文档、生产构建和页面验收均清零后才能删除。
- 实施完成后不得保留「新蓝白 + InkPaper + fusion-youth」三套并行主题。
- 每批完成同步正式进度文档；原型 QA 不代替项目进度事实。
