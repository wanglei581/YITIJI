# 合作机构成员账号安全移除设计

**日期：** 2026-07-16

**状态：** 已确认，待实施
**范围：** Admin 合作机构详情中的 `role=partner` 成员账号

## 1. 目标与边界

平台管理员需要能从既有的「合作机构详情」中安全移除某个合作机构成员账号。

本任务只处理**成员账号的访问权和可识别凭据**，不删除合作机构主体，也不触及岗位、招聘会、支付、终端、文件、求职者数据或生产环境。

不新增菜单、页面或 Partner 自助删除入口；仅在现有 Admin 机构详情页增加受控操作。

## 2. 已确认的业务规则

1. 只有平台 `admin` 可执行操作；Partner 账号不能删除自己或其他成员。
2. 仅能移除目标机构内、角色为 `partner`、尚未移除的成员账号。
3. 移除后机构必须仍有至少一个「已启用且未移除」的 Partner 账号。
   - 这里的“有效账号”定义为：`role='partner'`、同一 `orgId`、`enabled=true`、`deletedAt IS NULL`。
   - 因此，操作前若剩余有效账号不足以满足该条件，API 返回冲突；管理员必须先创建并启用接替账号。
   - 该规则同样适用于删除已停用账号：删除后不得让机构处于没有有效登录账号的状态。
4. 已移除账号不可恢复；如需继续使用，由管理员重新创建账号。
5. 原用户名和手机号在移除完成后可重新使用。

## 3. 方案比较与选择

| 方案 | 结论 | 原因 |
| --- | --- | --- |
| 仅停用或前端隐藏 | 不采用 | 保留原用户名/手机号占用，且历史查询易把账号误当作仍存在。 |
| 物理删除 `User` | 不采用 | 会破坏或迫使迁移审计、上传文件等历史外键，追溯性不足。 |
| 墓碑式逻辑移除 | **采用** | 保留历史主键与审计关联，同时立即剥夺访问并释放可复用凭据。 |

## 4. 数据模型与移除语义

`User` 增加可空的 `deletedAt`，并添加面向机构成员查询的索引。迁移必须同时提供 SQLite 与 PostgreSQL 版本；不在本任务执行生产迁移。

移除同一事务中完成以下状态转变：

- 写入 `deletedAt`；
- 设置 `enabled=false`；
- 递增 `tokenVersion`，使已签发令牌立即过期；
- 将 `username` 改为不可登录且全局唯一的墓碑值（例如基于账号 ID）；
- 将密码替换为不可知的随机高强度哈希；
- 清除 `phoneHash`、`phoneEnc`、`phoneVerifiedAt` 与 `lastLoginAt`；
- 将展示名称改为不可识别的墓碑值；
- 写入最小化审计记录：操作者、目标账号 ID、机构 ID、时间与操作类型；不在新增审计 payload 中保留原用户名或手机号。

历史 `AuditLog`、`FileObject` 等仍通过原 `User.id` 保持可追溯；这不是机构注销，也不改写历史业务归属。

## 5. 原子性与并发保护

「最后一个有效账号不可删」不是普通的前端提示，必须由数据库保证。

删除服务将使用短的交互式事务，明确请求可串行化隔离级别：

1. 读取并验证目标账号属于目标机构、角色为 Partner 且尚未移除；
2. 统计机构有效 Partner 账号数；
3. 若执行移除后有效账号数会变为零，则以 `409 LAST_ACTIVE_PARTNER_ACCOUNT_REQUIRED` 拒绝；
4. 以带 `deletedAt: null` 等状态条件的更新写入墓碑，避免读后被并发操作覆盖；
5. 一并写审计；事务提交后发布失效会话状态。

PostgreSQL 在可串行化事务发生冲突时，按有限次数重试 `P2034`；重试后必须重新计数。SQLite 同样走同一事务逻辑。不得用“先 `count`、后事务外 `update`”代替。

## 6. 认证、缓存与并发写入

逻辑移除后的任何认证路径都必须把 `deletedAt` 视为不可用：

- 登录、短信登录、找回密码、自助改密；
- 手机号绑定与首次手机号绑定；
- Admin 的启停与重置密码；
- JWT 守卫和 Partner 机构状态验证。

所有在校验后写入 `User` 的路径，均使用带 `deletedAt: null` / `enabled: true` 的条件更新，并检查受影响行数，避免“先开始绑定或改密、后被删除、最后又写回凭据”的竞态。

删除提交后需要发布比旧版本更高的禁用会话状态，而非仅删除 Redis key。JWT 守卫对缓存中的 Partner 状态还要以数据库中的 `enabled`、`tokenVersion`、`deletedAt` 重新校验，避免并发请求或 Redis 重启把旧会话短暂恢复。

## 7. API 与界面

新增 Admin 专用端点：

```text
DELETE /admin/orgs/:orgId/accounts/:accountId
```

响应约定：

- 成功：`200 { success: true }`；
- 目标不存在、目标不属于机构、目标角色不对或已移除：`404 ACCOUNT_NOT_FOUND`；
- 删除会使机构没有有效 Partner 账号：`409 LAST_ACTIVE_PARTNER_ACCOUNT_REQUIRED`；
- 无 Admin 权限：沿用现有认证与角色守卫的拒绝语义。

Admin 前端复用机构详情的账号管理区：

- 新增危险操作按钮及二次确认对话框；
- 清晰说明“删除后不可直接恢复，原用户名和手机号可重新使用”；
- 删除请求期间锁定同一列表的启停、重置密码和删除按钮；
- 成功后重新加载机构详情和计数；
- 收到 `409` 时提示先新增并启用接替账号；
- 提供键盘可达的确认、取消和焦点恢复行为。

前端提示仅改善体验，不替代服务端授权、状态校验或并发保护。

## 9. 验收标准

后端与安全：

1. Admin 成功移除第二个及以上有效 Partner 账号，历史关联仍可查询。
2. 被移除账号的旧 JWT、登录、短信登录、重置密码、改密和手机号绑定全部失败。
3. 用户名和手机号可被后续新账号使用。
4. 错误机构、非 Partner、已移除账号均不能被再次操作。
5. 两个并发删除请求不能让机构失去全部有效账号。
6. Redis 残留或晚到缓存写入不能接受旧令牌。
7. SQLite 与 PostgreSQL 的 schema / migration / 验证均通过。

前端：

1. 仅既有机构详情出现删除入口。
2. 二次确认、忙碌锁、错误提示和成功刷新完整可用。
3. 最后有效账号被服务端拒绝时，前端展示可执行的接替账号提示。

发布前：

1. 运行最小相关单元、集成、静态验证、类型检查和构建；
2. 检查 Git diff、迁移对称性和无密钥泄露；
3. 完成前后端与安全双模型审查；
4. 仅在真实验证完成后更新进度文档，不把本地验证表述为生产部署。
