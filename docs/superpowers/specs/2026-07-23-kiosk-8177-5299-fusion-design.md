# Kiosk 8177 / 5299 全量融合换装设计

> 状态：已确认设计，待实施计划评审
>
> 日期：2026-07-23
>
> 范围：`apps/kiosk` 全部用户可见页面
>
> 基线：`origin/main`（`1d9060c5`）

## 1. 目标

将当前 Kiosk 前端的全部用户可见页面换装为 8177 与 5299 原型的融合版本：

- 以 5299 的页面完整度、信息组织、版式和视觉呈现为主体；
- 吸收 8177 的任务流程、加载/空/错误/离线/支付失败等状态，以及法律文档入口；
- 保留当前生产代码里的真实路由、接口、数据库、认证、支付、文件、订单、打印、扫描和硬件逻辑；
- 保留已经验证的业务闭环，只调整用户看到的结构、样式、交互反馈和状态呈现；
- 最终形成一套可维护、可验证、可分批回退的正式商用前端，而不是静态演示页。

本任务的核心不是重新开发项目，也不是把 HTML 原型直接复制到 React，而是用融合原型统一现有真实 Kiosk 的呈现层。

## 2. 已核实的前提

8177 与 5299 不是两套独立产品，它们来自同一个生产代码基线，原型目录高度重合：

- 5299 目录的 82 个文件都存在于 8177，8177 另有 8 个独有文件；82 个共同文件中有 79 个内容完全一致；
- 首页 `01-home.html`、共享样式 `shared.css`、共享脚本 `shared.js` 和工具箱页 `76-toolbox-zone.html` 一致；
- 两边共同文件中，原型目录索引 `index.html`、个人中心 `14-profile.html` 与打印上传页 `77-print-upload.html` 存在实质差异；
- 8177 额外补齐了登录错误、意见反馈、支付失败、扫描离线、工具箱空状态等状态页；
- 当前 `origin/main` 已经完成过 75 屏视觉对齐和首页迁移，因此实施应采用“逐路由差异收口”，不能从零重写。

因此，融合应建立在最新生产前端上：5299 提供主视觉和页面组织，8177 提供状态与流程补全，现有 React 代码提供唯一真实业务行为。

仓库内现有的 `docs/design/kiosk-proto-2026-07/` 是已版本化的 75 屏基线。8177 与 5299 是基于同一提交 `1d9060c5` 形成的本地原型增量工作树，不是与该目录竞争的第二、第三套产品标准：

## 3. 不可破坏的边界

### 3.1 必须保留

- 当前全部 Kiosk 路由及其重定向、别名和参数语义；
- 当前 API、DTO、认证会话、支付、文件上传、订单、打印和扫描协议；
- 当前 PostgreSQL/Prisma 数据逻辑和服务端权限边界；
- 当前 Terminal Agent、打印机、扫描仪和硬件状态检测逻辑；
- 当前真实加载、错误重试、断网、登录过期、支付失败和任务恢复行为；
- 当前合规边界与外部来源跳转方式；
- 当前已通过验证的功能入口和用户数据闭环。

### 3.2 明确不做

- 不修改 `apps/admin`、`apps/partner` 和小程序；
- 不引入第二套路由、重复入口、同义卡片或平行页面；
- 不把静态 HTML、假数据或硬编码演示状态当成生产实现；
- 不把扫描入口伪装成文件上传方式；
- 不为复刻状态页而新增没有业务意义的 URL；
- 不新增招聘平台闭环、站内投递、收取简历、候选人管理或企业招聘能力；
- 不在本任务中修改数据库结构、后端契约或硬件协议；若实施中证明确有必要，必须单独提出证据和变更方案，经确认后另立任务。

## 4. 决策优先级

当视觉原型、当前代码和历史说明存在差异时，按以下顺序裁决：

1. 最新生产代码中已验证的真实业务行为；
2. `docs/compliance/`、`docs/product/` 和正式进度文档中的约束；
3. 5299 的视觉结构、内容层级和页面完整度；
4. 8177 的流程、异常状态、空状态和法律入口；
5. 原型共享 token、素材和微交互细节。

视觉不能覆盖业务事实，静态文案不能覆盖真实状态，原型导航不能覆盖生产路由。

## 5. 融合规则

### 5.1 总体规则

- 页面布局、信息密度、卡片组织、导航层级和视觉语言以 5299 为主；
- 8177 中独有且有真实业务意义的状态，合并到对应生产页面的状态分支；
- 每个用户动作继续调用现有 handler、hook、store、service 和 API；
- 原型只负责说明“用户看到什么”，生产代码负责决定“系统实际做什么”；
- 别名路由复用同一页面组件，不复制组件和业务逻辑。

### 5.2 原型索引

原型目录只保留一个内部验收索引：

- 使用 5299 的目录式完整展示作为主体；
- 把 8177 的流程说明、异常状态和覆盖信息并入同一个索引；
- 该索引随 `docs/design/kiosk-proto-2026-07-fusion/` 作为静态设计资产版本化；
- 该索引仅用于设计核对和测试定位，不进入正式 Kiosk 用户导航，也不得被 `apps/kiosk` import、复制到生产静态资源目录或打入生产 bundle。

### 5.3 打印上传页

打印上传页采用 5299 的 2×2 视觉结构，但业务含义必须明确：

- 文件、手机扫码、U 盘是三种真实上传来源；
- 第四个卡片是“前往扫描”的独立任务入口；
- 点击第四个卡片导航到现有 `/scan/start` 流程；
- 扫描不能作为第四个上传标签，也不能绕过现有扫描任务和硬件状态逻辑。

### 5.4 8177 独有状态

8177 的补充页面按以下方式进入生产代码：

| 原型状态 | 生产落点 | 实现规则 |
|---|---|---|
| `15A-login-error` | 现有登录页 | 作为登录失败状态分支，不新增登录路由 |
| `22B-me-feedback` | `/me/feedback` | 对齐现有意见反馈页面和真实提交逻辑 |
| `32A-cashier-failed` | 现有收银台 | 作为支付失败/取消/超时状态，透传现有错误分类，不创建第二套收银台 |
| `34A-scan-offline` | 现有扫描流程 | 显示硬件离线、重试和求助动作，透传真实设备错误分类 |
| `76A-toolbox-empty` | 现有工具箱 | 作为无可用服务时的空状态，不写死服务数量 |
| 法律入口 | `/legal/:doc` | 沿用现有法律文档路由和正式内容 |

## 6. 呈现层架构

### 6.1 分层原则

换装限定在呈现层及其直接测试层：

```text
现有路由与业务状态
        ↓
现有页面容器、hooks、stores、services
        ↓
融合后的页面结构与共享 UI 原语
        ↓
融合 token、页面级样式与响应式规则
```

不得把 API 调用、订单状态机、支付判断、文件处理或硬件判断移动到纯 UI 组件中。

### 6.2 样式职责

- `packages/ui/src/styles/tokens.css`：跨端基础 token，仅在确有跨端复用价值时调整；
- `packages/ui/src/styles/fusion-youth.css`：融合主题的颜色、字号、圆角、阴影和基础组件语义；
- `apps/kiosk/src/styles/prototype-v1.css`：Kiosk 壳层、竖屏布局和共享页面规则；
- 各业务模块样式：只负责所属页面的局部布局和状态，不通过大范围选择器影响其他模块。

禁止用一个不断膨胀的全局 CSS 文件覆盖所有页面，也禁止在页面内散落重复色值、字号和阴影常量。

### 6.3 共享 UI 原语

优先复用或小幅收口以下原语：

- Kiosk 页面壳层、顶部栏和底部导航；
- 页面标题、步骤条、状态标签和提示条；
- 服务卡、任务卡、信息卡和操作卡；
- 主按钮、次按钮、危险动作和文本链接；
- 加载、空、错误、离线、权限不足和成功结果状态；
- 弹窗、底部面板、确认提示和超时提示。

共享原语只承载表现和可访问性，不拥有具体业务流程。

## 8. 状态契约

每个页面至少核对与该业务相关的下列状态：

- 首次加载、局部加载和提交中；
- 有数据、无数据和筛选后无结果；
- 网络错误、接口错误、权限错误和配置缺失；
- 未登录、会话过期和恢复登录；
- 文件无效、上传失败、解析失败和可重试；
- 支付创建中、待支付、成功、失败、取消和超时；
- 打印机/扫描仪在线、忙碌、离线和任务恢复；
- 成功结果、后续动作和返回路径。

状态展示必须来自现有真实状态或可控测试夹具，不能在生产组件中加入演示开关或固定假数据。

错误页面和状态原语必须消费现有错误对象或已定义错误分类。用户文案可以友好化，但诊断信息、重试条件和埋点不得把不同错误压成一个固定“失败”状态；不得在前端臆造后端或硬件未定义的错误码。

换装过程中必须保持：

- 查询参数、路由 state 和返回路径不丢失；
- 文件引用、订单号、支付 token、会话和任务 ID 不丢失；
- 刷新、后退、重试和超时后的行为与当前生产逻辑一致；
- 动态数据长度增加时不截断关键事实，不产生按钮文字异常换行。

## 10. 并行开发边界

W1 的共享基础完成并冻结后，W2–W5 可按业务域交给不同子代理并行实施，但必须遵守：

- 每个文件同一时间只由一个实现者负责；
- 共享 token、壳层和公共组件由单一负责人维护；
- 各业务域不得顺手修改其他域或新增平行基础组件；
- 子任务必须列出允许修改的文件、禁止修改的文件和验收命令；
- 每个波次合入前先做行为回归和代码审查，再进入下一个依赖层。

## 11. 验证与验收标准

### 11.1 工程门禁

每个波次至少执行相关测试，最终必须通过：


上面的 build 命令对应当前部署 runbook 允许的“显式纯文字助手”基线，只用于本地可重复门禁。最终候选还必须按目标部署实际选择的助手模式再构建一次：纯文字模式保持 `VITE_ALLOW_TEXT_ONLY_ASSISTANT=true`；TRTC 模式则使用正式 `VITE_USE_TRTC_CALL` 等配置且不得用该开关绕过门禁。验收报告必须记录实际构建变量的名称与布尔选择，不记录密钥值。

同时执行与改动业务域对应的现有 `verify:*` 脚本，不得删除或放宽生产配置、真实服务、打印诚实性、上传来源、个人中心、登录、简历、面试、岗位、招聘会和校园等既有门禁。

### 11.2 路由与行为验收

- 当前全部用户可见路由均有明确页面组件和参考原型；
- 没有孤儿页面、重复入口、失效别名或新增的无意义路由；
- 所有真实 API 调用、参数、导航和状态恢复行为保持一致；
- 打印、扫描、支付、文件、登录和 AI 关键路径完成端到端回归；
- 服务不可用时明确失败，不用假成功、假数据或静态结果遮盖问题。

### 11.3 视觉验收

- Kiosk 主屏按 1080×1920 竖屏逐路由截图；
- 手机扫码登录和手机上传页按 390×844 截图；
- 与已提交的 `docs/design/kiosk-proto-2026-07-fusion/` 关键边界误差控制在 4px 内，动态内容差异需在审查记录中说明；
- 无横向溢出、内容截断、遮挡、异常滚动和按钮文字折行；
- 主要触控目标不小于 48px，主要按钮高度不小于 56px；
- 加载、空、错误、离线、支付失败和成功状态均有截图与交互验证；
- 动态长文本、空字段、图片失败和慢接口场景不会破坏布局。

大屏与移动辅助页必须使用可识别的页面壳层或作用域根节点隔离响应式规则。390×844 的手机规则不得缩小 1080×1920 一体机触控区，Kiosk 大屏规则也不得让手机页面产生固定大屏宽度或横向滚动。

### 11.4 合规与安全回归

- 岗位和招聘会仅作为第三方/官方来源信息入口；
- 合规动作使用“去来源平台投递/预约”或“扫码投递/预约”等允许文案；
- 不出现站内一键投递、收取简历或企业候选人闭环；
- 视觉换装不绕过鉴权、权限校验、支付校验、文件校验和硬件状态校验；
- 最终代码变更由 Antigravity 与 Claude 双模型审查，并进行安全回归审查。

## 12. 风险与控制

| 风险 | 控制方式 |
|---|---|
| 全局 CSS 影响大量页面 | 先冻结 W1，使用 token 和作用域样式，逐路由截图 |
| 静态原型覆盖真实逻辑 | 页面容器和业务 hooks 保持不变，先写行为回归再换装 |
| 8177/5299 历史说明过时 | 以最新生产代码和正式文档为最高事实来源 |
| 重复路由或组件 | 建立唯一清单，别名只复用现有组件 |
| 动态数据破坏版式 | 增加长文本、空值、错误图片和慢加载测试 |
| 大文件继续膨胀 | 按业务域拆分样式和纯呈现组件，业务逻辑不跨域搬迁 |
| 触控与可访问性退化 | 验证触控尺寸、焦点、对比度、语义标签和键盘路径 |
| 首屏性能下降 | 复用现有懒加载边界，监控构建分包和资源体积 |

## 13. 回退策略

- 每个实施波次使用独立、可审查的提交；
- 共享基础和各业务域分开提交，避免一个提交覆盖全站；
- 视觉或交互回归可按波次回退，不需要数据库迁移或数据修复；
- 若某页面无法在保持真实逻辑的前提下完成换装，该页面保留当前生产实现并记录阻塞，不使用静态替代方案；
- 不删除旧样式前先证明无路由、import、测试和文档依赖。
