# 用户文件上传流程设计

> 日期：2026-06-26
> 状态：方案已由用户确认，本文件只固化设计边界；不代表运行时代码已实现。
> 范围：一体机当前无小程序时的文件上传、游客临时上传、登录账号上传、后续小程序同步到账号。
> 非目标：不做平台投递、不把简历给企业、不新增招聘闭环、不重建「我的」资产聚合中心。

## 1. 结论

当前不需要等小程序上线才能解决文件上传。一体机应先支持两条当前可落地通道：

1. **手机扫码 H5 上传**：Kiosk 生成一次性上传二维码，用户用手机浏览器或微信打开普通 H5 页面上传文件。未来小程序上线后复用同一套上传会话和文件归属规则。
2. **本地 / U 盘上传**：桌面验证期可用浏览器文件选择器；生产一体机改为 Terminal Agent 读取 U 盘或本地中转目录，Kiosk 只展示 Agent 提供的文件列表。

用户是否登录决定文件归属和留存：

- **未登录临时上传**：文件只服务当前会话，`endUserId=null`，默认短 TTL，到期自动清理，不进入「我的简历 / 我的文档」。
- **登录账号上传**：文件最终绑定 `endUserId`，进入对应业务页面。简历类进入「我的简历 / AI服务记录 / 我的文档」，打印类进入「我的文档 / 打印订单」。手机扫码场景先进入待确认会话，由 Kiosk 二次确认后再归户，避免公共屏幕二维码被旁人抢扫污染账号。保存期限按现有文件留存策略执行。

## 2. 当前代码事实

- Kiosk 简历上传已经调用 `kioskUploadFile(file, 'resume_upload', getToken())`，有会员 token 时后端绑定 `endUserId`，无 token 时走匿名临时路径。
- Kiosk 打印上传当前调用通用打印上传，文件目的为 `print_doc`，后续创建 `PrintTask` 和 `Order`。
- 后端已有 `/files/kiosk-upload`，支持 Kiosk 匿名 / 会员 multipart 上传，并在审计里区分 `kiosk_member` 和 `kiosk_anonymous`。
- 后端已有 `/files/upload-intent`，但要求 User 或会员身份；匿名上传仍走 `kiosk-upload`。
- 现有 `FileObject` 已有 `purpose`、`endUserId`、`assetCategory`、`sourceFileId`、`retentionPolicy`、`expiresAt`。
- 当前留存策略中，会员账号内 `resume_upload`、`resume_scan`、`cover_letter` 默认 90 天；其他用途默认短期。`optimized` / `derived` 可长期保存，原始文件首批不允许长期保存。

## 3. 上传入口设计

### 3.1 Kiosk 页面入口

各业务页面不新增首页入口，只在现有上传页中提供上传方式：

- `手机扫码上传`：优先推荐，解决用户文件在手机上的现实问题。
- `U盘 / 本机上传`：现场带文件到机器时使用。
- `从我的文件选择`：仅登录后展示，读取本人已有文件，避免重复上传。

对应业务页：

- AI 简历诊断 / 优化：`/resume/source`
- 文档打印：`/print/upload`
- 后续扫描件：`/scan/start` 或打印扫描业务页

### 3.2 手机扫码 H5 上传

Kiosk 创建一次性上传会话，展示二维码。二维码打开的 H5 上传页只允许在会话有效期内上传文件。

推荐会话字段：

```text
UploadSession
- id
- terminalId
- purpose
- mode: temporary | member
- pendingEndUserId: nullable
- confirmedEndUserId: nullable
- status: pending | uploading | uploaded | confirmed | consumed | expired | cancelled
- fileId: nullable
- expiresAt
- createdAt
```

会话规则：

- 临时会话：不要求手机登录；上传成功后文件 `endUserId=null`，只返回给当前 Kiosk 会话使用。
- 会员会话：由已登录 Kiosk 创建，服务端先把 `endUserId` 写入 `pendingEndUserId`；手机 H5 上传成功后文件仍处于待确认状态，Kiosk 展示文件名、大小、格式和缩略信息，用户在当前机器上确认后才把文件绑定到该会员。
- `mode=member` 但会员 token 缺失或失效时，服务端必须拒绝创建会话，不得静默降级为临时上传。
- 会话默认 10 分钟过期，单次使用；过期后二维码失效。
- Kiosk 轮询或订阅会话状态，上传成功后拿到 `fileId` 和安全元数据。
- H5 页面不显示或保存会员 token；会员归属由服务端会话决定。
- 上传 token 只允许以一次性强随机凭证存在，服务端只存哈希；H5 上传接口必须复用现有文件格式和大小校验，首期以现有 per-purpose 限制为准，例如简历上传继续对齐当前 10MB 口径，避免同一用途两套上限。
- 创建上传会话必须校验可信终端来源，并按终端、IP、账号和会话数量做限流 / 配额控制，避免匿名临时上传被当成公网文件灌入口。

### 3.3 本地 / U 盘上传

当前桌面验证可以继续使用浏览器文件选择器。生产一体机不应依赖系统文件对话框，建议由 Terminal Agent 提供本地文件中转：

```text
U盘 / 本地目录 -> Terminal Agent 扫描白名单目录 -> Kiosk 展示文件列表 -> 用户选择 -> Agent 或 Kiosk 上传到 API
```

约束：

- Agent 只读白名单目录或可移动盘根目录下的安全文件类型。
- Kiosk 只展示文件名、大小、格式，不展示任意系统路径。
- Agent 本地接口只绑定 `127.0.0.1` 还不够。现有本地服务的 Origin 白名单不能单独承担文件桥鉴权，文件列表和上传动作都必须新增 `localAuthToken` 校验；上传动作还必须新增一次性 `actionToken` 校验，防止浏览器本地 CSRF 和重放。
- 上传后仍走统一 `FileObject` 和留存规则。
- 用户取消或流程结束后，Kiosk 清空本地会话状态。

### 3.4 后续小程序上传

小程序上线后不另起一套文件模型。它应复用同一套 `FileObject`、留存策略和会员身份。

两种小程序场景：

1. **账号文件同步**：用户在小程序登录后上传文件，文件直接绑定 `endUserId`，机器登录同一账号后可从「我的文档 / 我的简历」选择。
2. **发送到当前机器**：用户在 Kiosk 上打开上传二维码，小程序扫码后把文件上传到该 UploadSession，上传完成后当前机器立即可用。

## 4. 游客临时上传流程

适用场景：用户只想临时打印、临时 AI 诊断，不想登录账号。

流程：

```text
用户选择业务入口
-> 选择手机扫码上传 / U盘上传 / 本机上传
-> API 写 FileObject(endUserId=null, retentionPolicy=当前短期留存策略)
-> 当前页面拿到 fileId
-> 继续 AI 诊断或打印
-> 会话结束、文件到期或用户取消后自动清理
```

页面必须提示：

- 临时上传不会保存到账号。
- 离开页面、会话过期或文件到期后无法找回。
- 如果需要以后继续使用，请先登录再上传。

禁止：

- 用户后续登录后自动把匿名文件并入账号。
- 在公共一体机上保存可恢复的匿名文件列表。

## 5. 登录账号上传流程

适用场景：用户希望后续继续优化、再次打印、跨设备复用、从小程序同步到一体机。

流程：

```text
用户登录
-> 选择业务入口
-> 选择上传方式
-> API 写 FileObject(endUserId=当前会员) 或在手机扫码场景写待确认 FileObject
-> 手机扫码场景由 Kiosk 展示上传文件信息并二次确认
-> 根据 purpose 和 assetCategory 决定保存期限
-> 业务结果沉淀到对应页面
```

业务归位：

| 上传目的 | purpose | 默认承接页面 | 留存口径 |
| --- | --- | --- | --- |
| AI 简历上传 | `resume_upload` | 我的简历、AI服务记录、我的文档 | 登录会员默认 90 天，可延至 180 天 |
| 纸质简历扫描 | `resume_scan` | 我的简历、我的文档 | 登录会员默认 90 天，可延至 180 天 |
| 求职信 / 推荐信 | `cover_letter` | 我的文档 | 登录会员默认 90 天，可延至 180 天 |
| 通用打印文件 | `print_doc` | 我的文档、打印订单 | 默认只服务本次打印；如要账号长期复用，应提供明确「保存到我的文档」动作 |
| AI 导出成果 | `resume_upload` + `assetCategory=optimized` | 我的文档、我的简历动作链 | 可按规则长期保存 |

对于通用打印文件，不建议所有登录上传都默认保存 90 天。更稳妥的产品口径是：

- 默认按钮：`仅本次打印`
- 可选按钮：`保存到我的文档`

如果选择保存，后端应使用明确的账号文档 purpose 或新增 `job_material` purpose，并进入 90 天 / 180 天留存规则。是否新增 `job_material` 需要在实现分支中单独审查。

## 6. 从我的文件选择

登录用户在 AI 简历或打印页可以选择已有文件，减少重复上传：

- AI 简历页优先读取 `/me/resumes` 的可继续任务，以及 `/me/documents` 中可用于简历解析的 PDF / DOC / DOCX / 图片。
- 打印页读取 `/me/documents` 中可打印文件。
- 列表只展示元数据，点击后再换取短期访问 URL 或直接使用 `fileId` 进入业务流程。

安全规则：

- 只能读取本人 `endUserId` 文件。
- 过期、已删除、跨账号文件不可见。
- 不把 `payloadJson`、签名 URL、COS key 或文件正文放到列表响应。

## 7. 状态和失败处理

上传会话状态：

```text
pending -> uploading -> uploaded -> confirmed -> consumed
pending -> uploading -> uploaded -> consumed
pending -> expired
uploading -> expired
pending/uploading/uploaded/confirmed -> cancelled
```

失败提示：

- 二维码过期：提示重新生成二维码。
- 文件过大或格式不支持：手机端和 Kiosk 同时提示原因。
- 手机上传成功但 Kiosk 断线：用户可在当前账号的「我的文档」找回；临时会话则提示重新上传。
- 会员二维码被他人误扫或抢扫：Kiosk 展示上传文件信息，用户可取消；取消后文件不进入会员账号并按临时文件清理。
- 文件到期：提示重新上传，不展示空白或假文件。

## 8. 合规和隐私边界

- 上传文件只服务求职者本人和打印 / AI 工具服务，不向企业、合作机构或岗位来源方传递。
- 岗位 / 招聘会仍只做来源入口，不接受平台内投递。
- H5 上传页、小程序、Kiosk 都不能保存 COS 密钥或长期公开 URL。
- 公共屏幕上的会员上传二维码必须有 Kiosk 二次确认，不允许仅凭扫码上传就直接写入会员资产。
- 公共一体机必须有无操作自动退出和上传中 busy lock，避免用户文件残留在下一位用户界面。
- 管理员访问用户文件必须继续走审计链路。

## 9. 推荐落地顺序

1. 上传会话协议：先设计 `UploadSession` 的 API、状态和 Redis / DB 存储策略。
2. H5 扫码上传：当前无小程序时先上线普通网页上传。
3. Kiosk 上传页接线：扫码上传成功后自动回填文件。
4. Terminal Agent 文件中转：生产一体机替代系统文件选择器。
5. 从我的文件选择：登录用户复用账号文件。
6. 小程序接入：复用上传会话和会员文件资产，不新增文件模型。
